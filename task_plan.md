# PC端保存失败排查计划

> 创建日期：2026-05-24  
> 状态：已实施并完成验证  
> 范围：启动或复用本地前后端服务，复现 PC 端保存失败、连续弹出“保存失败，请重试”的问题，定位并制定修复与回归验证闭环；内容丢失先作为待验证风险。

## 当前判断

- 本地预览稳定入口：前端 `http://127.0.0.1:5173/?ui=pc`，后端 `http://127.0.0.1:8888/api/health`。
- PC 保存主链路：`src/js/pc-editor.js` -> `src/js/api-storage.js` -> `python/main.py`。
- 当前已确认现象：
  - 保存时连续出现多条红色 Toast“保存失败，请重试”。
  - 同一画面中出现绿色 Toast“已创建”，说明可能存在部分步骤成功、后续步骤失败或提示时机错误。
- 当前高风险点：
  - 新建流程可能先创建集合成功，再更新版本或上传图片失败，导致“已创建”和“保存失败”并存。
  - 保存按钮可能重复触发，导致多次请求和多条失败 Toast。
  - 成功提示可能出现过早，未等完整保存链路结束。
  - 编辑保存时 `versions` 整体覆盖，可能丢失未带回的版本字段或图片字段。
  - 后端 `load_data()` 遇到 JSON 解析失败直接返回空数组，若随后保存会把真实数据覆盖为空。
  - 保存接口缺少写盘前快照、原子写入或坏文件保护，异常时有数据清空风险。

## 实施计划

### Phase 0：启动与基线保护

- 探测 `5173`、`8888` 是否已有可用服务。
- 可复用则复用；不可用则启动 `python/main.py` 与 Vite。
- 用 HTTP 探针确认 `/api/health`、`?ui=pc` 返回正常。
- 在复现前备份当前数据文件，避免真实数据二次损坏。

### Phase 1：复现保存失败

- 打开 PC 端预览页。
- 记录保存前 `/api/prompt-sets` 数量与真实 `prompt_sets.json` 文件大小。
- 新建或编辑测试提示词，执行保存。
- 保存后再次读取 `/api/prompt-sets` 与 `prompt_sets.json`。
- 记录 Toast 顺序和次数，重点确认“保存失败，请重试”与“已创建”是否同时出现。
- 若列表归零或字段丢失，记录为数据丢失风险升级；若未归零，则优先按保存失败处理。

### Phase 2：根因定位

- 检查 `pc-editor.js` 的 `savePromptSet()` 是否发送了不完整对象。
- 检查保存按钮是否重复触发，保存中是否缺少禁用状态。
- 检查新建流程是否存在创建成功后更新版本或图片上传失败，导致提示状态不一致。
- 检查 `api-storage.js` 对非 JSON、HTML 响应、失败状态的错误处理。
- 检查 `python/main.py` 的 `load_data()`、`save_data()`、`handle_update_prompt_set()` 是否存在空数组覆盖、非原子写入或坏文件静默吞掉问题。
- 同步检查 `build/app_main.py`，防止安装版与源码版行为不一致。

### Phase 3：修复策略

- 优先修后端数据保护：
  - JSON 解析失败不能静默当作空数据继续覆盖。
  - 写入采用临时文件加替换，降低半写入风险。
  - 保存前可保留最近备份，便于异常恢复。
- 优先修前端保存契约：
  - 保存中禁用按钮，避免重复提交。
  - 只有完整保存成功后才显示“已创建”或“已保存”。
  - 保存失败区分创建失败、版本保存失败、图片上传失败和网络失败。
  - 编辑保存前确保完整版本字段回传。
  - 保存失败不得清草稿、不得跳转到列表页。
- 再修后端数据保护：
  - 对后端非 JSON 或空响应给出明确错误。
- 如接口契约变动，同步测试与文档。

### Phase 4：回归验证

- 自动化验证：
  - `npm.cmd run test`
  - `python -m pytest python/tests -q`
  - `npm.cmd run build`
- 运行时验证：
  - 新建或编辑保存不再连续弹出“保存失败，请重试”。
  - 成功提示不再与失败提示同时出现。
  - 保存前后 `/api/prompt-sets` 数量不减少。
  - 保存后刷新 PC 页面，原内容仍存在。
  - 人为构造坏 JSON 文件时，后端拒绝覆盖并保留备份。
