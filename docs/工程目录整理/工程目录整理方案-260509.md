# 工程目录整理方案-260509

## 目标

把当前工程从“根目录堆叠”整理为“源码、平台、文档、设计、脚本、发布产物各归其位”的结构，同时避免破坏现有 Vite、Capacitor、Tauri、Python 和打包脚本。

本方案只定义整理路线，不立即移动文件。

## 当前问题

1. 根目录承载过多职责

根目录同时放置配置、启动脚本、构建脚本、临时工具、历史计划、发布 APK 和源码目录。日常开发时很难快速判断哪些文件是入口、哪些是产物、哪些是一次性工具。

2. 构建目录职责混杂

`build` 既像生成目录，又包含 `app.spec`、`installer.nsi`、`app_main.py` 这类打包源码和配置。当前脚本已经写死这些路径，不能直接移动。

3. 文档分类不稳定

`docs` 顶层有多份技术文档，子目录又有 UI 计划、构建方案、计划文档、对话历史和修复记录。缺少统一的项目导航文件，新增文档容易继续散落。

4. 发布产物和本地环境混入仓库视野

根目录存在 `PromptImageManager-v2.2.1.apk`，`releases` 下也承载安装包。`android-sdk`、`.venv`、`node_modules`、`dist` 等目录应被视作本地环境或生成结果。

## 目标目录结构

建议最终演进为：

```text
prompt-image-tool/
  src/                         # 前端源码，保持 Vite root
  python/                      # Python 后端与数据
  src-tauri/                   # Tauri 桌面端工程
  android/                     # Capacitor Android 工程
  docs/                        # 项目文档总入口
    apps-code-map.md           # 文件导航和文档更新落点
    工程目录说明.md
    工程目录整理/
    UI计划/
    构建方案/
    计划文档/
    已修复问题/
    对话历史/
  UI设计稿/                    # 原始设计稿和视觉参考
  releases/                    # 发布产物，只保留需要归档的安装包
  scripts/                     # 后续承接专项工具脚本
  packaging/                   # 后续承接打包源码和配置
  package.json
  package-lock.json
  vite.config.js
  capacitor.config.ts
  requirements.txt
  .gitignore
```

说明：

- `src`、`python`、`src-tauri`、`android` 第一阶段不移动。
- `scripts` 和 `packaging` 是后续目标目录，只有改完脚本引用后再启用。
- 根目录继续保留项目标准配置和一键入口脚本，避免破坏现有习惯。

## 文件归类建议

### 必须暂留根目录

这些文件被工具链或开发习惯强依赖，暂不迁移：

- `.gitignore`
- `package.json`
- `package-lock.json`
- `vite.config.js`
- `capacitor.config.ts`
- `requirements.txt`
- `build.bat`
- `dev.bat`
- `start-dev.bat`
- `start.bat`

### 建议第一批整理

这些文件风险较低，适合在确认后先处理：

- `plan.md` 移入 `docs/计划文档/`，并改为有语义的中文文件名。
- `PromptImageManager-v2.2.1.apk` 移入 `releases/`，或确认是否删除旧版本产物。
- 顶层散落的历史文档统一登记到 `docs/apps-code-map.md`。
- `docs` 中英文技术文档保留原名，但在导航中明确职责。

### 建议第二批整理

这些文件可以整理，但需要同步修改引用：

- `compress_icon.py` 迁入 `scripts/`，并修正脚本内路径。
- `patch-java-version.ps1` 迁入 `scripts/`，并在文档中标注它会修改 `android` 与 `node_modules`。
- `build` 中的 `app.spec`、`installer.nsi`、`app_main.py` 后续迁入 `packaging/pc/`，同时更新 `build.bat`。

### 不建议纳入版本库或应加强忽略

这些路径更像本地环境或生成产物：

- `.venv/`
- `node_modules/`
- `src/node_modules/`
- `dist/`
- `android-sdk/`
- `android/.gradle/`
- `android/app/build/`
- `src-tauri/target/`
- 安装包产物：`*.apk`、`*.zip`、`*.exe`

## 推荐执行顺序

### 阶段一：建立文档秩序

- 新增 `docs/apps-code-map.md`。
- 新增 `docs/工程文档/工程目录说明.md`。
- 保留本方案和扫描记录。

### 阶段二：清理低风险根目录文件

- 将 `plan.md` 归档到 `docs/计划文档/`。
- 将根目录 APK 归档到 `releases/` 或删除。
- 更新 `.gitignore`，覆盖根目录和 `releases` 下的安装包产物。

### 阶段三：整理脚本

- 新建 `scripts/`。
- 迁移 `compress_icon.py` 和 `patch-java-version.ps1`。
- 给每个脚本补充用途说明和运行前置条件。

### 阶段四：整理打包配置

- 新建 `packaging/pc/`。
- 迁移 `build/app.spec`、`build/installer.nsi`、`build/app_main.py`。
- 更新 `build.bat` 与构建方案文档。

### 阶段五：回归验证

- 运行前端测试：`npm run test`。
- 运行 Python 测试：`npm run test:python`。
- 至少执行一次 Vite 构建：`npm run build`。
- 如果迁移打包配置，再执行对应 PC 或 Android 构建流程。

## 结论

当前项目最需要的不是一次性重构目录，而是先建立“文件应该去哪”的规则，再分批迁移。第一批只处理历史计划、发布包和文档导航；第二批再处理脚本；最后才考虑打包配置拆分。
