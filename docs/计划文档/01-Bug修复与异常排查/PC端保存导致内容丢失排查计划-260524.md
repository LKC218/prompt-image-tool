# PC端保存失败排查计划

> 版本：v1.0.0  
> 创建日期：2026-05-24  
> 状态：已实施并完成验证  
> 范围：打开或复用本地服务器，复现 PC 端新建或编辑保存失败的问题，定位保存链路并形成修复、测试、文档闭环；“所有内容丢失”先作为保存失败后的待验证风险，不作为已确认事实。

## 一、思考与分析

### 1.1 技能与上下文嗅探

- 使用 `planning-with-files` 技能：先建立持久化计划，再进入服务启动、复现、修复。
- 根目录未发现独立 `SKILL.md` 或 `AGENTS.md` 文件，本轮以用户提供的项目指令为准。
- 既有本地预览链路：
  - 前端：`http://127.0.0.1:5173/?ui=pc`
  - 后端：`http://127.0.0.1:8888/api/health`
  - 后端入口：`python/main.py`
- 当前保存链路：
  - PC 编辑页：`src/js/pc-editor.js` 的 `savePromptSet()`
  - API 存储层：`src/js/api-storage.js` 的 `createPromptSet()` / `updatePromptSet()`
  - Python 后端：`python/main.py` 的 `handle_create_prompt_set()` / `handle_update_prompt_set()` / `load_data()` / `save_data()`

### 1.2 问题初判

用户补充截图显示，当前已确认的问题是 PC 端保存失败：页面连续弹出多条红色 Toast“保存失败，请重试”，同时底部出现绿色 Toast“已创建”。这说明前端可能存在“部分步骤成功、后续步骤失败”的状态不一致，例如先创建集合成功，再更新版本、上传图片或跳转刷新失败。

因此，本轮主问题调整为“保存失败与成功提示并存”。“保存后所有内容丢失”暂时降级为待验证风险，只有在复现中确认 `/api/prompt-sets` 数量异常归零、数据文件变空或页面刷新后数据消失时，才按数据覆盖事故处理。

| 风险点 | 可能表现 | 优先级 |
|---|---|---|
| 创建成功但更新版本失败 | 出现“已创建”，随后连续出现“保存失败，请重试”，新集合可能只有空版本或缺少图片 | 高 |
| 图片上传失败导致整体保存失败 | 多张图片或压缩图上传中断，页面反复弹失败 Toast | 高 |
| 保存按钮可重复触发 | 用户一次点击触发多次保存请求，出现重复 Toast、重复创建或竞态失败 | 高 |
| 前端成功提示时机过早 | 后续步骤失败时仍出现“已创建”或“已保存”，用户无法判断真实状态 | 高 |
| 请求打到错误后端 | 响应异常或数据源不一致 | 中 |
| 后端坏 JSON 静默返回空数组 | 数据文件解析失败后，下一次保存把空数组写回 | 高 |
| 保存非原子写入 | 写入中断或异常导致 `prompt_sets.json` 变空或半截 | 高 |
| 编辑保存整体覆盖 `versions` | 保存一个版本时丢失其他版本或图片字段 | 中 |
| 前端保存失败后跳转或清草稿 | 用户误以为保存成功，刷新后内容消失 | 中 |

### 1.3 核心原则

- 先保护真实数据，再复现问题。
- 先确认服务和数据源，再判断代码错误。
- 修复优先落在共享保存链路和后端数据保护层，避免只在单个页面打补丁。
- 若发现源码版和安装版保存逻辑不一致，需要同步检查 `build/app_main.py`。

## 二、实施计划

### Phase 0：服务启动与数据保护

目标：确保复现发生在正确服务上，避免二次破坏真实数据。

- 检查 `5173` 和 `8888` 监听状态。
- 若已有服务且 HTTP 探针正常，优先复用。
- 若服务不可达，分别启动：
  - `python python/main.py`
  - `npm.cmd run dev -- --host 127.0.0.1`
- 验证：
  - `http://127.0.0.1:8888/api/health`
  - `http://127.0.0.1:5173/?ui=pc`
- 复现前复制当前数据文件：
  - `python/data/prompt_sets.json`
  - `python/data/folders.json`
  - 如安装版参与复现，再按日志确认真实数据目录。

### Phase 1：复现与证据记录

目标：把“保存失败”变成可重复、可定位的证据，并确认是否伴随数据丢失。

- 保存前读取 `/api/prompt-sets`，记录数量、首条 ID、文件大小。
- 在 PC 端新建或编辑一个测试提示词。
- 点击保存，记录：
  - 请求 URL
  - 请求体关键字段
  - 响应状态和响应体
  - 控制台错误
  - 后端终端输出
  - Toast 出现顺序和次数，重点记录“保存失败，请重试”与“已创建”是否同时出现
- 保存后再次读取 `/api/prompt-sets` 和数据文件。
- 先判断失败类型：
  - 创建请求成功，后续更新版本失败
  - 图片上传失败
  - 重复提交导致竞态失败
  - 保存成功提示过早
  - 后端返回错误但前端提示不准确
- 再判断是否存在数据丢失：
  - 全部集合消失
  - 当前集合字段丢失
  - 版本丢失
  - 图片丢失
  - 保存失败但页面跳转

### Phase 2：代码定位

目标：按保存链路逐层确认根因。

- 前端编辑页：
  - 检查保存按钮是否有提交中锁定，避免重复点击或重复事件绑定。
  - 检查新建流程是否先 `createPromptSet()` 再 `updatePromptSet()`，以及第二步失败时是否误报“已创建”。
  - 检查图片上传失败是否导致整体保存失败，以及失败后是否保留草稿。
  - 检查 `formData.versions[0]` 是否包含完整字段。
  - 检查编辑模式是否只回传当前版本导致其他版本被覆盖。
  - 检查异常时是否清理草稿或跳转。
- API 存储层：
  - 检查 `updatePromptSet()` 是否只负责透传。
  - 检查非 JSON、HTML 响应和失败状态是否会被误判成功。
- 后端数据层：
  - 检查 `load_data()` 对空文件、坏 JSON、IO 错误的处理。
  - 检查 `save_data()` 是否存在非原子写入。
  - 检查 `handle_update_prompt_set()` 是否允许空 `versions` 覆盖完整数据。
- 安装版入口：
  - 若问题来自安装包，同步检查 `build/app_main.py`。

### Phase 3：修复方案

目标：用最小改动修复保存失败，并阻止失败路径误报成功或破坏数据。

- 前端优先修复：
  - 保存按钮进入提交中状态后禁用，防止重复提交。
  - 只有创建、版本更新、图片上传全部成功后才显示“已创建”或“已保存”。
  - 保存失败不清草稿、不跳转，并保留用户已输入内容。
  - 错误提示区分“创建失败”“版本保存失败”“图片上传失败”“网络失败”。
- 后端优先修复：
  - `load_data()` 遇到坏 JSON 时抛出可控错误，不再返回 `[]` 参与写入。
  - `save_data()` 使用临时文件写入后原子替换。
  - 写入前保留最近一次有效数据备份。
  - 更新集合时校验 `versions` 必须是列表，且不能用异常空值覆盖完整数据。
- 测试补齐：
  - 新建流程第二步失败时不显示“已创建”的测试。
  - 保存按钮重复点击只发起一次保存的测试。
  - 图片上传失败时保留草稿的测试。
  - 后端坏 JSON 不覆盖测试。
  - 保存更新不影响其他集合测试。

### Phase 4：验证与文档

目标：确认问题真实修复，并把改动记录到正确文档入口。

- 自动化验证：
  - `npm.cmd run test`
  - `python -m pytest python/tests -q`
  - `npm.cmd run build`
- 手动验证：
  - PC 新建保存只出现一次成功提示，不再出现连续“保存失败，请重试”。
  - PC 编辑保存只出现一次成功提示，不再出现连续“保存失败，请重试”。
  - 保存失败时停留在编辑页，输入内容、图片草稿仍存在。
  - 保存后刷新，原有集合仍存在。
  - 构造坏 JSON 时后端拒绝覆盖。
- 文档同步：
  - 更新 `docs/apps-code-map.md`。
  - 更新就近模块说明文档，优先检查 `docs/技术文档/pc-technical-doc.md` 或 API 文档。
  - 若沉淀为通用经验，更新 `docs/项目开发经验/项目开发经验.md`。
