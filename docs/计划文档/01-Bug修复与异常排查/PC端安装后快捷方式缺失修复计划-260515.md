# PC端安装后快捷方式缺失修复计划-260515

## 思考与分析

### Skill嗅探结果

- 根目录未发现 `SKILL.md`。
- 本次属于计划制定任务，已按 `planning-with-files` 工作流沉淀为计划文档。
- 记忆与当前仓库均指向 PC 安装包主链路：`scripts/build_pc_package.py` 生成 NSIS 安装核心，`scripts/build_installer_shell_package.py` 再把核心安装包嵌入 Tauri 安装器壳。

### 当前事实

1. `build/installer.nsi` 已包含桌面快捷方式、开始菜单启动快捷方式和开始菜单卸载快捷方式：
   - `$DESKTOP\生图提示词管理器.lnk`
   - `$SMPROGRAMS\生图提示词管理器\生图提示词管理器.lnk`
   - `$SMPROGRAMS\生图提示词管理器\卸载生图提示词管理器.lnk`
2. `scripts/build_pc_package.py` 已在构建前检查 NSIS 脚本中是否存在快捷方式配置。
3. Tauri 安装器壳安装完成后还会调用：
   - `apply_desktop_shortcut`
   - `apply_start_menu_shortcut`
4. 历史文档显示曾经出现过“处理桌面快捷方式失败”导致安装流程卡在收尾阶段，后来改成快捷方式失败只显示警告，不阻断安装完成。

### 初步判断

如果用户现在安装后完全没有桌面快捷方式和开始菜单快捷方式，问题不宜直接判定为 NSIS 缺少配置。更可能的方向有三类：

1. 实际安装的不是当前新构建的安装器壳或 NSIS 核心，而是旧 release 产物。
2. NSIS 静默安装阶段创建快捷方式失败，但当前 `install_with_nsis` 只校验主程序和卸载器，没有校验快捷方式。
3. Tauri 壳收尾阶段补建快捷方式失败后被降级为警告，用户看到“安装成功”，但快捷方式实际没有落地。

## 实施计划

### Phase 1：复现与产物身份确认

1. 确认用户安装的是哪个文件：
   - `releases/PromptImageManager-Shell-Setup-2.3.2.exe`
   - 或 `releases/PromptImageManager-Setup-2.3.2.exe`
2. 读取安装器壳和核心安装包的大小、修改时间、必要时计算哈希，确认是否为当前源码构建产物。
3. 清理或记录现有同名快捷方式，避免旧快捷方式残留影响判断。
4. 用临时安装目录执行一次真实安装，记录：
   - 主程序是否存在。
   - 卸载器是否存在。
   - 桌面快捷方式是否存在。
   - 开始菜单目录和两个快捷方式是否存在。
   - 卸载项注册表是否存在。

### Phase 2：分层定位

1. 单独运行 NSIS 核心静默安装：
   - 验证 `build/PromptImageManager-Setup-2.3.2.exe /S /D=测试目录` 是否创建快捷方式。
   - 若核心安装包能创建快捷方式，问题集中在 Tauri 壳选择状态、收尾补建或发布产物嵌入版本。
   - 若核心安装包也不能创建快捷方式，问题集中在 NSIS 变量、用户目录解析、权限或脚本编码。
2. 单独调用 Tauri 壳的收尾逻辑进行等价验证：
   - 对照 `installer-shell/src-tauri/src/main.rs` 中 PowerShell 创建 `.lnk` 的脚本。
   - 在同一安装目录、同一用户环境下复现桌面和开始菜单快捷方式创建。
3. 检查完成页和附加任务状态：
   - `installer-shell/src/main.js` 的 `state.installTasks.desktopShortcut` 和 `state.installTasks.startMenuShortcut` 默认应为 `true`。
   - 如果用户在完成页关闭了开关，确认是否会把已创建快捷方式删除。

### Phase 3：修复策略

优先采用窄修复，避免重构安装器壳：

1. 在 `install_with_nsis` 返回值中补充快捷方式校验字段：
   - `desktop_shortcut_exists`
   - `start_menu_shortcut_exists`
   - `start_menu_uninstaller_shortcut_exists`
2. 在安装流程收尾后再次读取快捷方式状态，确保 UI 不把“主程序安装成功”误报为“快捷方式也成功”。
3. 如果 PowerShell 补建失败，保留明确错误信息到调试输出，并在完成页显示短警告。
4. 如果发现完成页开关会误删快捷方式，修正状态来源，确保默认勾选时不会执行删除分支。
5. 如果问题来自旧核心安装包嵌入，修正 `scripts/build_installer_shell_package.py` 的版本与产物校验，避免 `--skip-pc-build` 复用过期核心。

### Phase 4：验证闭环

1. 静态检查：
   - `node --check installer-shell/src/main.js`
   - `node --check installer-shell/src/pages/*.js`
   - `cargo check --manifest-path installer-shell/src-tauri/Cargo.toml`
2. 构建验证：
   - `python scripts\build_pc_package.py --skip-env-check`
   - `python scripts\build_installer_shell_package.py --skip-env-check`
3. 真实安装验证：
   - 使用临时目录安装。
   - 校验主程序、卸载器、桌面快捷方式、开始菜单启动快捷方式、开始菜单卸载快捷方式。
   - 启动主程序并检查 `/api/health`。
4. 卸载验证：
   - 执行卸载器。
   - 校验安装目录、桌面快捷方式、开始菜单目录、卸载项注册表均已清理。

### Phase 5：文档同步

如进入实施并修改构建或安装行为，需要同步：

1. `docs/apps-code-map.md`
2. `docs/构建方案/PC独立安装包快速打包流程.md`
3. 必要时补充 `docs/项目开发经验/安装包中文编码与快捷方式图标规范.md`
