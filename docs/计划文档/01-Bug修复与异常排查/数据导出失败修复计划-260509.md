# 数据导出失败修复计划

> 版本：v1.0.0  
> 创建日期：2026-05-09  
> 状态：接口与兜底落盘验证通过，待前端手动点击验收  
> 范围：修复 PC 端、Web 端、Android 端设置页和相关入口无法导出数据的问题，并补充可重复验证测试。

## 一、思考与分析

### 1.1 技能与上下文嗅探

- 已使用 `planning-with-files` 技能：本次先形成独立修复计划文档，不直接改代码。
- 记忆库未找到本项目“导出失败”的历史经验条目，本计划基于当前仓库源码和文档制定。
- 当前导出入口涉及：
  - `src/js/pc-settings.js`
  - `src/js/mobile-settings.js`
  - `src/js/mobile-library.js`
  - `src/js/pc-home.js` 和 `src/js/mobile-home.js` 的导入兼容入口
  - `src/js/api-storage.js`
  - `src/js/sqlite-storage.js`
  - `python/main.py`
  - `build/app_main.py`

### 1.2 初步判断

当前导出链路大致分三层：

1. 数据生成层：`exportData()` 生成完整备份对象，包含 `backup_meta`、`folders`、`prompt_sets` 和图片 Data URL。
2. 传输/序列化层：前端将备份对象 `JSON.stringify` 后生成 `Blob`。
3. 文件保存层：前端用 `<a download>` + `URL.createObjectURL(blob)` 触发下载。

“无法导出数据”最可能出现在第 3 层。普通浏览器通常支持 `<a download>`，但 Tauri/pywebview、Android WebView、部分移动浏览器可能不支持或拦截下载。因此修复重点不能只看 `exportData()` 是否返回数据，还要验证文件是否真正落盘。

### 1.3 风险点

| 风险点 | 说明 | 优先级 |
|---|---|---|
| WebView 不支持 `a.download` | 点击导出后无文件保存弹窗，也无错误 | P0 |
| Android 端无法用浏览器下载方式保存 Blob | Capacitor WebView 中 `<a download>` 可能不落盘 | P0 |
| 完整备份含图片后 JSON 体积变大 | 大图片多时 `JSON.stringify` 和 Blob 生成可能变慢或失败 | P0 |
| PC 后端导出图片时单张图片损坏 | `build_backup_payload()` 中跳过丢失图片，用户可能误以为完整导出 | P1 |
| 前端 Toast 只显示“导出失败” | 缺少错误详情，无法判断是数据生成失败还是文件保存失败 | P1 |

### 1.4 追加定位结论

2026-05-09 复测发现导出仍报 404，根因是前端 `ApiStorage.exportData()` 使用 `GET /api/export`，但 Python 后端只在 `do_POST()` 中注册了 `/api/export`，`do_GET()` 未处理该路径，导致请求落入静态文件逻辑并返回 `File not found`。

修复方式：在 `python/main.py` 和 `build/app_main.py` 的 `do_GET()` 中注册 `/api/export`，并补充真实 HTTP 请求测试，验证 `GET /api/export` 返回 200 和完整备份结构。

### 1.5 0KB 导出文件追加定位

2026-05-09 发现浏览器下载得到的 `prompt-backup-2026-05-09-230119.json` 为 0KB。复查后端 `GET /api/export` 返回 767893 字节，说明数据生成层正常；问题集中在前端 `<a download>` 保存层。

根因判断：`saveWithAnchor()` 点击下载后立即执行 `URL.revokeObjectURL(url)` 和移除链接，部分浏览器尚未完成 Blob 读取就失去对象 URL，可能生成 0KB 文件。

修复方式：下载触发后延迟 30 秒再移除链接并释放 Blob URL，给浏览器足够时间读取 Blob；同时补充单元测试，断言不会立即释放对象 URL。

## 二、实施计划

### Phase 1：复现与定位

目标：先确认失败发生在哪个平台和哪一层。

- 在 PC 浏览器访问 `http://localhost:5173/?ui=pc`，点击设置页导出。
- 在 PC 打包/pywebview 或 Tauri 环境点击设置页导出。
- 在 Android/Capacitor 环境点击设置页导出。
- 直接请求后端 `GET /api/export`，确认 API 是否返回完整备份对象。
- 对比不同入口：
  - PC 设置页导出
  - 移动端设置页导出
  - 移动端提示词库更多菜单导出

### Phase 2：抽象统一导出文件保存工具

目标：不要在多个页面重复写 `Blob + a.download`，统一判断平台并保存。

建议新增或扩展共享工具函数：

- PC/Web：
  - 优先使用 `showSaveFilePicker`（若可用）。
  - 回退到 `<a download>`。
  - 若处于 Tauri/pywebview 且下载无效，提供后端保存接口或提示打开导出目录。
- Android/Capacitor：
  - 使用 `@capacitor/filesystem` 的 `Filesystem.writeFile()` 写入 `Directory.Documents` 或 `Directory.Data`。
  - 写入后 Toast 显示保存位置或“已保存到应用数据目录”。
  - 如后续要分享文件，再接入分享插件；本轮先保证能生成文件。

候选文件：

- 新增 `src/js/backup-utils.js`
- 或扩展 `src/js/utils.js`

第一版建议新增 `backup-utils.js`，避免把设置页继续变厚。

### Phase 3：增强导出结果反馈

目标：用户能知道“导出成功到哪里”和“为什么失败”。

- 导出前统计：
  - 提示词数量
  - 版本数量
  - 图片数量
  - 备份文件大小
- 导出成功 Toast：
  - PC/Web：显示文件名和大小
  - Android：显示文件名、大小和保存位置
- 导出失败 Toast：
  - 显示具体错误原因
  - 控制台保留完整错误对象
- 对超大备份添加提示：
  - 例如超过 50MB 时提示“图片较多，导出可能需要等待”

### Phase 4：后端兜底方案

目标：PC WebView 如果前端下载机制不可用，可以让 Python 后端直接写文件。

如复现确认 Tauri/pywebview 的 `<a download>` 不可靠，则新增：

- `POST /api/export-file`
  - 后端调用 `build_backup_payload()`
  - 写入 `data/backups/prompt-backup-YYYY-MM-DD-HHMMSS.json`
  - 返回 `{ success, path, filename, size }`
- PC 设置页导出时：
  - 先尝试前端保存
  - 失败则调用后端兜底保存

注意：如果只是浏览器环境失败，不需要新增接口；先用前端能力修复。

### Phase 5：测试与验证

目标：修复后必须可重复验证，不只依赖手动点击。

#### 自动化测试

- 前端单元测试：
  - `backup-utils` 能生成正确文件名。
  - `saveJsonBackup()` 在支持 `showSaveFilePicker` 时走文件选择器。
  - `saveJsonBackup()` 在普通 Web 环境回退 `<a download>`。
  - Android 环境模拟 `Filesystem.writeFile()` 调用。
- 存储层测试：
  - `ApiStorage.exportData()` 调用 `/api/export`。
  - `SqliteStorage.exportData()` 返回 `backup_meta` 和 `prompt_sets`。
- Python 测试：
  - `build_backup_payload()` 包含 `backup_meta`、`folders`、`prompt_sets`。
  - 有图片时导出 Data URL。
  - 图片缺失时不崩溃，并在 meta 或日志中可追踪。

#### 手动验证

- PC 浏览器：
  - 点击导出后文件落盘。
  - JSON 可打开，包含 `backup_meta`。
  - 有图片数据时包含 `data:image/...;base64,`。
- PC 桌面端：
  - 点击导出后文件落盘，或后端兜底保存成功。
- Android：
  - 点击导出后文件写入成功。
  - 大图/多图场景不白屏。
  - 导出的文件可重新导入。
- 回归验证：
  - 导入同 ID 默认覆盖。
  - 局域网同步仍可拉取图片。
