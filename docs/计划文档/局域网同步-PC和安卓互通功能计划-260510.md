### 当前工程现状

- 现有同步链路是 PC 端作为 Python HTTP 服务端，Android 端作为客户端。
- PC 服务已提供：
  - `GET /api/health`
  - `GET /api/network-info`
  - `GET /api/sync`
  - `GET /api/sync/images/{filename}`
- Android 设置页已提供：
  - 搜索 PC
  - 手动输入 PC IP
  - 测试连接
  - 从 PC 拉取同步
- 现有测试记录显示：Android 真机通过手动 IP 已可连接 PC 并完成数据与图片同步。
- 当前同步策略是 PC 优先覆盖 Android 同 ID 数据，本质仍是单向拉取，不是真正互通。

### 问题判断

“PC 和 Android 互通”不宜直接理解为 PC 主动扫描并访问 Android 服务。Android 端在 Capacitor WebView 中常驻 HTTP 服务、后台保活、端口监听、局域网广播都需要额外原生能力，第一版直接做会增加复杂度和稳定性风险。

推荐第一阶段采用更稳的互通定义：

- PC 继续作为局域网服务端，保持用户容易理解的连接模型。
- Android 既能从 PC 拉取，也能把本机数据回传给 PC。
- PC 端新增可视化互通状态、配对令牌、接收 Android 数据、同步报告。
- 双向同步由 Android 发起一次请求完成或分步完成，避免 Android 常驻服务。

这样可以用现有 HTTP 服务、备份导入导出结构和设置页入口快速闭环，同时为后续 PC 主动发现 Android 留出扩展点。

## 实施计划

### 阶段一：定义互通边界与数据策略

1. 明确第一版互通能力：
   - Android 搜索并连接 PC。
   - Android 从 PC 拉取数据。
   - Android 将本机数据回传到 PC。
   - Android 发起双向合并，PC 返回合并报告。
   - PC 设置页展示服务地址、配对状态、最近连接设备和最近同步结果。
2. 暂不纳入第一版：
   - PC 主动搜索 Android。
   - Android 后台常驻 HTTP 服务。
   - UDP 广播、mDNS 自动发现。
   - 多端实时同步。
   - 云端中继。
3. 默认数据策略：
   - 拉取同步：延续 PC 优先覆盖 Android 同 ID 数据。
   - 回传同步：默认保护 PC 数据，同 ID 冲突不直接覆盖，生成冲突副本或进入待确认列表。
   - 双向同步：按 `updatedAt` 判断较新版本；无法判断时保留双份，避免静默丢数据。
4. 增加同步模式：
   - `pull`：PC 到 Android。
   - `push`：Android 到 PC。
   - `bidirectional`：双向合并。

### 阶段二：同步协议升级

#### PC 端新增或扩展接口

1. 扩展 `GET /api/health`
   - 返回 `device_id`、`device_name`、`platform`、`sync_version`、`capabilities`。
   - 示例能力：`pull`、`push`、`bidirectional`、`pairing`。
2. 新增 `GET /api/sync/capabilities`
   - 返回服务端支持的协议版本、数据格式版本和最大上传大小。
3. 新增 `POST /api/sync/import`
   - 接收 Android 上传的备份数据。
   - 支持 `mode`：
     - `add_only`：只新增 PC 不存在的数据。
     - `keep_pc`：同 ID 保留 PC，Android 数据生成冲突副本。
     - `android_wins`：Android 覆盖 PC，仅在用户明确选择时启用。
   - 返回新增、覆盖、冲突、图片写入失败等报告。
4. 新增 `POST /api/sync/bidirectional`
   - Android 上传本机快照。
   - PC 执行合并并返回合并后的 PC 快照或增量结果。
   - 第一版可先走全量快照，后续再优化增量。
5. 新增 `GET /api/sync/pairing`
   - 返回当前 PC 的配对二维码内容或短码信息。
   - 仅在 PC 设置页打开互通服务时启用。

#### 请求安全约束

1. 写入类接口必须增加配对令牌。
2. Android 请求写入接口时携带：
   - `X-Sync-Token`
   - `X-Device-Id`
   - `X-Device-Name`
3. PC 端校验失败时返回 `401`，并在 UI 中提示重新配对。
4. 配对令牌只存本地配置，不写入文档、不进入 Git。

### 阶段三：PC 端互通能力

#### Python 后端

1. 增加设备标识：
   - 首次启动生成 `device_id`。
   - 保存在运行时数据目录，例如 `data/sync-device.json`。
2. 增加配对令牌：
   - PC 设置页手动开启互通时生成。
   - 支持刷新令牌。
   - 支持清除已配对 Android 设备。
3. 复用现有导入导出逻辑：
   - Android 上传的数据走统一标准化函数。
   - 图片使用现有安全文件名处理与写入逻辑。
4. 增加合并报告：
   - 新增提示词数量。
   - 更新提示词数量。
   - 冲突副本数量。
   - 图片接收数量。
   - 失败项与原因。

#### PC 设置页

1. 将当前“PC 服务端”区域升级为“局域网互通”区域。
2. 展示：
   - 当前服务地址。
   - 配对状态。
   - 同步令牌或二维码入口。
   - 最近连接的 Android 设备。
   - 最近一次互通结果。
3. 增加操作：
   - 复制同步地址。
   - 刷新服务信息。
   - 开启/关闭 Android 回传。
   - 重置配对令牌。
4. 回传写入前必须有明确提示：
   - 默认模式不会覆盖 PC 同 ID 数据。
   - 若选择 Android 覆盖 PC，需要二次确认。

### 阶段四：Android 端互通能力

1. 设置页局域网同步区域增加同步方向选择：
   - 从 PC 拉取。
   - 回传到 PC。
   - 双向同步。
2. 搜索结果展示 PC 能力：
   - 仅可拉取。
   - 支持回传。
   - 支持双向。
3. 新增配对流程：
   - 扫描 PC 后，如果未配对，提示输入短码或扫码。
   - 保存最近设备时同时保存 `device_id` 和配对状态。
4. 回传流程：
   - Android 导出当前本机数据快照。
   - 请求 `POST /api/sync/import`。
   - 展示 PC 返回的合并报告。
5. 双向流程：
   - 先拉取 PC 快照。
   - 本地预合并并生成差异摘要。
   - 用户确认后调用双向接口。
   - 同步完成后刷新本地统计和列表。
6. 异常提示：
   - 未配对。
   - PC 关闭回传。
   - 协议版本不兼容。
   - 图片上传失败。
   - 合并冲突需人工确认。

### 阶段五：数据合并与冲突处理

1. 建立统一交换格式：
   - 优先复用现有备份结构：`folders`、`prompt_sets`、图片 Data URL。
   - 兼容现有 `/api/sync` 的扁平结构，必要时新增转换层。
2. 最小字段要求：
   - `id`
   - `name`
   - `folderId`
   - `tags`
   - `versions`
   - `images`
   - `createdAt`
   - `updatedAt`
3. 冲突判断：
   - 同 ID 且内容摘要不同，视为冲突。
   - `updatedAt` 较新的一侧可作为推荐版本。
   - 缺少更新时间或时间无效时保留双份。
4. 默认冲突处理：
   - PC 回传默认 `keep_pc`。
   - Android 拉取默认沿用 `pc_wins`。
   - 双向默认 `keep_both_on_conflict`。
5. 后续可增加：
   - 冲突详情对比弹窗。
   - 单条记录手动选择保留哪一侧。
   - 增量同步游标。

## 风险与处理

| 风险 | 影响 | 处理 |
|---|---|---|
| Android 端无法作为稳定服务端 | PC 主动搜索 Android 难以第一版实现 | 第一版由 Android 发起请求，后续再评估原生插件 |
| 写入接口暴露在局域网 | 局域网其他设备可能误写入 | 增加配对令牌，默认关闭 Android 回传 |
| 双向合并误覆盖 | 用户数据丢失 | 默认保留双份，覆盖模式必须二次确认 |
| 图片体积较大 | 回传耗时或失败 | 限制上传大小，分批处理，报告失败项 |
| 两端数据字段不完全一致 | 合并异常 | 复用备份格式并增加转换层 |
| 时间字段不可信 | 冲突判断错误 | 内容摘要优先，时间只作辅助 |
