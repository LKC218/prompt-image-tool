# 双端安装包构建验收计划

> 创建日期：2026-05-24  
> 状态：已实施并完成本地构建与校验  
> 范围：基于当前仓库版本 `2.3.4` 构建 PC 端安装包与 Android 移动端安装包；PC 最终交付以带安装壳 UI 的安装器为主，Android 最终交付以 Release APK 为主。

## 一、目标

1. 生成 PC 核心安装包，作为安装壳嵌入源和基础验收产物。
2. 生成带安装壳 UI 的 PC 安装器，作为 PC 端最终交付产物。
3. 生成 Android Release APK，作为移动端最终交付产物。
4. 对产物做版本、签名、大小、哈希和基础运行验收，避免把旧产物误判为本轮构建结果。

## 二、当前基线

- 当前应用版本：`package.json` 为 `2.3.4`。
- Android 当前版本：`android/app/build.gradle` 为 `versionName "2.3.4"`、`versionCode 9`。
- 安装壳嵌入核心安装器路径应保持为 `build/PromptImageManager-Setup-2.3.4.exe`。
- 稳定构建入口：
  - `python scripts/build_release_packages.py --all --skip-env-check`
  - `python scripts/build_installer_shell_package.py --skip-pc-build`

## 三、预期产物

| 平台 | 产物 | 用途 |
|---|---|---|
| PC | `releases/PromptImageManager-Shell-Setup-2.3.4.exe` | 对外交付的带安装壳 UI 安装器 |
| PC | `releases/PromptImageManager-Setup-2.3.4.exe` | 核心 NSIS 安装包，供安装壳嵌入与分层验收 |
| Android | `releases/PromptImageManager-v2.3.4-Android.apk` | 对外交付的移动端安装包 |

## 四、实施阶段

### Phase 0：构建前检查

- 检查 `git status --short`，记录当前未提交改动会进入本轮产物。
- 核对版本号与嵌入路径是否一致。
- 检查构建脚本、`android/gradlew.bat`、`build/app.spec`、`build/app_main.py`、`build/installer.nsi`、`installer-shell/` 是否存在。
- 检查 `releases/` 中旧产物的修改时间和大小，避免混淆。

### Phase 1：回归基线

- `npm.cmd run test`
- `python -m pytest python/tests -q`
- `python -m pytest python/tests/test_build_app_main.py -q`

若以上测试失败，先处理与构建或安装版运行直接相关的问题，再继续构建。

### Phase 2：构建 PC 核心安装包和 Android APK

- 执行：

```powershell
python scripts/build_release_packages.py --all --skip-env-check
```

- 核对：
  - `build/PromptImageManager-Setup-2.3.4.exe`
  - `releases/PromptImageManager-Setup-2.3.4.exe`
  - `releases/PromptImageManager-v2.3.4-Android.apk`
  - `build/dist/PromptImageManager/PromptImageManager.exe`
  - `build/dist/PromptImageManager/_internal/frontend/index.html`

### Phase 3：构建 PC 安装壳 UI 包

- 执行：

```powershell
python scripts/build_installer_shell_package.py --skip-pc-build
```

- 核对：
  - `releases/PromptImageManager-Shell-Setup-2.3.4.exe`
  - 安装壳嵌入的核心安装器版本为 `2.3.4`

- 必要检查：

```powershell
node --check installer-shell/src/main.js
node --check installer-shell/src/pages/*.js
cargo check --manifest-path installer-shell/src-tauri/Cargo.toml
```

### Phase 4：产物校验

- 生成 SHA256：

```powershell
Get-FileHash releases/PromptImageManager-Shell-Setup-2.3.4.exe -Algorithm SHA256
Get-FileHash releases/PromptImageManager-v2.3.4-Android.apk -Algorithm SHA256
```

- Android APK 校验：
  - `apksigner verify --verbose --print-certs`
  - `aapt dump badging`
  - 校验包名 `com.promptimagemanager.app`
  - 校验 `versionCode='9'`
  - 校验 `versionName='2.3.4'`

- PC 产物校验：
  - Windows 文件版本信息中 `ProductVersion=2.3.4`、`FileVersion=2.3.4`
  - 文件大小非零，修改时间属于本轮构建

### Phase 5：运行与安装验收

- PC 打包 EXE 冒烟：
  - 启动 `build/dist/PromptImageManager/PromptImageManager.exe`
  - 检查 `app.log`
  - 探针 `/`、`/index.html`、`/api/health`
  - 确认没有目录列表、错误端口或 API 返回 HTML 的问题

- PC 安装壳验收：
  - 安装到临时目录
  - 验证桌面快捷方式
  - 验证开始菜单启动项
  - 验证开始菜单卸载项
  - 验证安装后应用可打开 PC UI

- Android 验收：
  - 若有真机或模拟器，执行 `adb install -r releases/PromptImageManager-v2.3.4-Android.apk`
  - 验证应用可启动
  - 若无设备，记录为“已完成 APK 构建与签名校验，真机安装待补验”

### Phase 6：文档与交付收口

- 更新 `docs/版本记录/changelog.md`，记录本次构建产物、版本与校验摘要。
- 如构建链路或产物命名有变化，同步 `docs/构建方案/`。
- 同步 `docs/apps-code-map.md` 的计划入口。
- 最终交付时列出产物路径、大小、SHA256、验证命令和剩余风险。

## 六、风险与边界

- 本计划不默认升级版本号；若工程师要求升级版本，需要先新增版本升级任务。
- 若 Android 签名文件缺失，可能只能构建未完整签名的 APK，需要先补齐签名配置。
- 若 NSIS、Rust、Android SDK 或 Gradle 环境缺失，先按脚本报错补齐环境。
- 若工作区已有未提交功能改动，本轮安装包会包含这些改动；计划阶段只记录，不回滚。
