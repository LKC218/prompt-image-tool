# 导出导入闭环测试与修复计划

> 版本：v1.0.0  
> 创建日期：2026-05-09  
> 状态：自动化闭环验证通过，待 PC/Android 手动入口验收  
> 范围：验证 PC/Web、后端 API、Android/Capacitor 的数据导出与导入是否真正闭环；若发现错误，按分层修复并补充防回归测试。

## 一、思考与分析

### 1.1 技能与上下文嗅探

- 使用 `planning-with-files` 技能：先形成独立测试与修复计划，不直接修改功能代码。
- 已有导出修复基础：
  - `src/js/backup-utils.js` 已统一 Web 下载、Android 写入和桌面端后端兜底保存。
  - `python/main.py` 与 `build/app_main.py` 已支持 `GET /api/export` 和 `POST /api/export-file`。
  - `python/tests/test_main.py` 已覆盖 `GET /api/export` 和备份文件生成。
- 当前缺口：
  - 缺少“导出的完整备份再导入”的后端 HTTP 集成测试。
  - 缺少含图片 Data URL 的导入恢复断言。
  - 缺少前端设置页/首页导入入口的可重复验证记录。
  - Android 真机导出到文件系统、再导入备份文件仍未验收。

### 1.2 风险判断

导入操作会按同 ID 覆盖本机数据，因此不能直接用真实用户数据做破坏性测试。自动化测试必须使用临时数据目录；手动测试前必须先保留当前备份文件，并优先使用测试集合验证。

### 1.3 分层验证对象

| 层级 | 验证对象 | 重点 |
|---|---|---|
| 后端数据层 | `build_backup_payload()` / `handle_import()` | 结构完整、同 ID 覆盖、图片恢复、无效数据拒绝 |
| 后端 HTTP 层 | `GET /api/export` / `POST /api/import` / `POST /api/export-file` | 状态码、响应体、落盘文件、导入统计 |
| 前端存储层 | `ApiStorage.exportData()` / `ApiStorage.importData()` / `SqliteStorage.importData()` | 请求方法、备份结构兼容、SQLite 写入 |
| 前端交互层 | `pc-settings.js` / `mobile-settings.js` / `mobile-library.js` / 首页导入入口 | 文件选择、Toast、错误信息、下载文件可读 |
| Android 层 | Capacitor Filesystem + SQLite | 导出写入 `backups/`，导入恢复记录和图片 |

## 二、实施计划

### Phase 0：测试保护与基线记录

目标：避免测试导入覆盖真实数据。

- 记录当前服务状态：前端 `5173`、后端 `8888`。
- 记录当前可用备份文件：例如 `Downloads/prompt-backup-2026-05-09-224929.json`。
- 自动化测试统一使用 `tmp_path` 临时目录，禁止写入真实 `python/data/prompt_sets.json`。
- 手动测试如需导入真实导出文件，先确认该文件可作为恢复点。

### Phase 1：后端导出导入闭环测试

目标：验证后端 API 能完整导出并重新导入。

- 新增或扩展 Python 测试：
  - 临时数据目录写入 1 个分类、1 个提示词集合、1 个版本、1 张图片。
  - 请求 `GET /api/export`，断言返回 `backup_meta`、`folders`、`prompt_sets` 和图片 Data URL。
  - 清空临时数据后请求 `POST /api/import` 导入刚导出的备份。
  - 断言导入响应包含 `imported`、`added`、`updated`、`imagesRestored`。
  - 断言重新读取数据后提示词集合、版本、图片文件均恢复。
  - 再次导入同一备份，断言走同 ID 覆盖并返回 `updated > 0`。
  - 发送无效请求体，断言返回错误，不写入脏数据。

### Phase 2：前端存储层与工具测试

目标：验证前端调用路径和保存工具不回退。

- 扩展 `api-storage.test.js`：
  - `exportData()` 继续调用 `GET /api/export`。
  - `exportFile(filename)` 调用 `POST /api/export-file`。
  - `importData(data)` 调用 `POST /api/import`，请求体保持完整备份对象，不再包一层数组。
- 扩展 `backup-utils.test.js`：
  - 后端兜底不可用时继续回退到前端下载。
  - 保存成功消息包含文件名、大小、图片数量。
- 评估 `sqlite-storage` 测试：
  - 若 mock 成本可控，补 `exportData()` 和 `importData()` 的 SQLite 单元测试。
  - 若 mock 成本过高，先保留 Android 真机验证清单。

### Phase 3：PC/Web 手动验收

目标：验证用户实际入口可用。

- 打开 `http://localhost:5173/?ui=pc`。
- 在设置页点击导出，确认浏览器下载 `.json` 文件。
- 打开下载文件，确认包含：
  - `backup_meta`
  - `folders`
  - `prompt_sets`
  - 如存在图片，包含 `data:image/...;base64,`
- 在设置页点击导入，选择刚导出的文件。
- 断言导入 Toast 显示新增、覆盖、图片恢复数量。
- 刷新页面后确认提示词列表仍可正常展示。
- 从首页导入入口再做一次同文件导入，确认入口行为一致。

### Phase 4：桌面端兜底验收

目标：验证 WebView 下载不可靠时也能保存备份。

- 调用 `POST /api/export-file`，确认写入 `python/data/backups/`。
- 打开保存文件，确认 JSON 可解析。
- 若 PC 桌面端可运行，点击设置页导出：
  - 优先验证是否直接后端保存。
  - Toast 中应包含文件名、大小、保存路径。

### Phase 5：Android/Capacitor 验收

目标：验证移动端原生文件写入和导入恢复。

- 在 Android 端设置页点击本地备份。
- 确认 `Filesystem.writeFile()` 成功，保存到 `backups/`。
- 使用 Android 文件导入入口选择备份文件。
- 断言导入后列表、详情、图片展示正常。
- 大图/多图场景至少验证一次不卡死、不白屏。

## 三、错误修复策略

若任一阶段失败，按以下顺序定位并修复：

1. API 404 或方法错误：
   - 检查 `ApiStorage` 方法与 Python `do_GET()` / `do_POST()` 路由是否一致。
   - 同步修改 `python/main.py` 和 `build/app_main.py`。
   - 补 HTTP 集成测试。
2. 备份结构错误：
   - 检查 `backup_meta`、`folders`、`prompt_sets` 命名是否一致。
   - 保持兼容旧字段 `promptSets` / `backupMeta`。
3. 图片导入失败：
   - 检查 Data URL 头、base64 内容、扩展名推断和文件落点。
   - 补缺失图片不崩溃测试。
4. 同 ID 覆盖异常：
   - 检查导入前旧图片清理和 `existing_map` 更新。
   - 补重复导入回归测试。
5. 前端交互失败：
   - 检查文件选择、异常 Toast、控制台错误对象。
   - 优先修复共享工具或存储层，不在多个页面重复补丁。
