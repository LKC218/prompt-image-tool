# Tauri安装器壳-正式打包接入计划

## 思考与分析

当前 `installer-shell/` 已经能够构建并显示安装器壳 UI，现有 NSIS 安装核心也已通过静默安装、启动、快捷方式、注册表和卸载闭环验证。下一步目标是让交付产物双击后第一屏就是安装器壳 UI。

本轮不采用 Tauri 自带 `nsis` bundle 作为最终交付形态，因为那会先显示 Tauri 应用安装器，而不是项目已经设计好的安装器壳 UI。更合适的首版方案是：

1. 先复用现有 `build/PromptImageManager-Setup-2.3.1.exe` 作为安装核心。
2. Tauri release 构建时把该 NSIS 安装核心嵌入安装器壳 EXE。
3. 输出单个安装器壳 EXE 到 `releases/`。
4. 用户双击该 EXE 后先看到安装器壳 UI，再由壳内按钮调用嵌入的 NSIS 静默安装核心。

## 实施计划

### 1. 嵌入式安装核心

- 在 `installer-shell/src-tauri/Cargo.toml` 增加 `embedded-installer` feature。
- 在 `installer-shell/src-tauri/src/main.rs` 中仅当启用该 feature 时使用 `include_bytes!` 嵌入 `build/PromptImageManager-Setup-2.3.1.exe`。
- `find_installer()` 保留外部资源查找能力；若外部资源不存在，再把嵌入的安装核心释放到临时目录。
- 默认 `cargo check` 不启用 feature，避免普通 UI 开发时强依赖安装包已存在。

### 2. 打包脚本

新增 `scripts/build_installer_shell_package.py`：

- 读取根目录 `package.json` 版本号。
- 可选调用 `scripts/build_pc_package.py` 先刷新标准 NSIS 安装核心。
- 校验 `build/PromptImageManager-Setup-{version}.exe` 存在。
- 执行 JS 语法检查和 Rust 检查。
- 执行 `npm.cmd --prefix installer-shell run tauri -- build --features embedded-installer --no-bundle`。
- 将 release 构建出的 Tauri 壳 EXE 复制到：

```text
build/installer-shell/PromptImageManager-Shell-Setup-{version}.exe
releases/PromptImageManager-Shell-Setup-{version}.exe
```

### 3. 验证

- 启动 `releases/PromptImageManager-Shell-Setup-{version}.exe`。
- 确认窗口标题为 `提示词管家安装向导`。
- 确认进程响应正常。
- 不在本轮强制自动点击安装按钮；真实安装核心已由 UI 实装测试计划验证，正式链路进入发布前再补一次手工点击或稳定桌面自动化。

## 缺陷修复记录-260512

### 1. 问题

用户在安装器壳“正在安装”页看到：

```text
当前环境无法执行安装
请在 Tauri 安装器壳窗口中开始真实安装。
```

### 2. 根因

安装器壳前端是直接由 `installer-shell/src/index.html` 加载的静态页面，`main.js` 通过 `window.__TAURI__.core.invoke` 调用 Rust 命令。`installer-shell/src-tauri/tauri.conf.json` 未启用 `app.withGlobalTauri` 时，release 窗口不会暴露 `window.__TAURI__`，导致真实安装包也被前端判断成无法执行安装的环境。

### 3. 修复

- `installer-shell/src-tauri/tauri.conf.json` 增加 `app.withGlobalTauri: true`。
- `docs/apps-code-map.md` 与 `docs/设计文档/Tauri-安装器壳-UI-设计规则-260510.md` 同步记录静态页面调用 Tauri 命令的配置要求。

### 4. 验证要求

后续正式打包验证不能只停留在“窗口可打开”，还需要至少确认“开始安装”按钮能够进入 Rust `install_with_nsis` 命令，而不是落入前端预览环境兜底提示。

## 缺陷修复记录-260512-快捷方式收尾

### 1. 问题

用户在安装器壳“正在安装”页看到安装进度停在 `94%`，状态为：

```text
安装遇到问题
处理桌面快捷方式失败。
```

此时主程序安装已经完成，问题发生在收尾任务阶段。

### 2. 根因

`install_with_nsis` 完成后，前端会继续调用 `apply_desktop_shortcut` 和 `apply_start_menu_shortcut`。旧逻辑把桌面快捷方式创建失败直接抛出到正在安装页，导致用户看到整次安装失败。Rust 侧也只返回笼统的“处理桌面快捷方式失败”，没有保留 PowerShell 标准输出和错误信息，排障成本较高。

### 3. 修复

- `installer-shell/src-tauri/src/main.rs` 新增统一 PowerShell 执行封装，使用 `-Sta` 运行快捷方式 COM 创建逻辑，并捕获 stdout/stderr。
- 桌面目录和开始菜单目录读取失败时返回明确错误。
- `installer-shell/src/main.js` 将桌面快捷方式、开始菜单快捷方式处理改为收尾警告；主程序和卸载器已经安装成功时，不再因为快捷方式失败卡在 `94%`。
- 完成页只显示短提示，避免长 PowerShell 错误撑乱 UI。
