# 移动端下载记录UI布局排版优化计划-260524

> 版本：v1.0.0  
> 创建日期：2026-05-24  
> 状态：已实施并完成验证  
> 范围：优化移动端设置页“图片下载记录”区域的列表卡片、字段层级、时间与平台信息排版，解决窄屏下时间逐字换行、记录项过高、信息层级不清晰的问题。

## 一、思考与分析

### 1.1 技能与上下文嗅探

- 使用 `planning-with-files` 技能：本次先沉淀计划文件，再等待确认实施。
- 当前功能入口：
  - `src/js/mobile-settings.js`：移动端设置页下载记录渲染。
  - `src/css/mobile.css`：`.m-download-history-*` 相关样式。
  - `src/js/download-history.js`：下载记录数据结构与时间格式化。
- 现有模块文档入口：
  - `docs/UI计划/移动端UI设计/06-设置与本地存储.md`
  - `docs/apps-code-map.md`

### 1.2 截图问题定位

当前截图中“图片下载记录”区域已经具备基础功能，但 UI 排版存在明显可读性问题：

1. 时间信息被挤到左侧窄列，变成 `2026- / 05- / 24 / 02:56` 这类逐段换行，扫描效率低。
2. 平台信息“移动端”被拆成多行，说明记录项内部没有稳定的元信息容器。
3. 文件名较长时占用主内容宽度，但来源与保存位置缺少更轻量的二级信息样式。
4. 记录卡片上下留白偏大，单条记录高度接近一个大卡片，不利于设置页继续展示后续“局域网同步”内容。
5. 数量与“清空历史”按钮在标题右侧可用，但视觉重量偏散，需要统一成轻量工具区。

### 1.3 优化原则

- 优先保持现有数据结构不变，只调整渲染结构和 CSS。
- 移动端记录项采用单列信息流，不再让时间独立占据窄列。
- 长文件名只允许单行省略，不能撑破卡片或挤压时间。
- 元信息统一放到文件名下方或底部一行，用小号文字与标签分隔。
- 清空历史按钮保持可发现，但视觉权重低于列表内容。

## 二、实施计划

### Phase 1：重构记录项信息结构

1. 调整 `renderDownloadHistory()` 输出结构：
   - 左侧保留下载图标。
   - 右侧主区域分为文件名、来源/保存位置、时间/平台三层。
   - 时间与平台放在同一行或紧凑标签组，避免逐字换行。
2. 对文件名使用 `title` 和单行省略。
3. 对来源和保存位置使用 `·` 分隔，保持二级文字弱化。

### Phase 2：优化卡片布局与密度

1. 将记录卡片从“大块网格”调整为紧凑列表项：
   - 图标固定 `36px` 左列。
   - 主内容 `minmax(0, 1fr)`。
   - 记录项内边距控制在 `12px 14px` 左右。
2. 去掉时间元信息的自动换行风险：
   - 使用 `white-space: nowrap`。
   - 必要时在窄屏下让元信息整行换到下一行，但不拆字。
3. 调整记录卡片圆角、阴影和分隔线，让它更像设置页内的列表，而不是独立大卡片。

### Phase 3：优化标题工具区

1. “图片下载记录”标题保持左侧主标题。
2. 数量显示改为轻量徽标，如 `1 条`，避免只有孤立数字。
3. “清空历史”按钮改成更小的胶囊按钮，并保证触控高度不低于 28px。
4. 历史为空时禁用或弱化清空按钮，减少误触。

### Phase 4：空态与多记录状态

1. 空态保持轻量文案，不增加大插画。
2. 多条记录时检查分隔线、间距和滚动体验。
3. 长文件名、多字节中文、英文无空格文件名都要覆盖。

### Phase 5：回归验证

1. 单元/页面回归：
   - 更新 `src/js/mobile-regression.test.js` 中下载记录区域断言。
   - 确认清空历史仍能弹出确认并刷新计数。
2. 视觉验证：
   - 移动端宽度 `360px`、`390px`、`430px` 下截图检查。
   - 确认时间不会逐字换行。
   - 确认“局域网同步”标题不会被下载记录卡片异常推远。
3. 构建验证：
   - `npm.cmd test`
   - `npm.cmd run build`
