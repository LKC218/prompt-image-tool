- 已使用 `frontend-design` 技能：本次属于既有 Vanilla HTML/CSS/JS 前端体验增强，应复用现有视觉 token、交互组件和页面结构，不新增 UI 框架。
- 已使用 `ui-ux-pro-max` 技能：重点校验表单反馈、图片优化、交互控件稳定尺寸、悬停和焦点反馈，避免粘贴图片、草稿恢复和筛选标签造成布局跳动。

### 当前代码定位

本次 4 项需求集中在 PC 端提示词模块：

| 需求 | 主要文件 | 当前现状 |
|---|---|---|
| 新建/编辑提示词页支持外部图片复制到上次图片里 | `src/js/pc-editor.js`、`src/js/image-utils.js`、`src/css/pc.css` | 已支持文件选择上传、图片压缩和最多 10 张预览；暂未监听剪贴板图片，也没有粘贴入口提示。 |
| 新建/编辑提示词支持临时保存 | `src/js/pc-editor.js`、必要时新增小型草稿工具 | 页面内已有 `hasUnsavedChanges` 和离开确认，但 `unmount()` 会清空 `formData` 与 `importedImages`，返回后不能恢复未保存内容。 |
| 提示词详情页正向提示词第一行存在空白区域 | `src/js/pc-detail.js`、`src/css/pc.css` | 详情页用 `white-space: pre-wrap` 原样展示提示词；如果数据开头含换行或空白，会在第一行展示空白。截图中红框位置更像数据前导空白被保留，而不是单纯 CSS 对齐问题。 |
| 提示词库筛选标签栏增加分类夹标签 | `src/js/pc-library.js`、`src/css/pc.css` | `getFilteredItems()` 已支持 `folder:` 和 `uncategorized` 过滤，但 `renderFilterBar()` 只渲染“全部、收藏、最近使用”，没有把分类夹渲染成筛选按钮。 |

### 设计目标

1. 保持 KISS：优先在 `pc-editor.js`、`pc-detail.js`、`pc-library.js` 和 `pc.css` 内做窄修复，除非草稿序列化明显需要独立小工具。
2. 不新增后端和数据库表：草稿属于临时编辑态，优先使用浏览器本地会话存储；正式保存仍走现有 `storage.createPromptSet()`、`storage.updatePromptSet()` 和 `storage.uploadImage()`。
3. 图片入口统一复用现有图片处理链路：剪贴板图片转换为 `File` 后进入现有 `handleImageUpload()`，继续受格式、大小、压缩和数量上限约束。
4. 详情页展示修正不污染原始数据：只在展示层去除前导空白，复制按钮继续复制原始提示词，避免悄悄改用户保存内容。
5. 筛选标签栏增加分类夹但不挤爆布局：分类夹标签使用现有按钮体系，数量多时优先换行或横向滚动，不改变提示词库主工作台结构。

## 实施计划

### Phase 1：编辑页支持复制外部图片后粘贴追加

1. 在 `pc-editor.js` 编辑页挂载期间增加剪贴板图片处理：
   - 监听编辑页作用域内的 `paste` 事件。
   - 从 `event.clipboardData.items` 或 `files` 中筛选 `image/*`。
   - 将图片条目转换为 `File` 对象，并调用现有 `handleImageUpload(pageEl, files)`。
2. 在图片预览区域增加低干扰粘贴提示：
   - 无图时占位文案补充“可点击上传或粘贴图片”。
   - 有图时新增小型“粘贴图片”按钮或提示，不改变当前网格高度。
3. 继续沿用现有限制：
   - `MAX_IMAGES = 10`。
   - `MAX_SOURCE_IMAGE_BYTES = 15MB`。
   - `ALLOWED_IMAGE_TYPES = jpeg/png/webp`。
   - `optimizeImageDataUrl()` 统一压缩。
4. 增加边界反馈：
   - 剪贴板没有图片时不打扰用户。
   - 超过数量上限时沿用 Toast 提醒。
   - 不支持的图片类型或过大图片沿用现有错误提示。

### Phase 2：编辑页临时保存与返回恢复

1. 设计草稿键：
   - 新建页：`pc-editor-draft:new`。
   - 编辑页：`pc-editor-draft:${promptSetId}`。
2. 草稿内容：
   - `formData`：标题、正向提示词、负向提示词、标签、分类、比例等字段。
   - `importedImages`：仅保存已经压缩后的 `compressedUrl`、`name`、`type`、`size` 和必要元数据。
   - `updatedAt`：用于判断是否存在可恢复草稿。
3. 自动保存触发：
   - 标题、正向提示词、负向提示词输入。
   - 标签增删。
   - 分类、比例变更。
   - 图片上传、粘贴、删除。
4. 页面恢复策略：
   - `mount()` 先加载正式数据，再读取对应草稿。
   - 若存在草稿，自动恢复并 Toast 提示“已恢复上次未保存内容”。
   - 保存成功后删除对应草稿。
   - 用户明确放弃返回时，仍保留草稿，满足“去别的页面再回来还能记录”的要求。
5. 存储容错：
   - 草稿写入失败时不影响正式编辑。
   - 如果图片草稿过大导致写入失败，至少保留文本、标签、分类和比例草稿，并 Toast 轻提示。

### Phase 3：详情页正向提示词首行空白修正

1. 新增展示层格式化函数，例如 `formatPromptForDisplay(text)`：
   - 去除字符串开头的空格、制表符和换行。
   - 保留正文中的换行、缩进和段落结构。
   - 空字符串继续显示现有空态。
2. 在 `renderPositivePromptCard()` 和 `renderNegativePromptCard()` 中使用展示文本：
   - 正向提示词优先修复截图问题。
   - 负向提示词同步处理，保证同类展示规则一致。
3. 保留复制逻辑不变：
   - `copyToClipboard()` 继续使用版本原始字段。
   - 不在详情页写回修剪后的内容。
4. 检查 CSS：
   - `.pc-detail-prompt-content` 保留 `white-space: pre-wrap`。
   - 若仍有视觉偏移，再微调 `padding`，不优先用 CSS 掩盖数据前导空白。

### Phase 4：提示词库筛选栏增加分类夹标签

1. 扩展 `renderFilterBar()`：
   - 保留固定筛选：“全部、收藏、最近使用”。
   - 增加“未分类”按钮。
   - 根据 `libraryData.folders` 渲染分类夹按钮，按钮 key 为 `folder:${id}`。
   - 每个分类夹显示当前提示词数量。
2. 复用已有过滤逻辑：
   - `getFilteredItems()` 已处理 `uncategorized` 和 `folder:`，只需确保字段名 `folderId` 与数据返回一致。
3. 交互细节：
   - 点击分类夹后重置 `currentPage = 1`。
   - 当前选中分类夹保留蓝色激活态。
   - 搜索关键字与分类筛选叠加生效。
4. 样式增强：
   - 分类夹按钮可增加统一文件夹图标，但优先用现有内联 SVG 或本地图标体系，不引入新依赖。
   - 分类数量较多时确保按钮不挤压搜索栏和表格。

### Phase 5：文档同步

1. 更新 `docs/apps-code-map.md`：
   - 补充本计划文档导航。
   - 实现完成后再同步 `src/js/pc-editor.js`、`src/js/pc-library.js`、`src/js/pc-detail.js` 和 `src/css/pc.css` 的职责描述。
2. 更新最近模块说明文档：
   - 若只改 PC 前端交互，优先同步 `docs/技术文档/pc-technical-doc.md`。
   - 若沉淀交互规则，补充到 `docs/UI计划/` 中最贴近的 PC 端说明文档。
3. 如草稿恢复或剪贴板图片处理形成可复用经验，再同步 `docs/项目开发经验/项目开发经验.md`。

## 风险与处理

| 风险 | 影响 | 处理方式 |
|---|---|---|
| 浏览器剪贴板权限差异 | 主动读取剪贴板可能失败 | 优先使用 `paste` 事件数据；按钮读取剪贴板只作为增强，不作为唯一入口。 |
| 草稿图片过大 | `localStorage` 或 `sessionStorage` 容量不足 | 图片先压缩；写入失败时降级保存文本字段；必要时后续再升级 IndexedDB。 |
| 自动恢复草稿覆盖正式数据 | 编辑页进入后看到旧草稿 | 通过草稿 key 隔离新建和具体提示词；保存成功立即清理草稿。 |
| 修剪展示文本改变复制语义 | 用户复制到的内容与保存内容不一致预期 | 详情展示层修剪，复制按钮保留原始内容；如工程师希望复制也清理，可在实现前确认。 |
| 分类夹过多导致筛选栏拥挤 | 1366 宽度下换行或遮挡 | 使用现有 `flex-wrap`，必要时加最大高度和横向滚动，不改主工作台结构。 |

## 追加计划：详情页前导空白专项复验

1. 复核根因：
   - 当前截图中的空白出现在提示词正文第一行起始位置，优先按“原始提示词字段开头含换行、空格或制表符，并被 `white-space: pre-wrap` 保留”处理。
   - 不把 `.pc-detail-prompt-content` 的整体内边距当成首要问题，避免用 CSS 负向偏移掩盖数据展示问题。
2. 实现边界：
   - 展示层使用 `formatPromptForDisplay(text)` 清理字符串开头连续空白。
   - 正文内部的换行、段落和缩进必须保留。
   - 复制按钮继续使用原始版本字段，不把展示层修剪结果写回数据。
3. 验证路径：
   - 运行 `npm.cmd run test -- src/js/pc-prompt-ui-utils.test.js`，确认纯函数覆盖前导空白清理。
   - 运行 `node --check src/js/pc-detail.js`，确认详情页脚本语法正确。
   - 启动前端预览，打开含前导空白的提示词详情页，对照工程师截图检查红框空白是否消失。
   - 点击正向提示词复制按钮，粘贴到临时文本区，确认复制内容仍为原始提示词。
4. 收口文档：
   - 若只完成复验，不额外改动 `docs/apps-code-map.md`。
   - 若再次调整 `pc-detail.js`、`pc-prompt-ui-utils.js` 或 `pc.css`，同步更新 `docs/apps-code-map.md` 与 `docs/UI计划/PC端/03-提示词详情.md`。
