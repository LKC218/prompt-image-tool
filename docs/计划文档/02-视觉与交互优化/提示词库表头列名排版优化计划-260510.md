# 提示词库表头列名排版优化计划-260510

## 思考与分析

### Skill 嗅探结果

- 根目录未发现项目级 `SKILL.md`。
- 本次任务是制定优化计划，已使用 `planning-with-files` 技能，将计划沉淀为持久 Markdown 文档。

### 问题定位

提示词库 PC 页面表格由 `src/js/pc-library.js` 的 `renderLibraryContent()` 渲染，表头结构为：

- `pc-library-workspace`
- `pc-library-table-card`
- `pc-library-table`
- `thead > tr > th`

当前表头样式集中在 `src/css/pc.css`：

- `.pc-library-table` 使用 `table-layout: fixed`。
- `.pc-library-col-prompt` 使用 `width: 48%`。
- 第 2 至第 6 列通过 `nth-child()` 设置固定宽度。
- `.pc-library-table th` 使用 `font: 800 13px/1 var(--pc-font-family)`，未设置不换行、最小宽度或稳定的表头内容容器。

在窗口变窄、右侧预览面板仍占宽、系统字体渲染差异或开发者工具缩放场景下，第一列表头可用宽度被压缩，中文列名会逐字换行，出现“提示词”三字垂直堆叠甚至重叠的问题。

### 优化目标

- 表头列名在常规 PC 宽度下保持横向可读，不出现逐字挤压。
- 表格列宽和右侧预览面板在窄宽度下有明确退化策略。
- 不引入复杂表格组件，不改变提示词库现有数据结构和交互行为。
- 优先通过 CSS 解决，必要时微调表头 DOM，使布局更稳定。

## 实施计划

### Phase 1：表头文本稳定化

1. 为表头文本建立稳定排版规则：
   - 设置 `white-space: nowrap`，避免中文逐字换行。
   - 设置合理 `line-height`，避免视觉重叠。
   - 设置 `overflow: hidden` 与 `text-overflow: ellipsis`，极窄场景下优雅截断。
2. 优先只调整 `.pc-library-table th`，保持影响范围收敛在提示词库表格。
3. 如需更精细控制，再给表头文字增加内部包裹节点，例如 `.pc-library-th-label`，但第一版建议先不改 DOM。

### Phase 2：列宽约束优化

1. 调整第一列策略：
   - 保留第一列为主内容列，但增加 `min-width` 或改用 `colgroup` 明确列宽。
   - 避免 `.pc-library-col-prompt { width: 48%; }` 在窄宽度下压缩到不可读。
2. 复核第 2 至第 6 列宽度：
   - “标签”列保持足够展示 2 至 3 个标签。
   - “创建时间”和“最近使用”列保持完整表头可读。
   - “收藏”和“操作”列保持居中，并控制为图标按钮宽度。
3. 必要时为 `.pc-library-table-card` 增加横向滚动：
   - 使用 `overflow-x: auto`。
   - 为 `.pc-library-table` 设置 `min-width`，保证表格不会被压到表头不可读。

### Phase 3：响应式退化策略

1. 在现有 `@media (max-width: 1180px)` 基础上复核布局：
   - 当前右侧预览会下沉为单列，这是正确方向。
   - 表格在单列布局下应恢复更宽可用空间。
2. 为 1180px 至 1400px 之间的临界宽度增加保护：
   - 可适当缩小右侧预览宽度，或提前触发单列布局。
   - 避免主表格和右侧预览同时挤压。
3. 保留当前视觉风格：白色圆角表格、浅色表头、选中行蓝色描边不改变。

### Phase 4：文档同步

1. 更新 `docs/apps-code-map.md`：
   - 仅确认现有 `src/css/pc.css` 与 `src/js/pc-library.js` 导航描述仍准确。
   - 若只是样式微调且描述已覆盖，无需新增修改历史。
2. 更新就近模块说明文档：
   - `docs/UI计划/PC端/02-提示词库.md`
   - 补充表头列名应保持单行可读、窄宽度通过表格最小宽度或横向滚动兜底。
3. 若修复过程中总结出可复用规则，可考虑补充：
   - `docs/项目开发经验/项目开发经验.md`

## 风险与注意事项

- 直接增加表格 `min-width` 可能带来横向滚动，需要确认滚动条视觉是否可接受。
- 直接缩小右侧预览宽度可能影响图片预览体验，应优先保护表格可读性。
- 如果只设置 `white-space: nowrap`，但不处理最小宽度，极窄场景可能出现文本裁切；需要配套 `text-overflow` 或横向滚动。
- 表头和表体列宽必须保持同一张表内自然对齐，避免单独给表头做脱离表格流的定位。
