### 当前工程现状

- 现有局域网同步以 PC 作为 Python HTTP 服务端，Android 作为发起方。
- Android 端已支持搜索 PC、手动输入 `IP:端口`、拉取、回传和双向同步。
- PC 端已提供 `/api/sync`、`/api/sync/preview`、`/api/sync/import`、`/api/sync/bidirectional`、`/api/sync/capabilities`、`/api/sync/pairing` 等接口。
- 当前文档定义的默认策略是：
  - 拉取模式：PC 端同 ID 数据覆盖 Android 本机同 ID 数据。
  - 回传模式：默认 `keep_pc`，PC 同 ID 数据保留，Android 数据生成冲突副本。
  - 双向模式：Android 发起，先回传再拉取 PC 最新快照。

### 核心问题判断

现有同步冲突处理可以防止直接覆盖 PC 数据，但用户体验和数据一致性仍有风险：

1. 同步前没有冲突预览，用户只能在写入后看到结果。
2. 冲突判断粒度偏粗，容易把字段命名差异、时间字段、图片编码差异误判为内容冲突。
3. 双向同步前端流程和后端 `/api/sync/bidirectional` 容易产生语义漂移，后续维护时可能一边改了另一边没跟上。
4. 冲突副本缺少稳定来源标识，重复执行同步可能生成多份相似副本。
5. 写入前缺少统一备份点与事务式失败边界，一旦图片写入、JSON 写入或移动端本地写入中途失败，恢复成本较高。
6. PC 与 Android 两端对“本端独有、对端独有、同 ID 同内容、同 ID 不同内容”的展示不够明确，用户难以判断同步是否安全。

本次优化不建议直接引入云同步、多端实时协作或复杂版本向量。第一阶段应保持 KISS：围绕现有全量快照和本地 HTTP 协议，加一层差异预览、稳定冲突键、安全写入和可读报告。

## 实施计划

### 阶段一：统一冲突模型

1. 定义同步差异类型：
   - `only_pc`：仅 PC 端存在。
   - `only_android`：仅 Android 端存在。
   - `same`：同 ID 且业务内容一致。
   - `conflict`：同 ID 但业务内容不同。
   - `invalid`：数据结构不完整或无法合并。
2. 定义业务内容摘要规则：
   - 纳入标题、分类、标签、收藏状态、版本提示词、反向提示词、备注、参数、图片元数据。
   - 排除生成型 ID、创建时间、更新时间、图片 Data URL、图片体积、MIME 细节、字段命名差异。
   - 图片内容优先比较稳定文件名、图片注释和可选哈希；没有哈希时不把编码差异当成冲突。
3. 定义冲突记录结构：
   - `id`
   - `name`
   - `type`
   - `pcSummary`
   - `androidSummary`
   - `fieldDiffs`
   - `recommendedAction`
   - `conflictKey`

### 阶段二：新增同步前差异预览

1. PC 后端新增轻量预览能力，推荐接口：
   - `POST /api/sync/preview`
2. 请求内容复用 Android 当前导出的备份快照：
   - `mode: pull | push | bidirectional`
   - `payload`
3. 返回内容包含：
   - 新增数量。
   - 跳过数量。
   - 冲突数量。
   - 字段级差异摘要。
   - 推荐动作。
   - 是否需要用户确认。
4. Android 端在执行回传或双向同步前先调用预览接口。
5. 拉取模式若只读 PC 快照，也要在 Android 本地生成预览摘要，避免静默覆盖本机修改。

### 阶段三：冲突处理策略收敛

1. 保留当前默认安全策略：
   - PC 默认不被 Android 覆盖。
   - Android 拉取默认以 PC 为准，但覆盖前必须提示冲突数量。
2. 增加用户可理解的操作选项：
   - 保留 PC，Android 冲突生成副本。
   - 保留 Android，覆盖 PC（默认不暴露，后续可在高级选项开启）。
   - 两边都保留。
   - 跳过冲突项，只同步无冲突项。
3. 第一版 UI 推荐只开放两种低风险选择：
   - 安全同步：无冲突项直接同步，有冲突项生成副本。
   - 只同步无冲突项：冲突项全部跳过。
4. 后端保留 `android_wins` 能力，但 UI 不默认展示，避免误覆盖。

### 阶段四：冲突副本幂等化

1. 为每个冲突副本生成稳定 `conflictKey`：
   - 来源设备 ID。
   - 原始记录 ID。
   - 业务内容摘要。
   - 冲突发生方向。
2. 再次同步同一份冲突内容时：
   - 已存在同 `conflictKey` 的副本则跳过或更新报告。
   - 不重复生成多个“Android冲突副本”。
3. 冲突副本命名增加来源和简短时间：
   - `原名称（Android冲突副本-260521）`
4. 在数据中保留最小元信息：
   - `syncMeta.sourceDeviceId`
   - `syncMeta.conflictKey`
   - `syncMeta.originalId`
   - `syncMeta.createdBySync`

### 阶段五：安全写入与失败恢复

1. PC 后端写入前创建本地自动备份：
   - JSON 数据备份。
   - 图片写入失败不应破坏已有图片。
2. 写入顺序调整为：
   - 校验 payload。
   - 生成预览和写入计划。
   - 备份当前数据。
   - 写入图片到临时文件。
   - 写入 JSON 数据。
   - 提交图片文件。
   - 返回同步报告。
3. 任一关键步骤失败时：
   - 返回失败报告。
   - 保留备份路径。
   - 不删除原始数据。
4. Android 本地拉取覆盖前也要保留一次本地同步前快照，至少保留最近一份。

### 阶段六：前端交互优化

1. Android 设置页同步操作流程调整为：
   - 选择同步方向。
   - 连接 PC。
   - 生成差异预览。
   - 用户确认策略。
   - 执行同步。
   - 展示结果报告。
2. 预览弹层展示：
   - 本机独有数量。
   - PC 独有数量。
   - 同 ID 冲突数量。
   - 图片失败风险。
   - 推荐策略。
3. 同步结果报告展示：
   - 新增、跳过、冲突副本、失败项。
   - 冲突副本可点击查看。
   - 失败项可复制诊断信息。
4. PC 设置页展示最近同步摘要：
   - 最近连接设备。
   - 最近同步方向。
   - 冲突数量。
   - 自动备份位置。

### 阶段七：双向同步入口统一

1. 梳理 `src/js/lan-sync.js` 中 `bidirectional()` 与后端 `/api/sync/bidirectional` 的职责。
2. 推荐后续实现统一为：
   - Android 调用预览接口。
   - 用户确认后调用后端 `/api/sync/bidirectional`。
   - 后端返回 `pushReport` 和 `syncData`。
   - Android 再按返回的 `syncData` 写入本地。
3. 避免前端继续用 `push()` + `sync()` 拼装双向语义，减少协议漂移。

### 阶段八：测试验证

1. 前端单元测试：
   - 业务摘要忽略时间字段和字段命名差异。
   - 预览报告能区分 `only_pc`、`only_android`、`same`、`conflict`。
   - 双向同步使用统一接口和真实动态端口。
2. Python 测试：
   - `merge_import_payload()` 同内容不生成冲突。
   - 同一 `conflictKey` 重复回传不生成重复副本。
   - 冲突副本保留来源元信息。
   - 写入失败时保留原始数据和备份。
   - `python/main.py` 与 `build/app_main.py` 行为一致。
3. 集成验证：
   - PC 独有数据同步到 Android。
   - Android 独有数据回传到 PC。
   - 同 ID 同内容不误报冲突。
   - 同 ID 不同内容生成且只生成一份冲突副本。
   - 连续执行双向同步两次，第二次应主要为跳过或无差异。
4. 真机验证：
   - Android 真机连接 PC 实际端口。
   - 拉取、回传、双向同步均可显示预览和报告。
   - 图片同步失败时报告可读，已有数据不损坏。

## 风险与处理

| 风险 | 影响 | 处理 |
|---|---|---|
| 字段级差异做得过细 | 实现复杂、UI 变重 | 第一版只展示关键字段和数量，详情放报告 |
| 业务摘要遗漏字段 | 可能误判同内容 | 摘要函数集中维护，并用测试覆盖核心字段 |
| 图片哈希计算成本高 | 大图片同步变慢 | 第一版可选哈希，先用元数据和文件名，必要时后台补算 |
| 双端时间不可信 | 按时间合并误覆盖 | 时间只做辅助提示，不作为默认覆盖依据 |
| 后端源码入口和安装包入口分叉 | 源码测试通过但安装版异常 | `python/main.py` 和 `build/app_main.py` 同步改动并补回归 |
| 写入中断 | 数据不完整 | 写入前备份、临时文件提交、失败报告 |
