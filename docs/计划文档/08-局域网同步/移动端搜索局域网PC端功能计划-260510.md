# 移动端搜索局域网PC端功能计划-260510

> 范围：移动端设置页局域网同步区、局域网PC端搜索能力、搜索结果交互、同步前连接校验、文档同步与验证。

## 思考与分析

### Skill嗅探结果

- 本次任务是制定功能计划，已启用 `planning-with-files` 技能，采用持久化 Markdown 计划文档沉淀方案。
- 项目内已有 `.github/skills/frontend-design/SKILL.md`，但本轮不进入视觉实现，仅在计划中约束移动端交互与状态，不实际改 UI 代码。

### 当前现状

- `src/js/mobile-settings.js` 已在移动端设置页提供局域网同步区，包含“扫描 PC”“测试连接”“同步”入口。
- `src/js/lan-sync.js` 已有 `LanScanner` 和 `LanSync`：
  - `LanScanner.scan()` 会尝试通过 WebRTC 获取本机 IP，并按网段请求 `http://{ip}:8888/api/health`。
  - WebRTC 获取失败时回退扫描 `192.168.1`、`192.168.0`、`192.168.31`、`10.0.0`。
  - `LanSync.sync(ip)` 负责连接 PC、拉取 `/api/sync`、合并数据、下载图片和校验。
- PC 端后端已提供：
  - `GET /api/health`
  - `GET /api/network-info`
  - `GET /api/sync`
  - `GET /api/sync/images/{filename}`
- PC 端 Python 服务监听 `0.0.0.0:8888` 是局域网访问的基础前提。

### 问题判断

当前“扫描 PC”能力已经存在，但还不够像一个稳定的“搜索局域网 PC 端”功能：

1. 搜索过程缺少清晰的进度反馈，用户不知道正在扫哪个网段、是否仍在工作。
2. 搜索结果只展示设备列表，缺少“已验证可同步”的信号。
3. WebRTC 获取本机 IP 在 Android WebView、浏览器隐私策略或部分网络环境下可能失败。
4. 回退网段覆盖不完整，例如手机热点、公司网、路由器非典型网段下可能扫不到。
5. 未充分利用已保存的最近设备作为快速搜索候选。
6. 缺少面向搜索能力的单元测试和真机验证清单。

## 实施计划

### 目标

在移动端设置页增加一个明确、稳定、可回退的“搜索局域网 PC 端”能力，让用户无需手动查 PC IP，也能发现并选择可同步的 PC 服务。

### 功能边界

- 本次只做“移动端发现 PC 端并填入 IP”的增强。
- 不改变现有同步协议，仍通过 `8888` 端口和现有 `/api/health`、`/api/sync` 接口完成验证与同步。
- 不实现双向同步，不改变 PC 优先覆盖策略。
- 不引入后端广播服务，优先复用 HTTP 探测，保持实现简单。

### 搜索策略设计

1. 优先扫描最近设备
   - 从 `localStorage` 读取 `lan-sync-recent-devices`。
   - 先并发探测最近使用过的 IP，命中后立即展示在结果顶部。

2. 获取当前局域网网段
   - 保留现有 WebRTC 获取本机 IP 的方案。
   - 若获取到 `192.168.6.109`，则优先扫描 `192.168.6.1-254`。

3. 增强回退网段
   - 保留当前默认网段：`192.168.1`、`192.168.0`、`192.168.31`、`10.0.0`。
   - 增加常见热点和局域网候选：`192.168.43`、`192.168.50`、`192.168.100`、`172.16.0`。
   - 仍允许用户手动输入 IP，避免搜索失败时卡死。

4. 探测协议
   - 对每个候选地址请求 `GET http://{ip}:8888/api/health`。
   - 必须校验 `status === "ok"` 才认为是可用 PC 服务。
   - 可选读取 `device_name` 作为设备名称。

5. 搜索控制
   - 搜索中禁用重复搜索按钮。
   - 支持取消搜索，调用 `LanScanner.abort()`。
   - 搜索结束后恢复按钮状态。

### 移动端交互设计

移动端设置页局域网同步区域调整为：

1. 顶部状态卡
   - 默认提示：搜索同一局域网内已打开的 PC 端。
   - 搜索中：显示当前阶段，例如“正在检查最近设备”“正在扫描 192.168.6.x”。
   - 成功：显示发现数量。
   - 失败：提示手动输入 IP 的兜底方式。

2. 操作区
   - 主按钮：“搜索 PC”。
   - 搜索中主按钮变为“停止搜索”或禁用并显示加载状态。
   - 保留“测试连接”和“同步”。

3. 结果区
   - 每个结果显示设备名、IP、端口、来源：
     - 最近设备
     - 当前网段
     - 常用网段
   - 点击结果后填入 IP，并自动执行一次连接测试。
   - 已连接成功的设备显示“可同步”状态。

4. 空状态
   - 没搜到时给出明确排查项：
     - PC 端是否已打开。
     - 手机与 PC 是否在同一 Wi-Fi。
     - PC 防火墙是否允许 8888 端口。
     - 可手动输入 PC 设置页展示的同步地址。

### 技术方案

#### `src/js/lan-sync.js`

- 扩展 `LanScanner.scan()` 参数：
  - `port`
  - `timeout`
  - `subnets`
  - `recentHosts`
  - `onProgress`
- 新增内部阶段：
  - `scanRecentHosts()`
  - `detectSubnets()`
  - `scanSubnets()`

- 搜索结果去重，按发现顺序和来源排序。

#### `src/js/mobile-settings.js`

- 将“扫描 PC”文案升级为“搜索 PC”。
- 在 `handleLanScan()` 中接入进度回调。
- 搜索结果点击后：
  - 填入 IP。
  - 更新状态。
  - 自动调用连接测试逻辑，确认可用后保存最近设备。
- 搜索失败或无结果时保持手动输入路径可用。

#### `src/css/mobile.css`

- 复用现有 `.m-lan-*` 样式体系。
- 补充搜索中、可同步、失败、来源标签等状态样式。
- 确保移动端按钮和结果卡片文字不会溢出。

### 验证方案

1. 单元测试
   - 为 `LanScanner` 增加测试覆盖：
     - 最近设备优先探测。
     - 当前网段扫描。
     - 回退网段扫描。
     - 重复 IP 去重。
     - Abort 后停止搜索。
     - `/api/health` 非 `ok` 不进入结果。

2. 本机接口验证
   - 启动 PC 后端。
   - 验证 `http://本机局域网IP:8888/api/health` 返回 `200`。
   - 验证移动端开发页 `http://本机局域网IP:5173/?ui=mobile` 可访问。

3. 真机验证
   - 手机和 PC 接入同一 Wi-Fi。
   - 移动端打开设置页，点击“搜索 PC”。
   - 搜索到 PC 后点击设备，确认 IP 自动填入。
   - 点击“同步”，确认能完成数据同步和图片下载。
   - 关闭 PC 服务后再次搜索，确认失败提示清晰。

4. 回归测试
   - `npm run test`
   - `npm run build`
   - `python -m pytest python/tests -q`
   - 若打 Android 包，再执行 `npx cap sync android` 和 Android 构建流程。

## 风险与处理

| 风险 | 影响 | 处理 |
|---|---|---|
| Android WebView 无法通过 WebRTC 获取本机 IP | 无法精准定位当前网段 | 使用最近设备和常见网段回退，保留手动 IP |
| 路由器或系统限制局域网互访 | 搜索不到 PC | 提供防火墙、同 Wi-Fi、端口 8888 排查提示 |
| 扫描 254 个地址耗时偏长 | 用户等待时间长 | 并发扫描、阶段提示、支持取消 |
| 公司网络网段复杂 | 常见网段扫描不完整 | 手动输入作为稳定兜底，后续可考虑二维码配对 |
| 局域网内多台 PC 同时运行服务 | 用户可能选错设备 | 显示设备名、IP 和最近同步时间 |

## 暂不纳入本轮

- UDP 广播发现。
- mDNS 服务发现。
- PC 端二维码配对。
- Token 认证。
- 双向同步。
- 断点续传。

这些能力可以作为后续增强，避免本轮搜索功能过度工程化。
