## 一、思考与分析

### 1.1 技能与上下文嗅探

- 已检查项目根目录：未发现项目级 `SKILL.md`。
- 使用 `planning-with-files` 技能：本次先生成持久化计划文档，等待审核后再进入实现。
- 使用 `ui-ux-pro-max` 技能：设置页属于产品工具型界面，重点参考设置页可访问性、错误反馈、响应式与表单交互规范。
- 当前项目技术栈：Vite + 原生 JavaScript + 原生 CSS，PC 端与移动端分模块实现，共享 `storage.js`、`api-storage.js`、`sqlite-storage.js`、`lan-sync.js`。

### 1.2 当前功能现状

| 功能域 | PC 端现状 | 移动端现状 | 主要问题 |
|---|---|---|---|
| 主题色 | `pc-settings.js` 已有主题色选择，写入 `pc-accent` | `mobile-settings.js` 已有主题色选择，写入 `accent` | PC 与移动端存储键不一致，是否需要统一策略需明确 |
| 数据导出 | PC 与移动端均可导出 JSON | PC 与移动端均使用浏览器下载方式 | 现有导出需要升级为包含图片文件内容的完整备份，并补充失败原因展示、移动端原生文件保存兼容确认 |
| 数据导入 | PC 与移动端均可导入 JSON | 移动端导入前有简单确认 | 缺少文件结构校验、导入预览、同 ID 默认覆盖策略说明与恢复前备份提示 |
| 局域网同步 | PC 端已改为只展示服务地址、复制和刷新 | 移动端已接入扫描、手动 IP、测试连接和同步入口 | 后续仍需补最近设备快速重连和更细的导入/同步预览 |
| 存储信息 | PC 显示提示词数量和数据大小 | 移动端显示数据大小与浏览器存储估算 | PC `ApiStorage` 没有 `estimateStorageSize`，显示可能退化；统计维度不够完整 |
| 关于信息 | PC 有版本、运行环境、存储类型 | 移动端缺少完整关于区 | 版本来源固定为 meta，和 `package.json` 版本可能不同步 |
| 可访问性 | 大量按钮和色块缺少 ARIA 语义 | 移动端同样依赖视觉表达 | 错误信息缺少 `role="alert"` 或 `aria-live`，主题色不能只靠颜色表达 |

### 1.3 关键结论

1. 这不是单纯补 UI，而是要先修通设置页背后的能力边界：PC 端是 API 存储，移动端是 SQLite/Capacitor，两端同步能力不能混用。
2. 局域网同步是最大风险点，已确定职责边界：PC 端只作为局域网同步服务端，移动端作为客户端从 PC 端拉取并覆盖同步。
3. 导入导出必须补“校验、预览、确认、反馈”，导入遇到同 ID 记录时默认覆盖；导出必须包含图片文件内容，不能只导出图片引用。
4. 主题色属于低风险体验项，可先统一状态命名与视觉反馈，再做更复杂的跟随系统主题。
5. 设置页后续改动涉及前端页面、样式、同步逻辑与文档，需要同步检查 `docs/apps-code-map.md`、相关技术文档和就近模块说明。

## 二、实施计划

### Phase 0：范围决策固化

目标：把已确认的产品策略写入计划，避免实现阶段反复摇摆。

- PC 端只作为局域网同步服务端：展示本机服务信息、网络地址、端口、复制连接信息，不主动连接或拉取其他设备。
- 移动端作为局域网同步客户端：从 PC 端拉取全量数据。
- 移动端同步允许覆盖本机较新的记录，以 PC 端数据为同步基准。
- 导入数据遇到同 ID 记录时默认覆盖。
- 导出数据必须包含图片文件内容。
- 确认主题色键名是否统一，或维持 PC/移动端独立主题偏好。

### Phase 1：设置页基础体验补齐

目标：先完成低风险、可独立验证的设置项。

- 统一设置页分区结构：隐私安全、外观设置、数据管理、局域网同步、存储信息、关于。
- 主题色选择增加按钮语义、选中态文本、键盘可访问性。
- 存储信息增加提示词数量、版本数量、图片数量、数据估算大小。
- 关于信息统一展示应用名、版本号、运行环境、存储方式、数据位置说明。
- 所有错误和成功状态改为可读文本，不只依赖颜色。

### Phase 2：数据导入导出闭环

目标：让用户清楚知道导出了什么、导入了什么、失败在哪里。

- 新增导出前数据汇总：提示词、版本、图片数量、图片文件总体积。
- 导出文件名保留日期，并增加应用版本字段或备份元信息。
- 导出内容必须包含图片文件内容，可采用单文件嵌入式备份或压缩包备份；实现前优先评估 PC 与 Android 的文件保存兼容性。
- 导入前解析 JSON，校验是否为合法备份结构。
- 导入前展示预览：新增数量、同 ID 覆盖数量、图片恢复数量、无效项数量。
- 导入时默认覆盖同 ID 数据；覆盖前必须显示确认，导入失败时不能留下半导入状态。
- 导入后刷新设置页、侧边栏统计和相关列表页缓存。

### Phase 3：局域网同步修复与完善

目标：修复当前 PC 设置页同步按钮实际不可用的问题，并把同步入口按平台职责收敛。

- 在 `lan-sync.js` 中补稳定公共 API，或在设置页直接使用 `LanScanner`、`LanSync` 类，避免调用不存在的函数。
- PC 端设置页：
  - 展示本机 IP、端口、服务状态。
  - 提供复制连接信息。
  - 提供刷新网络信息。
  - 只作为局域网同步服务端，不提供主动连接其他设备的同步入口。
- 移动端设置页：
  - 新增局域网同步区域。
  - 支持自动扫描 PC 端。
  - 支持手动输入 IP。
  - 支持连接测试、同步进度、同步报告、失败重试。
  - 保存最近设备列表。
  - 同步时以 PC 端数据为准，允许覆盖本机较新的同 ID 记录。
- 同步过程用 `time.time()` 或平台等价的系统时间差思路记录耗时，不用累计 `sleep` 型计时。

### Phase 4：响应式与视觉整理

目标：保持截图中的卡通浅色风格，同时让设置项更像可操作工具界面。

- PC 端卡片间距、按钮尺寸、输入框宽度补齐桌面端扫描效率。
- 移动端保持基础类名为手机端的正向响应式规约。
- 替换关键操作中的装饰性 emoji 为已有图标资源或统一图标样式；内容说明中的吉祥物保留。
- 所有点击项补 hover、focus、disabled、loading 状态。
- 设置页长内容区域避免横向滚动，检查 320、375、768、1024、1440 宽度。

### Phase 5：文档、测试与验收

目标：改完功能后形成工程闭环。

- 更新 `docs/apps-code-map.md` 的文档落点检查。
- 更新就近模块说明文档：
  - 前端设置页：建议补充或更新 `docs/UI计划/` 下设置页相关说明。
  - 局域网同步：更新 `docs/技术文档/lan-sync-design-doc.md`。
  - API 变化：如改后端接口，更新 `docs/技术文档/api-reference.md`。
  - 移动端同步：更新 `docs/技术文档/mobile-technical-doc.md`。
  - PC 端设置：更新 `docs/技术文档/pc-technical-doc.md`。
- 增加或补充测试：
  - 导入导出格式校验单元测试。
  - `lan-sync.js` 公共 API 和错误状态测试。
  - PC/移动端设置页冒烟测试。
- 运行验证：
  - `npm run test`
  - 如涉及 Python API：`npm run test:python`
  - 如涉及 Tauri/Rust：`npm run test:rust`
  - 手动验证 PC 浏览器、Tauri、Android WebView 关键路径。

## 六、已确认决策与待确认问题

### 已确认决策

1. PC 端只作为局域网同步服务端。
2. 导入数据遇到同 ID 记录时，默认覆盖。
3. 移动端同步允许覆盖本机较新的记录。
4. 导出数据必须包含图片文件内容。
5. 备份文件加入 `backup_meta`。
6. 第一版图片完整导出采用“单 JSON 内嵌 Base64”方案。

### 待确认问题

1. 主题色偏好是否需要 PC 与移动端共用同一个 localStorage 键？
