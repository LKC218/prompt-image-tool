# 双端图片预览下载按钮计划-260522

> 范围：PC 端和移动端提示词详情页图片预览查看器。目标是在图片预览层增加下载按钮，其中 PC 端支持自定义保存位置，移动端按平台能力下载或保存到可访问目录。

## 思考与分析

### Skill 嗅探结果

- 根目录未发现额外 `SKILL.md`，本次遵循项目 `AGENTS.md` 指令集与开发规范。
- 已使用 `planning-with-files` 技能：本次先沉淀计划文档，等待确认后再进入实现。
- 已使用 `ui-ux-pro-max` 技能：下载按钮属于固定遮罩层工具按钮，设计时需避免遮挡图片、保证可点击区域和焦点状态。

### 当前代码定位

- PC 图片查看器集中在 `src/js/pc-utils.js`：
  - `ensureImageViewer()` 创建 `#pcImageViewer`、右上工具栏、复位按钮和关闭按钮。
  - `showImageViewer(src)` 目前只接收图片地址，不携带文件名、图片对象或保存策略。
  - `pc-detail.js` 的封面点击通过 `showImageViewer(imageUrls[currentImageIndex].url)` 打开预览。
- 移动端图片查看器集中在 `src/js/mobile-detail.js`：
  - `showImageViewer(pageEl, startIndex)` 创建 `.m-image-viewer-overlay`。
  - 预览层顶部目前只有关闭按钮和计数，底部展示备注，多图时显示上一张 / 下一张。
  - `imageUrls` 当前只保存 `url`、`name`、`note`。
- 现有自定义保存能力集中在 `src/js/backup-utils.js` 与后端 `/api/export-file`：
  - `showSaveFilePicker`、浏览器 `a.download`、桌面后端原生保存窗口都已有可参考实现。
  - 该链路当前只面向 JSON 备份，不能直接承担图片保存；但“前端优先文件保存选择器，桌面端回退后端原生保存窗口”的策略可以复用。
- 图标出口：
  - PC 使用 `src/js/pc-icon-assets.js` 和 `src/assets/icons/pc/`。
  - 移动端使用 `src/js/mobile-icon-assets.js` 和 `src/assets/icons/mobile/`。
  - 当前两端都没有预览下载专用本地图标，需要补充 `download.svg` 并登记到统一出口。

### 问题本质

当前用户在预览大图时只能查看，不能从预览层直接保存当前图片。PC 端要求支持自定义位置下载，不能只触发浏览器默认下载；移动端更适合一键保存当前图片，尽量走浏览器 / WebView 可用下载能力，避免引入系统级文件选择复杂度。

### 设计原则

- KISS：首版只下载“当前正在预览的图片”，不做批量下载、不做图片重命名管理器。
- 不破坏现有查看器：缩放、拖拽、关闭、多图切换保持原行为。
- PC 自定义位置优先：有 `showSaveFilePicker` 时直接保存；桌面 WebView 不支持时增加后端图片保存接口并弹出原生保存窗口。
- 移动端少打扰：按钮放在预览顶部工具区，点击后保存当前图；多图切换后下载目标同步更新。
- 文件名可预测：优先使用图片原始 `name`，没有名称时使用提示词标题、图片序号和原始扩展名生成安全文件名。

## 实施计划

### Phase 1：图片下载共享工具

1. 新增 `src/js/image-download-utils.js`，集中处理：
   - 从图片 URL 拉取 `Blob`。
   - 根据 `Content-Type` 或 URL 扩展名推断图片扩展名。
   - 生成安全文件名，避免 Windows 路径非法字符。
   - Web 标准下载：`showSaveFilePicker` 优先，失败或取消时明确返回状态。
   - 普通浏览器下载：`a.download` 兜底。
2. 保持工具为纯前端模块，不耦合 PC / 移动端页面结构。
3. 补充 `src/js/image-download-utils.test.js`，覆盖文件名清理、扩展名推断、文件选择器取消和 `a.download` 兜底。

### Phase 2：PC 端自定义位置保存

1. 调整 `showImageViewer()` 入参，从 `showImageViewer(src)` 扩展为兼容形式：
   - 仍支持旧调用：`showImageViewer(src)`。
   - 新调用支持对象：`showImageViewer({ src, filename, image })`。
2. 在 `pc-utils.js` 查看器工具栏增加下载按钮：
   - 放在复位按钮和关闭按钮之间或复位按钮之前。
   - 使用本地 `download.svg` 图标，提供 `title="下载图片"` 和 `aria-label="下载当前图片"`。
   - 点击时阻止事件冒泡，避免触发遮罩关闭。
3. PC 保存策略：
   - 优先调用 `showSaveFilePicker`，允许用户选择文件名和保存位置。
   - 若处于桌面后端环境且文件选择器不可用，调用新增后端接口弹出原生保存窗口。
   - 最后兜底为浏览器默认下载。
4. 新增后端图片保存接口：
   - `POST /api/image-download-file` 或同类语义接口。
   - 入参建议：`filename`、`sourceFile` 或 `imageUrl` 对应的安全图片文件名。
   - 后端只允许保存 `IMAGES_DIR` 内的真实图片，禁止任意路径读取。
   - 使用与 JSON 导出相同的原生保存窗口思路，默认目录指向系统下载目录，用户可自定义位置。
5. 同步 `python/main.py` 与 `build/app_main.py`，避免源码版正常、安装版缺接口。

### Phase 3：移动端预览下载

1. 在 `mobile-detail.js` 的 `.m-image-viewer-header` 增加下载按钮：
   - 左侧关闭，右侧放下载按钮，中间保留计数；或采用“关闭 + 计数 + 下载”的三段结构。
   - 按钮最小触控区域不小于 `44px`。
   - 不遮挡图片主体，不影响已有左右切换按钮。
2. 下载行为：
   - 当前环境支持浏览器下载时，使用共享工具的 `a.download`。
   - 若后续 Android WebView 需要更强保存能力，再接入 Capacitor Filesystem；首版不扩大到系统相册写入。
3. 多图切换时，下载按钮使用当前 `viewerIndex` 对应的图片 URL、名称和扩展名。
4. 点击下载后给出移动端 toast 或轻提示，区分“已开始下载 / 已保存 / 已取消 / 下载失败”。

### Phase 4：图标和样式

1. 补充本地下载图标：
   - `src/assets/icons/pc/download.svg`

2. 更新图标出口：

## 待确认决策

- PC 点击下载时是否总是弹出自定义保存窗口。建议是：PC 端预览下载默认弹出自定义保存窗口，符合“PC 端支持自定义位置下载”的明确需求。
- 移动端是否需要保存到系统相册。建议首版不做，先提供当前 WebView / 浏览器可控的一键下载，避免引入相册权限和平台差异。
- 下载文件名是否加入提示词标题。建议使用“提示词标题-图片序号.扩展名”，同时保留原图片名作为优先来源，减少重复文件名。

## 验收标准

| 场景 | 目标 |
|------|------|
| PC 单图预览 | 工具栏显示下载按钮，点击不关闭预览 |
| PC 自定义位置 | 弹出文件保存选择，保存后目标文件可打开 |
| PC 回退路径 | 文件选择器不可用时仍能通过后端或浏览器下载 |
| PC 多图切换 | 当前显示哪张图，下载哪张图 |
| 移动端单图预览 | 顶部显示下载按钮，触控区域稳定 |
| 移动端多图预览 | 左右切换后下载当前图 |
| 图片缺失 | 显示失败提示，不产生空文件 |
| 路径安全 | 后端拒绝路径穿越和非图片目录文件 |
| 文档同步 | `apps-code-map`、PC 技术文档、移动端技术文档和 API 文档按实际改动更新 |
