# 大量原图 ZIP 备份恢复实施计划

> 创建日期：2026-07-15  
> 状态：实施中（PC 导出与预检已完成）  
> 类型：新功能实装与增强  
> 范围：为 PC 与 Android 增加可长期承载大量原图的 ZIP 完整备份与恢复能力，同时保持既有 JSON 备份导入兼容。

## 一、背景与问题

当前完整备份采用 JSON 内嵌 Base64 图片的数据结构。该方案适合少量图片的单文件迁移，但不适合长期保存大量原图：

1. Base64 编码会增加约三分之一文件体积。
2. 导出、JSON 序列化、解析和导入解码会多次持有完整图片数据，造成较高内存峰值。
3. 大量高分辨率图片会放大浏览器 WebView、Python 服务和 Android 端的内存压力，可能导致导出失败、导入中断或应用崩溃。
4. 当前完整恢复缺少统一的格式版本校验、资源上限、临时恢复区和全量提交机制；图片损坏或磁盘写入失败时可能留下部分恢复结果。

本计划将完整备份升级为 ZIP 容器：业务数据继续使用 JSON，原图以原始文件写入 ZIP。旧 JSON v1 保留导入能力，不破坏已有用户备份。

## 二、目标与非目标

### 目标

- 默认提供包含原始图片的 ZIP 完整备份，避免 Base64 图片膨胀。
- PC 端与 Android 端均采用文件流处理图片，禁止在运行时一次性加载全部原图。
- 支持 PC 与 Android 的 ZIP 双向导入导出，并保持图片字节一致性。
- 导入执行预检、恢复点、临时写入、校验与原子提交；失败后保留导入前有效数据。
- 继续支持 JSON v1 备份导入，并提供显式的兼容 JSON 导出选项。
- 为损坏包、危险路径、压缩炸弹、空间不足及版本不兼容提供可理解的错误提示。

### 非目标

- 本期不移除 JSON v1 导入能力。
- 本期不实现云端同步、云备份或多设备自动合并。
- 本期不对原图重新压缩、转码或修改其内容。
- 本期不支持任意 ZIP 文件；仅接受符合本项目清单和目录规则的备份包。

## 三、备份格式设计

### 格式标识

- 格式名：`prompt-image-tool-backup`
- 新版本：`2`
- 容器扩展名：`.zip`
- 默认文件名：`prompt-image-tool-backup-YYYYMMDD-HHmmss.zip`
- JSON v1 定位：兼容格式，仅用于历史导入与主动选择的轻量导出。

### ZIP 目录结构

```text
prompt-image-tool-backup-YYYYMMDD-HHmmss.zip
├── manifest.json
├── data/
│   ├── folders.json
│   └── prompt_sets.json
└── images/
    ├── {image-id}.png
    ├── {image-id}.webp
    └── {image-id}.jpg
```

### 清单职责

`manifest.json` 应包含以下内容：

- 格式名、协议版本、创建时间、导出端与最低兼容版本。
- 分类、提示词集合、版本、图片数量和总字节数统计。
- 每张图片的 ZIP 相对路径、MIME 类型、原始字节数、SHA-256 摘要和关联版本标识。
- `missingImages`：元数据存在但原文件缺失的图片记录。
- `warnings`：可恢复但需要提示用户的问题。
- 数据文件的摘要信息，用于导入前完整性校验。

`data/prompt_sets.json` 中图片字段仅保留相对路径及业务元数据，不再保存 Data URL、Base64 数据或其临时派生字段。

### 压缩策略

- `manifest.json` 与 `data/*.json` 使用 DEFLATE 压缩。
- PNG、JPEG、WebP、GIF 等已压缩图片采用存储模式，不再重复压缩。
- ZIP 内路径统一使用正斜杠，禁止绝对路径、盘符、`..` 和反斜杠绕过。

## 四、技术方案

### PC 端

- 在 `python/main.py` 与 `build/app_main.py` 同步实现 ZIP 导出、预检和导入接口。
- 使用 Python 标准库 `zipfile`：导出通过 `ZipFile.write()` 将图片文件直接写入压缩包；导入逐条读取、校验并落盘。
- 不新增 Python 第三方压缩依赖。
- 导入过程加进程内互斥锁。恢复未结束时，其他数据写操作返回“正在恢复”状态，避免并发写入造成数据竞争。

### Android 端

- 新增原生 Capacitor `BackupZipPlugin`，以 Java `ZipOutputStream` 和 `ZipInputStream` 流式操作应用私有目录中的原图文件。
- JavaScript 层只传递导出目标、导入文件 URI、进度和结果元数据，禁止传递原图 Base64。
- 插件提供进度、取消、空间不足、格式错误、校验失败和恢复失败等状态。
- SQLite 继续存储业务元数据，图片文件继续存放于应用数据目录；ZIP 仅作为跨端交换与恢复容器。

### 前端接口与交互

- 扩展 `ApiStorage`，增加 ZIP 导出、ZIP 预检、ZIP 导入和进度查询能力。
- 在 `backup-utils.js` 区分 ZIP 完整备份与 JSON 兼容备份，禁止根据文件体积静默切换格式。
- PC 与移动端设置页提供以下入口：
  1. `导出完整备份`：ZIP，包含原图，默认推荐。
  2. `导出兼容备份`：JSON，用于旧版本或少量数据迁移。
  3. `导入备份`：接受 `.zip,.json`，按格式分流。
- ZIP 导入前展示格式、版本、分类数、提示词数、图片数、文件大小、缺失图片与覆盖影响；用户确认后才开始恢复。

## 五、恢复事务与安全边界

### 恢复状态机

```text
预检 → 创建恢复点 → 临时解压 → 清单与文件校验 → 数据切换 → 图片切换 → 清理
```

1. 预检 ZIP 目录、清单字段、版本、文件类型、文件数、压缩比、单文件大小与预计解压总量。
2. 为当前数据创建可回退的恢复点。
3. 在正式数据目录同盘的临时目录解压图片和数据文件。
4. 校验每张图片的路径、大小、MIME 类型与 SHA-256；校验业务数据引用关系。
5. 所有校验成功后，原子替换业务数据与图片目录。
6. 替换成功后清理临时目录和无引用的旧图片；失败则恢复恢复点并清理临时文件。
7. 应用启动时检查未完成恢复状态，自动清理临时目录或执行明确的恢复流程。

### 安全限制

- 拒绝 ZIP Slip：禁止 `..`、绝对路径、Windows 盘符、反斜杠路径逃逸和不在白名单中的文件。
- 限制 ZIP 条目总数、图片总数、单文件大小、压缩包大小、预计解压总量和最大压缩比。
- 仅允许 `manifest.json`、`data/folders.json`、`data/prompt_sets.json` 与 `images/` 白名单路径。
- 图片仅允许项目已支持的扩展名与 MIME 类型。
- 对格式错误、版本不支持、校验不一致、存储空间不足和取消操作提供独立错误码与用户提示。

- 将 Android ZIP 导入导出接入现有 SQLite 元数据与图片目录。

- 完成单元、接口、原生插件、跨端端到端和大文件压力测试。
- 首个发布周期默认 ZIP 完整备份，仍显示 JSON 兼容导出。
- 观察实际导入失败率、文件规模和兼容反馈后，再评估是否弱化 JSON 导出入口。

## 八、测试与验收

### 自动化测试

- ZIP 结构、清单字段、JSON/ZIP 格式识别和图片 SHA-256 一致性。
- 缺失图片、损坏 ZIP、损坏图片、未知版本、无效 MIME 类型与错误清单。
- ZIP Slip、超限条目、超限解压大小、异常压缩比和路径白名单拦截。
- 导入中断、磁盘空间不足、校验失败和数据切换失败后的回滚。
- 恢复锁对并发导入、保存和编辑操作的保护。
- Android 原生插件的大图、多图、取消、低存储和恢复中断场景。

### 跨端验收

1. PC 导出 ZIP 后可在 Android 导入，分类、提示词、版本和原图均可用。
2. Android 导出 ZIP 后可在 PC 导入，原图 SHA-256 与导出前一致。
3. 历史 JSON v1 可在 PC 和 Android 正常导入。
4. 任一导入失败场景下，导入前数据、图片与可用恢复点保持完整。
5. 大量原图导入导出过程中，不出现将全部原图加载进 JavaScript 或 Python 内存的实现。

## 九、风险与决策

- Android 原生插件是大图低内存处理的必要条件，不能用 JavaScript 压缩库替代。
- ZIP 格式升级涉及 PC 后端、Android 原生层、前端设置页与存储抽象，必须按阶段交付并保持双格式兼容。
- SHA-256 会增加导出与导入耗时，但为跨端可验证恢复提供必要完整性保障，默认保留。
- 恢复点会占用额外磁盘空间；实现时需要在导入前校验可用空间，并限制历史恢复点数量。
- 默认覆盖或合并策略会影响数据安全，实施前需固定产品语义，并在导入确认页完整展示影响范围。
