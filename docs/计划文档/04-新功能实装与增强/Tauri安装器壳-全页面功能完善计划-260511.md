# Tauri安装器壳-全页面功能完善计划

## 思考与分析

本计划使用 `planning-with-files` 技能制定，并沿用当前项目内已经形成的 installer-shell 约束：一个 Tauri 窗口、一个 `index.html` 壳体、多个 `installer-shell/src/pages/*.js` 页面模块，由 `installer-shell/src/main.js` 统一管理状态与步骤流转。

当前 `docs/设计文档/Tauri-安装器壳-UI-设计规则-260510.md` 规定的是 7 步安装体验：欢迎页、许可协议页、选择安装位置页、选择附加任务页、准备安装页、正在安装页、安装完成页。现有源码已经实现 6 个页面模块，缺少独立的“准备安装页”；安装位置、附加任务、正在安装和完成页已经有 UI，但功能还存在原型边界：

- 安装位置页的路径输入为只读，浏览按钮只更新提示文案，尚未接入真实目录选择、路径校验和错误提示。
- 附加任务页保存了桌面快捷方式、开始菜单快捷方式和自动检查更新的 UI 状态，但当前 NSIS 脚本默认创建桌面和开始菜单快捷方式，尚未按选择项动态处理。
- 正在安装页已接入 `install_with_nsis`，但进度仍是阶段式状态，缺少可复用的安装结果对象、失败重试入口和更清晰的错误诊断展示。
- 完成页只处理立即启动和桌面快捷方式收尾；开始菜单快捷方式、自动检查更新等选择项还没有闭环。
- 当前 `installer-shell/src-tauri/src/main.rs` 已有 `install_with_nsis`、`launch_installed_app`、`apply_desktop_shortcut` 三个命令，后续应优先在这个窄边界内扩展，不重做安装器架构。

因此，本轮目标不是重新设计视觉，而是把所有页面从“可展示”推进到“可交互、可校验、可真实影响安装结果、可失败兜底、可测试验收”的工程闭环。

## 实施计划

### 1. 基线与范围确认

- 保持 `installer-shell/src/index.html` 只做窗口壳、标题栏、品牌区和页面挂载点。
- 保持页面内容在 `installer-shell/src/pages/` 内拆分，不引入多 HTML、前端路由或大型框架。
- 保持 `styles.css` 为共享样式入口，新增页面优先复用现有书本页布局基准。
- 先确认 `build/installer.nsi`、`installer-shell/src-tauri/src/main.rs` 和 `scripts/build_installer_shell_package.py` 的现有职责，避免前端宣称 NSIS 尚未支持的能力。

### 2. 补齐安装步骤模型

- 新增 `installer-shell/src/pages/prepare-install.js`，实现“准备安装页”。
- 在 `main.js` 中把步骤调整为 7 步：`welcome -> license -> install-location -> install-tasks -> prepare-install -> installing -> complete`。
- 准备安装页展示安装目录、附加任务、即将执行动作和安装包来源状态。
- “准备安装页”的主按钮使用“开始安装”，只有点击后才进入正在安装页并触发 `install_with_nsis`。
- 左侧步骤栏、页码文案和设计规则文档中的步骤编号统一为 7 步口径。

### 3. 完善安装位置页功能

- 将安装路径状态从单纯字符串升级为结构化状态：原始输入、展开后路径、校验状态、错误文案。
- 接入真实路径校验：不能为空、不能只有盘符、不能包含非法字符、父目录不可写时给出明确提示。
- 浏览按钮优先接入 Tauri 侧目录选择能力；若引入目录选择依赖会显著扩大构建风险，则先允许用户编辑路径并保留浏览按钮的明确“暂不可用”状态。
- 调用 NSIS 前始终使用 Rust 侧展开后的真实路径，不把 `%LOCALAPPDATA%` 原样传给安装核心。
- 空间展示先采用保守口径：如果无法低成本读取真实磁盘剩余空间，就显示“需要约 256 MB”，不伪造精确值。

### 4. 完善附加任务页功能

- 桌面快捷方式：继续由完成收尾命令按最终选择保留或删除。
- 开始菜单快捷方式：新增对应收尾命令，按最终选择保留或删除开始菜单目录中的主程序和卸载器快捷方式。
- 自动检查更新：先核对主程序是否已有开机或启动后检查更新能力；若没有现成能力，不写入系统自启动项，只把 UI 文案调整为“安装后在应用内开启更新提醒”，避免误导。
- 附加任务状态需要贯穿准备安装页、正在安装页和完成页，不能只停留在当前页面内。

### 5. 完善正在安装页功能

- 保持阶段式进度，不假装精确文件复制百分比。
- 把安装过程拆成清晰阶段：准备安装核心、执行 NSIS、校验主程序和卸载器、执行收尾任务、完成。
- `install_with_nsis` 返回更完整的结果：安装器路径、目标目录、退出码、主程序存在、卸载器存在、错误类型。
- 失败时停留在正在安装页，显示失败发生在哪个阶段，并提供“重试”和“返回修改设置”两个可理解操作。
- 安装中禁止重复触发安装；关闭窗口必须保留确认提示。

### 6. 完善完成页功能

- 完成页显示最终安装目录和实际执行过的附加任务，不只显示笼统成功文案。
- “立即启动软件”调用 `launch_installed_app`，失败时不直接关闭窗口，而是显示可理解状态。
- “创建桌面快捷方式”与“开始菜单快捷方式”若在完成页保留开关，需要与附加任务页状态一致，避免两个页面各自为政。
- 点击完成前执行收尾命令，全部成功后关闭安装器壳；失败时允许用户重试或手动关闭。

### 7. Tauri 与 NSIS 边界修正

- 优先通过 Tauri Rust 命令做安装前校验和安装后收尾，减少改动 NSIS 脚本的范围。
- 如果需要让 NSIS 原生支持可选快捷方式，再单独评估命令行参数或临时配置文件方案，并同步 `build/installer.nsi` 与构建脚本预检规则。
- 保持 `embedded-installer` feature 的正式打包路径不变，确保发布产物双击后第一屏仍是自定义安装器壳 UI。

### 8. 验证与文档同步

- 基础语法验证：

```powershell

Get-ChildItem installer-shell/src/pages -Filter *.js | ForEach-Object { node --check $_.FullName }
cargo check --manifest-path installer-shell/src-tauri/Cargo.toml
```

- 页面预览验证：逐页打开 `step=welcome/license/install-location/install-tasks/prepare-install/installing/complete`，核对布局、按钮状态和错误提示。
- Tauri 运行态验证：真实点击完整流程，确认安装只在准备页点击“开始安装”后触发。
- NSIS 闭环验证：安装到 `%LOCALAPPDATA%\PIM-Test`，检查主程序、卸载器、桌面快捷方式、开始菜单快捷方式、完成页启动、卸载清理。
- 如修改源码结构，同步 `docs/apps-code-map.md` 和 `docs/设计文档/Tauri-安装器壳-UI-设计规则-260510.md`。
- 如沉淀出可复用经验，同步 `docs/项目开发经验/项目开发经验.md` 或就近安装包经验文档。

## 审核点

本计划需要先确认两个实现边界，再进入代码实施：

- “自动检查更新”是否必须在安装器壳内真正落地为系统级开机任务；当前建议不做系统自启动，只在已有应用能力内对齐。
- “浏览安装目录”是否允许引入最小 Tauri/Rust 依赖；若不引入依赖，则先支持用户编辑路径与 Rust 校验。
