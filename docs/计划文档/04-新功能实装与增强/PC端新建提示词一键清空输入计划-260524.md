### 问题定义

工程师希望在 PC 端“新建提示词”页面增加一个一键清除输入内容的按钮，并且清除一键上传或导入预填的图片。

这里的关键边界是：

1. “清空”应作用于新建页当前未保存编辑态，而不是删除已经保存的提示词数据。
2. 按钮需要清掉用户当前页可见的输入内容，包括标题、正向提示词、负向提示词、标签、分类、比例和图片预览。
3. prompt-image-tool JSON 导入联动会自动预填标题、提示词和图片，这些也属于当前新建页未保存输入内容，应纳入一键清空范围。
4. 新建页已有本地草稿恢复，如果只清 UI 不清草稿，刷新或切页回来会再次恢复旧内容，因此必须同步清理 `pc-editor-draft:new`。
5. 编辑模式涉及已保存数据，直接“清空全部”容易被误解为清掉已保存版本内容；第一阶段建议只在新建模式显示，编辑模式暂不显示。

### 设计原则

1. KISS：优先在 `pc-editor.js` 内增加一个小型重置函数，复用 `createDefaultFormData()`、`clearEditorDraft()`、`renderEditorContent()` 和 `setupEditorEvents()`。
2. 人可控：一键清空属于高风险动作，点击后必须弹出确认弹窗，文案明确“会清除当前未保存输入和图片预览”。
3. 低视觉权重：按钮放在顶部保存按钮左侧或同组操作区，采用次级/浅色样式，避免和“保存”抢主操作。
4. 不破坏现有局部清空：正向/负向提示词旁边的“清空”按钮继续保留，用于单字段快速清理。
5. 数据安全：清空只影响当前新建页本地状态和新建草稿，不调用后端删除图片，不修改 `prompt_sets.json`。

## 实施计划

### Phase 1：明确清空范围与交互入口

1. 入口位置：
   - 在 PC 新建提示词欢迎横幅右侧操作区新增“清空”按钮。
   - 按钮位于“保存”按钮左侧，视觉权重低于保存按钮。
   - 使用现有 PC 本地图标体系，优先复用 `pcIcon('x')` 或新增更贴切的清空图标，不使用 Emoji 或字符图标。
2. 显示策略：
   - 第一阶段仅在新建模式显示。
   - 编辑模式暂不显示整页清空按钮，避免误清已保存提示词内容；如后续需要编辑模式支持，另行确认是否只清未保存修改还是清空版本字段。
3. 按钮状态：
   - 当页面没有任何可清空内容时可禁用或保持可点但提示“当前没有可清空内容”。
   - 推荐实现为禁用态，依据 `formData` 与 `importedImages` 计算是否存在输入内容。

### Phase 2：实现新建页一键清空逻辑

1. 新增状态判断函数，例如 `hasEditorInputContent()`：
   - 标题 `formData.name` 非空。
   - 正向提示词、负向提示词、备注等版本字段非空。
   - 标签数组非空。
   - 分类 `folderId` 非空。
   - 比例不是默认 `1:1`。
   - `importedImages` 非空。
   - `versions[0].images` 非空时，在新建页也按可清空图片处理。
2. 新增执行函数，例如 `clearNewPromptEditor(pageEl)`：
   - `formData = createDefaultFormData()`。
   - `importedImages = []`。
   - `hasUnsavedChanges = false` 或按产品语义设置为已清空后的干净态。
   - 调用 `clearEditorDraft()` 清掉 `pc-editor-draft:new`。
   - 重新渲染 `renderEditorContent(pageEl)` 并重新绑定 `setupEditorEvents(pageEl)`。
   - Toast 提示“已清空当前输入”。
3. 确认弹窗：
   - 点击按钮后先判断是否有内容。
   - 有内容时弹出 `showConfirmModal('确定清空当前未保存内容和图片预览吗？', ...)`。
   - 确认后再执行清空。

### Phase 3：覆盖图片来源与草稿恢复链路

1. 本地点击上传图片：
   - 清空后 `importedImages` 置空，图片预览区回到空态。
2. 剪贴板粘贴图片：
   - 与本地上传同属 `importedImages`，同样被清空。
3. prompt-image-tool JSON 导入预填图片：
   - 导入预填后图片也进入 `importedImages`，同样被清空。
4. 新建草稿：
   - 清空时同步删除 `pc-editor-draft:new`，避免页面切换后恢复被清空前内容。
5. 文件选择框残留：
   - 当前文件选择框是临时 `input`，不需要额外持久化清理；如实现后发现浏览器保留选择状态，再补充清理。

### Phase 4：样式与可访问性

1. 在 `pc.css` 中增加或复用按钮样式：
   - 例如 `.pc-editor-clear-all-btn`。
   - 固定高度、内边距和图标尺寸，与保存按钮同组但视觉更轻。
   - Hover、focus-visible、disabled 状态完整。
2. 响应式处理：
   - 在 1366 x 768 默认窗口下按钮不挤压保存按钮。
   - 在窄宽度下允许按钮文本隐藏或按钮组换行，但不得遮挡标题、插画和保存按钮。
3. 可访问性：
   - 按钮设置 `type="button"`。
   - 图标有 `aria-hidden` 或通过按钮文字表达语义。
   - 禁用态颜色仍有足够可辨识度。

## 风险与处理

| 风险 | 影响 | 处理方式 |
|---|---|---|
| 清空后草稿再次恢复 | 用户以为已清空，返回页面却恢复旧内容 | 清空时必须调用 `clearEditorDraft()`，并确认新建草稿键为 `pc-editor-draft:new`。 |
| 编辑模式误清已保存图片 | 可能造成用户误解或误保存空内容 | 第一阶段只在新建模式显示一键清空按钮。 |
| prompt-image-tool JSON 导入预填后无法一次清掉图片 | 导入链路体验不完整 | 将导入预填图片纳入 `importedImages` 清空范围，手工验证导入预填场景。 |
| 清空动作没有确认 | 误触导致输入丢失 | 使用 `showConfirmModal()` 二次确认。 |
| 顶部按钮拥挤 | 1366 或窄宽度下影响保存按钮 | 清空按钮使用次级小按钮样式，必要时在窄宽度下隐藏按钮文字或换行到按钮组内。 |
