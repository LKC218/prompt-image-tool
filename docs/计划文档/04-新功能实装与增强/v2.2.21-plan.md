## 一、版本目标

v2.2.21 聚焦于**同步体验增强**、**图片存储优化**和**暗色模式视觉一致性**三个方向，提升移动端与 PC 端的协作效率与视觉品质。

## 二、任务总览与执行顺序

| 优先级 | 编号 | 任务名称 | 核心原因 | 预计影响范围 |
|--------|------|----------|----------|-------------|
| **P0** | T1 | 暗色模式顶栏图标与全局按钮配色一致性修复 | 视觉缺陷，暗色模式下图标/按钮不可见或风格不统一，影响基础使用体验 | `main.css`、`index.html`、SVG 图标文件 |
| **P1** | T2 | 移动端同步 — 局域网自动发现 PC 端 | 功能增强，当前需手动输入 IP，操作门槛高，易出错 | `lan-sync.js`、`app.js`、`python/main.py` |
| **P1** | T3 | 同步进度条 UI 增强 — 进度百分比、已完成/总数量、预估剩余时间 | 体验增强，当前进度反馈不足，大量图片同步时用户无法判断剩余时间 | `app.js`、`lan-sync.js`、`main.css`、`index.html` |
| **P2** | T4 | 图片导入优化 — 自动转换为 JPG 格式并压缩 | 存储优化，PNG/WebP 等格式占用空间大，JPG 压缩可显著减少存储和同步传输量 | `app.js`、`sqlite-storage.js`、`api-storage.js`、`python/main.py` |

**执行顺序：** `T1 → T2 → T3 → T4`

**排序逻辑：**

1. **视觉缺陷优先（T1）** — 暗色模式图标不可见是基础体验缺陷，且修改范围集中在 CSS/HTML/SVG，不影响业务逻辑，风险最低，适合首先修复
2. **同步发现先于进度增强（T2 → T3）** — 自动发现 PC 端是同步流程的前置体验优化，先降低操作门槛，再增强进度反馈，两者共享同步弹窗 UI，按流程顺序实施可避免 UI 返工
3. **图片优化最后（T4）** — 图片格式转换涉及存储层变更（PC 端 Python 后端 + 移动端 SQLite/Filesystem），影响图片读写全链路，需在同步功能稳定后再实施，避免与同步功能变更交叉

## 三、T1 — 暗色模式顶栏图标与全局按钮配色一致性修复（P0）

### 3.1 问题描述

| 现象 | 根因分析 |
|------|----------|
| 暗色模式下顶栏按钮图标（导出、导入、同步、主题切换、帮助）显示为黑色，在深色背景上几乎不可见 | SVG 图标文件使用 `stroke="currentColor"`，但通过 `<img>` 标签加载时无法继承父元素的 CSS `color`，`currentColor` 回退为浏览器默认黑色 |
| 暗色模式下部分按钮/控件配色不统一，不符合暗色风格 | 部分硬编码颜色值未使用 CSS 变量，或 CSS 变量在暗色模式下对比度不足 |

### 3.2 修复方案

**核心策略：将 `<img>` 引用 SVG 改为内联 SVG（`<svg>` 标签），使 `currentColor` 能正确继承父元素 CSS `color`。**

#### F1.1 顶栏按钮图标改为内联 SVG

- 将 [index.html](file:///d:/Work/git/prompt-image-tool/src/index.html) 中 Header 区域的 `<img src="./assets/icons/xxx.svg" class="btn-icon">` 替换为对应的内联 `<svg>` 标签
- 涉及图标：`export.svg`、`import.svg`、`sync.svg`、`sun.svg`/`moon.svg`、`help.svg`
- 内联 SVG 保留 `class="btn-icon"` 和 `stroke="currentColor"` 属性
- 确保 `currentColor` 继承 `.btn` 的 `color: var(--text)`（暗色模式为 `#e0e0e0`，亮色模式为 `#1a1a1a`）

#### F1.2 全局排查硬编码颜色

- 在 `main.css` 和 `responsive.css` 中搜索所有 `#000`、`#fff`、`black`、`white`、`rgb(` 等硬编码颜色值
- 将硬编码颜色替换为对应的 CSS 变量：
  - 白色文字/图标 → `var(--text)` 或 `currentColor`
  - 黑色背景/边框 → `var(--bg)` 或 `var(--border)`
  - 白色背景 → `var(--surface)` 或 `var(--surface2)`
- 重点关注：`.btn-primary` 的 `color: #fff`（应改为 `color: var(--btn-primary-text, #fff)`，因主色按钮在双主题下均为白色文字，可保留但加变量注释）

#### F1.3 暗色模式全局按钮配色一致性审查

- 审查所有按钮变体在暗色模式下的视觉表现：
  - `.btn`（默认按钮）：背景 `--surface2`、边框 `--border`、文字 `--text` — ✅ 已使用变量
  - `.btn-primary`（主要按钮）：背景 `--accent`、文字 `#fff` — 需确认暗色模式下对比度
  - `.btn-danger`（危险按钮）：边框/文字 `--danger` — ✅ 已使用变量
  - `.copy-btn`（复制按钮）：边框/文字 `--accent` — ✅ 已使用变量
  - `.btn-sm`（小号按钮）：继承 `.btn` 样式 — ✅
- 审查所有图标按钮（含详情页操作按钮、右键菜单图标）在暗色模式下的可见性
- 确保所有 `stroke="currentColor"` 或 `fill="currentColor"` 的 SVG 图标在暗色模式下正确继承文字颜色

#### F1.4 亮色模式回归验证

- 内联 SVG 的 `currentColor` 在亮色模式下应继承 `--text: #1a1a1a`（深色文字），确保图标在浅色背景上清晰可见

## 四、T2 — 移动端同步：局域网自动发现 PC 端（P1）

### 4.1 当前状态

- 移动端同步需手动输入 PC 端 IP 地址（如 `192.168.1.100`）
- PC 端已实现 `/api/network-info` 接口返回本机 IP
- PC 端 Header 区域已显示本机 IP 供用户参考
- 无自动发现机制，用户需手动查看 PC 端 IP 并在手机上输入

### 4.2 功能设计

**核心目标：移动端点击同步时，自动扫描局域网内运行中的 PC 端实例，提供可选择的设备列表。**

#### 方案：HTTP 轮询扫描 + 手动输入回退

**实现思路：**

1. 移动端获取本机 IP 地址（通过 Capacitor Network 插件或 `/api/network-info` 代理）
2. 根据本机 IP 推算局域网网段（如 `192.168.1.x`）
3. 并发请求网段内所有 IP 的 `http://{ip}:8888/api/health` 接口
4. 收集响应成功的 IP，构建可同步设备列表
5. 用户从列表中选择目标 PC 端，或手动输入 IP

**UI 交互流程：**

```
点击「同步」按钮
  → 打开同步弹窗
  → 自动开始扫描（显示扫描动画 + "正在搜索局域网内的 PC 端..."）
  → 扫描结果：
     ├── 找到设备 → 显示设备列表（IP + 连接状态标识）
     │              用户点击选择 → 自动填入 IP → 可直接开始同步
     └── 未找到设备 → 显示"未发现设备"提示 + 手动输入框
  → 手动输入框始终可见（作为回退方案）
```

#### F2.1 新增局域网扫描模块

- 在 `src/js/lan-sync.js` 中新增 `LanScanner` 类
- 核心方法：
  - `getLocalSubnet()` — 获取本机 IP 所在网段
  - `scan(subnet, port, timeout)` — 并发扫描网段内所有 IP 的指定端口
  - `checkHost(ip, port)` — 单个 IP 健康检查（`GET /api/health`，超时 1.5s）
- 扫描策略：
  - 并发数限制：同时最多 20 个请求，避免网络拥塞
  - 超时时间：单个请求 1.5s
  - 扫描范围：C 类网段（`192.168.x.1` ~ `192.168.x.254`），约 254 个 IP
  - 预计耗时：254 / 20 × 1.5s ≈ 19s（最坏情况），通常 5-10s 可发现目标

## 六、T4 — 图片导入优化：自动转换为 JPG 格式并压缩（P2）

### 6.1 当前状态

- 图片导入流程：`processImageFiles()` → `readFileAsDataURL()` → `storage.uploadImage()`
- 移动端 `uploadImage()`：根据 DataURL header 判断格式（png/jpg/webp），原样写入 Filesystem
- PC 端 `uploadImage()`：将 DataURL 原样发送到 Python 后端，后端解码后原样保存
- 无格式转换和压缩处理，PNG/WebP 等大格式图片占用大量存储空间

### 6.2 优化设计

**核心目标：导入图片时自动转换为 JPG 格式，在保持最大视觉无损的前提下压缩文件体积。**

#### F4.1 前端图片格式转换与压缩

- 在 `processImageFiles()` 中，`readFileAsDataURL()` 之后增加图片处理步骤：
  1. 将图片加载到 `<canvas>` 元素
  2. 以 `canvas.toBlob('image/jpeg', quality)` 导出为 JPG
  3. 压缩质量参数：`quality = 0.92`（经验值，肉眼几乎无法察觉质量损失，体积可减少 60%-80%）
  4. 对比原始大小和压缩后大小，若压缩后更大（极小概率），则保留原始格式
- 实现函数：
  ```javascript
  async function compressImageToJpeg(dataUrl, quality = 0.92) {
    return new Promise((resolve) => {
      const img = new Image();
      img.onload = () => {
        const canvas = document.createElement('canvas');
        canvas.width = img.naturalWidth;
        canvas.height = img.naturalHeight;
        const ctx = canvas.getContext('2d');
        ctx.drawImage(img, 0, 0);
        canvas.toBlob((blob) => {
          resolve(blob);
        }, 'image/jpeg', quality);
      };
      img.src = dataUrl;
    });
  }
  ```
- **透明通道处理**：若原始图片包含 Alpha 通道（PNG 透明图），转换前在 Canvas 上先填充白色背景，避免透明区域变黑：
  ```javascript
  ctx.fillStyle = '#FFFFFF';
  ctx.fillRect(0, 0, canvas.width, canvas.height);
  ctx.drawImage(img, 0, 0);
  ```

#### F4.2 移动端存储适配

- 修改 `sqlite-storage.js` 的 `uploadImage()` 方法：
  - 接收压缩后的 Blob 而非原始 DataURL
  - 将 Blob 转为 Base64 后写入 Filesystem
  - 文件扩展名统一为 `.jpg`
- 或：在 `processImageFiles()` 中完成压缩后，将 Blob 转回 DataURL（`image/jpeg` 格式），保持 `uploadImage()` 接口不变

#### F4.3 PC 端存储适配

- 修改 `api-storage.js` 的 `uploadImage()` 方法：
  - 发送压缩后的 JPG DataURL 到 Python 后端
- Python 后端 `main.py` 的图片保存逻辑：
  - 解码 DataURL 时识别 `image/jpeg` 格式
  - 保存为 `.jpg` 文件
  - 无需额外修改，后端已支持任意格式的 DataURL 解码

#### F4.4 已有图片兼容性

- 已存储的 PNG/WebP 图片不做迁移，保持原格式
- 仅新导入的图片应用 JPG 转换
- `getImageUrl()` 方法无需修改，已支持按文件扩展名返回图片

#### F4.5 压缩效果验证

- 准备测试图片集：PNG 截图（~2MB）、PNG 透明图（~500KB）、WebP 图片（~800KB）、已压缩 JPG（~200KB）
- 对比压缩前后文件大小和视觉质量
- 预期效果：
  - PNG 截图 → JPG：体积减少 70%-85%，视觉无损
  - PNG 透明图 → JPG（白底）：体积减少 60%-75%，透明区域变白底
  - WebP → JPG：体积可能略增，需对比后决定是否转换
  - 已压缩 JPG → JPG：体积基本不变或略减，可跳过二次压缩

### 6.3 涉及文件

## 七、执行依赖关系

```
T1 (暗色模式图标修复) ──→ 无依赖，可立即开始

T4 (图片 JPG 压缩)  ──→ 无强依赖，可与 T2/T3 并行，但建议在 T3 之后（同步传输量受图片体积影响，压缩后同步进度更准确）
```

| 批次 | 任务 | 说明 |

## 九、风险提示

| 风险项 | 影响范围 | 应对措施 |
|--------|---------|---------|
| 内联 SVG 增加 HTML 体积 | `index.html` 文件变大 | SVG 图标均为单行代码，5 个图标增加约 500 字节，可接受 |
| 局域网扫描耗时 | 移动端用户体验 | 扫描在后台异步进行，用户可随时手动输入 IP 跳过扫描 |
| 局域网扫描可能被防火墙拦截 | 自动发现功能失效 | 提供手动输入作为回退方案，扫描失败不影响核心同步功能 |
| JPG 压缩导致透明通道丢失 | PNG 透明图导入后白底 | 在 UI 中提示用户透明图片将转为白底，或提供"保留原始格式"选项 |
| JPG 二次压缩导致质量下降 | 已压缩 JPG 图片 | 对比压缩前后体积，若压缩后更大则保留原始格式 |
| 同步进度预估时间不准确 | 用户体验 | 初期预估波动较大，随进度推进趋于准确；显示"预计剩余"而非"精确剩余" |
