# PC端版本更新安装保留JSON提示词信息计划

> 创建日期：2026-05-25  
> 状态：已实施，待真实安装升级验收  
> 范围：PC 核心安装包、Tauri 安装器壳、Python 数据目录、JSON 备份导入导出和升级安装验收。  
> 不在本计划内：移动端数据迁移、云同步、账号体系、数据库重构。

## 思考与分析

### 技能与上下文嗅探

- 已使用 `planning-with-files` 技能，将本次方案沉淀为持久计划文档。
- 仓库根目录未发现额外 `SKILL.md`。
- 已参考 PC 安装包工作流，重点核对 `build/app_main.py`、`python/main.py`、`build/installer.nsi`、`installer-shell/src-tauri/src/main.rs`、`src/js/api-storage.js` 和现有 PC 技术文档。

### 当前事实

- PC 提示词主体数据保存在 `data/prompt_sets.json`，分类保存在 `data/folders.json`，参考图片保存在 `data/images/`。
- `python/main.py` 和 `build/app_main.py` 当前都优先把 `data/` 放在应用目录下；应用目录不可写时才回退到 `~/.prompt-image-tool`。
- NSIS 默认安装目录是 `$LOCALAPPDATA\PromptImageManager`，卸载逻辑目前使用 `RMDir /r "$INSTDIR"`。
- Tauri 安装器壳调用 NSIS 静默安装，不主动卸载旧版本；如果是同目录覆盖安装，旧 `data/` 理论上会保留，但如果用户或安装流程先卸载再安装，`$INSTDIR\data` 会被一起删除。
- 完整备份 JSON 已能通过 `/api/export` 和 `/api/export-file` 导出，图片会以内嵌 Data URL 进入备份；导入会按同 ID 覆盖恢复。

### 核心风险

| 风险 | 触发条件 | 后果 | 优先级 |
|------|----------|------|--------|
| 数据跟随安装目录被卸载删除 | 用户先卸载旧版本，或升级流程未来改为先卸载再安装 | `prompt_sets.json`、`folders.json`、`images/` 丢失 | 高 |
| 安装路径变更导致旧数据不可见 | 新版本安装到不同目录 | 新版本像空白应用，用户误以为数据丢失 | 高 |
| 只保护 JSON 不保护图片 | 仅复制 `prompt_sets.json` / `folders.json` | 提示词在，但参考图片失效 | 中 |
| 安装器壳缺少升级前快照 | 安装失败、中断、杀进程 | 旧数据缺少可恢复点 | 中 |
| 源码后端与安装包后端策略不一致 | 只改 `python/main.py` 或只改 `build/app_main.py` | 开发版正常、安装版仍丢数据 | 高 |

## 实施计划

### Phase 0：确认数据保护策略

1. 明确产品规则：版本更新安装默认保留用户数据，卸载应用默认也保留用户数据，只删除程序文件和快捷方式。
2. 将“删除用户数据”作为未来可选的显式动作，不放进默认卸载路径。
3. 数据保护范围包含：
   - `prompt_sets.json`
   - `folders.json`
   - `images/`
   - `backups/`
   - `sync-device.json`
   - `*.bak`

### Phase 1：把用户数据移出安装目录

1. 新增统一数据根目录解析规则，Windows 安装版优先使用 `%APPDATA%\PromptImageManager\data` 或等价的用户级稳定目录。
2. 保留开发模式的低摩擦行为：源码运行仍可使用 `python/data`，测试可通过环境变量指定临时数据目录。
3. 同步修改 `python/main.py` 与 `build/app_main.py`，避免源码后端和安装包后端分叉。
4. 给数据目录解析增加日志输出，安装版启动时能在 `app.log` 看到真实 `DATA_DIR`。

### Phase 2：旧数据自动迁移

1. 启动时检查旧安装目录 `data/` 是否存在。
2. 若新用户数据目录为空且旧目录存在，则复制旧目录到新目录，不直接移动，降低迁移失败风险。
3. 若新目录已有数据，则不覆盖，只记录旧目录位置和迁移跳过原因。
4. 写入迁移标记文件，例如 `data-migration.json`，记录来源、目标、时间、文件数量和结果。
5. 迁移必须覆盖 JSON、图片、备份和 `.bak` 文件。

### Phase 3：安装器升级前快照

1. 在安装器壳开始调用 NSIS 前增加数据快照步骤。
2. 快照同时扫描：
   - 新用户数据目录
   - 旧安装目录 `data/`
3. 将快照保存到用户级目录，例如 `%APPDATA%\PromptImageManager\update-backups\YYYYMMDD-HHMMSS\`。
4. 快照内容使用原始目录复制，不依赖后端在线，避免后端未启动时无法备份。
5. 快照完成后再进入安装；快照失败时给出明确警告，并允许用户返回或继续。

### Phase 4：调整卸载脚本边界

1. NSIS 卸载仅清理程序安装目录、快捷方式和注册表卸载项。
2. 不删除 `%APPDATA%\PromptImageManager\data`。
3. 如果旧版本仍有 `$INSTDIR\data`，卸载前先尝试复制到用户数据目录或更新备份目录，再删除安装目录。
4. 在构建文档中明确：普通卸载不等于清空用户数据。

### Phase 5：导出 JSON 兜底与设置页提示

1. 保持设置页 `导出 JSON` 能导出完整备份，作为用户主动备份入口。
2. 设置页本地存储区域补充真实数据目录显示或复制入口，方便用户排障。
3. 若检测到旧目录迁移成功，可在设置页或日志中展示一次性提示。
4. 不把升级保护依赖在用户手动导出上，手动导出只作为额外保险。

### Phase 6：验证闭环

1. 单元测试：
   - 数据目录解析优先级。
   - 旧目录迁移到新目录。
   - 新目录已有数据时不覆盖。
   - 快照目录包含 JSON、图片和备份。
2. 安装包后端测试：
   - `python/tests/test_build_app_main.py` 增加安装版数据目录与迁移测试。
3. 安装器壳测试：
   - `cargo check --manifest-path installer-shell/src-tauri/Cargo.toml`
   - 检查升级前快照命令返回值和失败提示。
4. 真实升级验收：
   - 安装旧版本到临时目录。
   - 创建测试提示词、分类和参考图片。
   - 运行新版本安装器壳覆盖安装。
   - 启动新版本，调用 `/api/export` 验证测试提示词仍存在。
   - 再执行卸载和重装，验证用户数据仍能恢复或继续读取。

## 实施结果

- `python/main.py` 与 `build/app_main.py` 已统一数据目录策略：开发模式保留源码目录数据，安装版优先使用 `%APPDATA%\PromptImageManager\data`，并支持 `PROMPT_IMAGE_TOOL_DATA_DIR` 测试覆盖。
- 新版本启动时会检测旧安装目录 `data/`，当用户级目录为空时复制迁移 JSON、图片、备份和 `.bak` 文件，并写入 `data-migration.json`。
- 用户级目录已有数据时不会被旧目录覆盖，只记录 `skipped-target-has-data`。
- `installer-shell/src-tauri/src/main.rs` 已在 NSIS 安装前创建升级快照，来源包括旧安装目录 `data/` 与用户级 `data/`。
- `installer-shell/src/main.js` 已将升级快照失败展示为完成页警告，不阻断主安装成功。
- `build/installer.nsi` 已在卸载前把旧 `$INSTDIR\data` 移到 `%APPDATA%\PromptImageManager\legacy-install-data`，再删除程序目录。
- PC 设置页本地存储区已通过 `/api/health` 展示真实数据目录。
