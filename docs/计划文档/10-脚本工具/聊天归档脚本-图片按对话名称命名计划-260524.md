# 聊天归档脚本图片按对话名称命名计划

## 背景

当前用户脚本 `gpt本地档案馆工具/ChatGPT Conversation Vault - CheeseTa Integrated-2026.05.24.10.user.js` 已支持在 `导出当前` 时把当前页面已渲染图片写入 ZIP 的 `current/assets/` 目录。

现有图片文件名由内部编号生成，例如：

```text
current/assets/image-001.png
current/assets/image-002.webp
```

这种命名稳定但可读性不足。用户希望图片文件能以对话名称进行重命名，方便解压后直接通过文件名识别图片归属。

## 可行性结论

可以实现。当前导出流程已经在 `exportCurrentZip()` 中拿到 `stored` 对话对象，`stored.title` 就是当前对话名称；图片写入路径集中在 `collectCurrentConversationImageAssets(folder)` 中生成，因此只需要在生成图片文件名时引入对话标题，并同步更新 `localPath`、`relativePath`、Markdown、HTML 和 `image-manifest.json` 即可。

## 命名规则

建议第一版采用：

```text
{安全化对话名称}-图片-{三位序号}.{扩展名}
```

示例：

```text
current/assets/平面化图片生成要求-图片-001.png
current/assets/平面化图片生成要求-图片-002.webp
```

命名细则：

1. 对话名称来自 `stored.title`。
2. 复用现有 `safeFileName()` 清理 Windows 非法字符。
3. 对话名为空时回退到 `Untitled Conversation` 或 `Untitled`。
4. 多张图片使用 `001`、`002`、`003` 递增序号避免重名。
5. 保留现有扩展名推断逻辑：优先 MIME，其次 URL 扩展名，最后默认 `png`。
6. 文件名需要控制长度，避免标题过长导致 ZIP 内路径过长。

## 实现方案

### 1. 新增图片文件名构造函数

新增纯函数：

```js
function buildImageAssetFileName(stored, imageNo, extension) {
  const title = safeFileName(stored.title || 'Untitled Conversation');
  return `${title}-图片-${imageNo}.${extension}`;
}
```

如需控制长度，可拆出标题裁剪：

```js
function imageTitlePrefix(title) {
  return safeFileName(title || 'Untitled Conversation').slice(0, 72) || 'Untitled';
}
```

### 2. 调整图片资产收集函数签名

当前函数：

```js
collectCurrentConversationImageAssets('current')
```

建议调整为：

```js
collectCurrentConversationImageAssets(stored, 'current')
```

这样函数内部可以直接读取 `stored.title` 构造图片文件名。

### 3. 同步 ZIP 路径和相对引用

当前逻辑：

```js
const filename = `${asset.id}.${extension}`;
asset.localPath = `${folder}/assets/${filename}`;
asset.relativePath = `assets/${filename}`;
```

目标逻辑：

```js
const filename = buildImageAssetFileName(stored, imageNo, extension);
asset.localPath = `${folder}/assets/${filename}`;
asset.relativePath = `assets/${filename}`;
```

Markdown 和 HTML 已经通过 `image.relativePath` 引用图片，因此只要 `relativePath` 更新，导出文本会自动指向新文件名。

### 4. 保持内部 ID 稳定

`asset.id` 建议继续保留为 `image-001`、`image-002`，不要改成对话名。

原因：

- `id` 是内部稳定编号。
- 文件名是面向用户的可读名称。
- `image-manifest.json` 同时记录两者，便于排查。

## 验收标准

1. 含图片对话导出的 ZIP 中，图片文件名包含对话名称。
2. 多张图片以序号区分，不互相覆盖。
3. Markdown 中的图片链接能正确指向新文件名。
4. HTML 图片画廊能正确加载新文件名。
5. `image-manifest.json` 中 `localPath`、`relativePath` 与真实 ZIP 路径一致。
6. 无图片对话导出不受影响。
7. 官方历史备份、本地库导出和历史单条下载行为不变。

## 风险点

- 对话标题可能很长，需要裁剪，避免 ZIP 内路径过长。
- 对话标题可能含 `/ \ ? % * : | " < >` 等非法字符，必须走 `safeFileName()`。
- 对话标题可能包含敏感信息，文件名会暴露该信息；这是按用户可读性换取隐私暴露的取舍。
- Markdown 和 HTML 引用依赖 `relativePath`，只改 ZIP 文件名但不改引用会导致离线图片失效。
