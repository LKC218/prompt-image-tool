# 桌面端与移动端安装包构建计划-260510

## 思考与分析

### Skill 嗅探结果

- 根目录未发现独立 `SKILL.md` 或 `AGENTS.md` 文件。
- 本次任务是构建计划类任务，使用 `planning-with-files` 技能，将方案、风险和任务清单沉淀为持久 Markdown 文档。

### 当前工程现状

- PC 端已有 `Vite -> PyInstaller -> NSIS -> releases` 打包主线。
- PC 快速构建入口为 `scripts/build_pc_package.py`，已使用 `subprocess.run([...])` 封装外部命令。
- PC 安装包配置位于 `build/installer.nsi`，当前已创建开始菜单快捷方式、卸载快捷方式和桌面快捷方式，并指定 `build/icon.ico`。
- Android 端已有 `Capacitor -> Gradle assembleRelease` 构建主线，配置入口为 `capacitor.config.ts`、`android/app/build.gradle`。
- 现有 `build.bat`、`package.json`、`capacitor.config.ts`、`docs/apps-code-map.md` 在默认读取方式下容易出现中文乱码，后续必须统一编码策略和构建输出方式。

### 目标

本轮计划目标是形成稳定、可复用、可验收的 PC 端与移动端安装包构建闭环：

- PC 端生成 Windows 安装包。
- PC 安装完成后必须具备正确图标的桌面快捷方式和开始菜单快捷方式。
- PC 安装器、快捷方式、卸载项、构建脚本输出不能出现中文乱码。
- 移动端优先构建 Android Release APK，并复制到 `releases/`。
- 构建流程必须可一键执行、可校验、可追踪产物。

## 实施计划

### 阶段一：构建链路确认

1. 确认当前版本号来源，以 `package.json` 为主。
2. 确认 PC 构建链路：
   - `npm run build`
   - `python -m PyInstaller build\app.spec --workpath build\build --distpath build\dist --clean -y`
   - `makensis /INPUTCHARSET UTF8 installer.nsi`
   - 复制安装包到 `releases/`
3. 确认 Android 构建链路：
   - `npm run build`
   - `npx cap sync android`
   - `scripts\patch-java-version.ps1`
   - `android\gradlew assembleRelease`
   - 复制 APK 到 `releases/`

### 阶段二：PC 安装包专项加固

1. 检查 `build/icon.ico` 是否存在且可被 NSIS 安装器和快捷方式引用。
2. 检查 `build/installer.nsi`：
   - `Unicode true`
   - `!insertmacro MUI_LANGUAGE "SimpChinese"`
   - `Icon "icon.ico"`
   - `UninstallIcon "icon.ico"`
   - `CreateShortCut "$DESKTOP\生图提示词管理器.lnk" ... "$INSTDIR\icon.ico" 0`
   - `CreateShortCut "$SMPROGRAMS\生图提示词管理器\生图提示词管理器.lnk" ... "$INSTDIR\icon.ico" 0`
   - 注册表 `DisplayIcon` 指向 `$INSTDIR\icon.ico`
3. 将安装后验收项固化到脚本或文档：
   - 桌面快捷方式存在。
   - 开始菜单快捷方式存在。
   - 快捷方式图标不是系统默认白板图标。
   - 控制面板卸载项显示名称无乱码。
   - 安装器页面中文无乱码。

### 阶段三：编码与乱码治理

1. 所有构建脚本统一保存为 UTF-8。
2. Python 构建脚本显式设置标准输出编码：
   - `sys.stdout.reconfigure(encoding="utf-8")`
   - `sys.stderr.reconfigure(encoding="utf-8")`
3. NSIS 构建必须使用：
   - `makensis /INPUTCHARSET UTF8 installer.nsi`
4. 批处理入口保留 `chcp 65001 >nul`，但优先使用 Python 非交互脚本作为正式构建入口。
5. 文档中明确 PowerShell 读取中文文件时使用 `-Encoding UTF8`。
6. 对 `package.json`、`capacitor.config.ts`、`build.bat`、`build/installer.nsi` 执行中文字段抽检，确认不是文件内容损坏，而是读取方式导致的显示问题。

### 阶段四：移动端安装包构建

1. 确认 Android 版本信息：
   - `android/app/build.gradle` 的 `versionCode`
   - `android/app/build.gradle` 的 `versionName`
   - `package.json` 的 `version`
2. 确认签名配置：
   - `android/keystore.properties` 存在。
   - `storeFile` 指向的密钥库存在。
   - 不将密钥、密码、真实签名文件写入 Git。
3. 执行 Android Release 构建。
4. 识别输出：
   - 优先 `android/app/build/outputs/apk/release/app-release.apk`
   - 若只有 `app-release-unsigned.apk`，标记为签名未完成，不作为正式发布包。
5. 复制正式 APK 到：
   - `releases/PromptImageManager-v{version}-Android.apk`

### 阶段五：一键构建脚本整理

1. 保留 `scripts/build_pc_package.py` 作为 PC 正式构建入口。
2. 新增或完善移动端非交互构建脚本，例如 `scripts/build_android_package.py`。
3. 可选新增总入口脚本，例如 `scripts/build_release_packages.py`：
   - `--pc`
   - `--android`
   - `--all`
   - `--skip-env-check`
4. 所有命令调用使用 `subprocess.run([...])`，避免字符串拼接执行。
5. 脚本必须在每一步后校验产物，不只依赖退出码。

### 阶段六：文档同步与交付闭环

1. 若修改构建脚本或安装器配置，同步更新：
   - `docs/apps-code-map.md`
   - `docs/构建方案/PC独立安装包快速打包流程.md`
   - `docs/构建方案/Android安装包构建方案.md`
2. 若新增统一构建入口，同步更新：
   - `README.md`
   - `docs/构建方案/README.md`
   - `scripts/README.md`，如果该文件存在或需要建立脚本目录说明。
3. 若沉淀出可复用乱码治理经验，同步补充：
   - `docs/项目开发经验/项目开发经验.md`
4. 检查 `.gitignore`：
   - 保持 `releases/`、构建中间产物、签名密钥、运行时数据不入库。

## 风险与处理

- 中文乱码风险：优先采用 UTF-8 文件、Python 非交互脚本、NSIS `/INPUTCHARSET UTF8` 三层处理。
- 快捷方式图标缺失风险：安装时必须复制 `icon.ico` 到 `$INSTDIR`，快捷方式与卸载项均指向该文件。
- Android 签名风险：若缺少 `keystore.properties` 或密钥文件，只允许输出未签名包，不标记为正式发布包。
- Java 版本风险：`cap sync android` 后必须执行 Java 兼容修补脚本，再执行 Gradle 构建。
- 版本漂移风险：正式构建前统一读取并核对 `package.json`、`android/app/build.gradle`。
