## 思考与分析

### 技能与上下文嗅探

- 根目录未发现项目级 `SKILL.md`。
- 本次使用 `planning-with-files` 技能：先沉淀计划文档，再进入实施。
- 本次使用 `ui-ux-pro-max` 技能：设置页按钮属于产品工具型 UI，需要保证按钮语义、响应式布局、焦点态和可访问性。
- 既有代码入口：
  - `src/js/pc-settings.js` 当前在“数据备份与恢复”区域渲染 4 个操作卡，其中两组都映射到相同的 `export` / `import` 行为，存在语义重复。
  - `src/js/backup-utils.js` 当前 PC 桌面 WebView 会优先走后端 `exportFile()`，普通 Web 环境才使用 `showSaveFilePicker` 或 `a.download`。
  - `src/js/api-storage.js` 当前 `exportFile(filename)` 只向 `/api/export-file` 传文件名，不能指定目标目录或完整路径。
  - `python/main.py` 与 `build/app_main.py` 当前 `save_backup_file()` 默认写入应用数据目录下的 `backups`，不符合“固定默认导出位置到下载目录”的新要求。
  - `python/tests/test_main.py` 与 `src/js/backup-utils.test.js` 已覆盖导出文件、导入导出闭环和前端保存兜底，是本次回归验证的基础。

### 问题判断

当前设置页 4 个按钮从功能上只有两个真实动作：导出与导入。继续保留“本地备份 / 导入数据 / 导出 JSON / 从 JSON 导入”会让用户误以为有 4 种不同能力。应收敛为两个强语义按钮：

- `导出 JSON`：默认保存到系统下载目录，提供“自定义位置”选择能力。
- `导入 JSON`：选择 `.json` 文件并导入恢复。

导出路径策略不能只在前端做文案调整。PC 桌面端当前优先走后端保存，因此默认目录必须在 Python 后端实现为下载目录；自定义位置需要从前端传给后端，或在支持文件保存选择器的环境下直接写入用户选择的文件。

## 实施计划

### Phase 1：确认现状与风险边界

1. 复核 `pc-settings.js` 中 4 个操作卡的 DOM、事件绑定和图标引用。
2. 复核 `backup-utils.js` 的导出优先级：桌面后端、文件选择器、浏览器下载三条路径。
3. 复核 Python 源码版与安装版入口：`python/main.py` 和 `build/app_main.py` 必须同步，避免源码版正常、PC 安装版不一致。
4. 复核现有测试：前端 `backup-utils.test.js`、后端 `python/tests/test_main.py`、导入导出闭环用例。

### Phase 2：后端导出目录策略

1. 新增“下载目录”解析函数：
   - Windows 优先使用用户配置的 Downloads 目录，兜底为 `%USERPROFILE%\Downloads`。
   - 非 Windows 兜底为 `~/Downloads`。
   - 若下载目录不存在，创建目录；创建失败时返回明确错误，不静默写入旧目录。
2. 调整 `save_backup_file()` 默认保存目录：
   - 未传自定义位置时，固定写入下载目录。
   - 返回值保留 `filename`、`path`、`directory`、`size`、统计信息。
3. 扩展 `/api/export-file` 请求参数：
   - `filename`：继续保留，仍要做文件名清洗。
   - `targetPath` 或 `directory`：用于自定义保存位置。
   - 自定义路径必须校验扩展名为 `.json`，并禁止目录穿越。
4. 同步 `python/main.py` 与 `build/app_main.py`，避免 PC 打包入口漏改。

### Phase 3：前端导出保存体验

1. `backup-utils.js` 增加 PC 导出模式：
   - 默认导出：不弹路径选择，直接请求后端保存到下载目录。
   - 自定义位置：调用 `showSaveFilePicker`，可用时直接写入用户选择文件；不可用时回退到后端自定义路径能力或普通下载。
2. `exportBackup(storage, options)` 扩展选项：
   - `preferBackend` 保持桌面默认优先。
   - `saveMode: 'downloads' | 'custom'`。
   - `targetPath` / `directory` 用于后端自定义保存。
3. 成功提示明确展示位置：
   - 默认：`已导出到下载目录：文件名.json`。
   - 自定义：`已导出到自定义位置：完整路径或文件名`。
4. 保留 30 秒延迟清理 Blob URL 的既有机制，避免 0KB 下载问题回归。

### Phase 4：设置页按钮与交互重设

1. 将“数据备份与恢复”操作区从 4 个卡片缩减为 2 个按钮：
   - 左侧：`导入 JSON`，使用导入图标，负责选择 `.json` 并导入。
   - 右侧：`导出 JSON`，使用导出图标，负责导出完整备份。
2. 导出按钮交互建议：
   - 主按钮点击：默认导出到下载目录。
   - 在按钮内或按钮旁增加“小型自定义位置入口”，用于选择自定义保存位置。
   - 若必须严格只有两个可点击按钮，则采用点击 `导出 JSON` 后弹出轻量菜单或对话框，提供“保存到下载目录 / 选择位置”两个选项；页面常驻按钮仍只有两个。
3. 响应式规则：
   - 桌面宽屏两列并排。
   - 中窄宽度保持两列但不压字。
   - 小屏退化为单列。
4. 视觉规则：
   - 使用现有 `src/assets/icons/settings/export-json.svg` 和 `import-json.svg`。
   - 去掉重复的 `backup.svg` 与 `import-data.svg` 卡片引用，但不删除资源，避免影响后续或移动端复用。
   - 保持彩色图标容器、清晰焦点态、150ms 到 300ms hover 过渡。

### Phase 5：导入流程保持稳定

1. 保持 `input[type=file]` 的 `.json` 选择逻辑。
2. 导入前解析 JSON，非法文件显示明确错误。
3. 导入成功后刷新设置页统计。
4. 本轮不扩展导入预览和覆盖确认，避免扩大范围；若要补导入预览，可作为后续任务接入既有“设置功能完善计划”。

### Phase 6：测试验证

1. 前端单元测试：
   - 默认 PC 桌面导出调用后端，并确认请求参数表示下载目录模式。
   - 自定义位置模式优先调用 `showSaveFilePicker`。
   - 后端失败时仍可回退到普通下载。
   - Blob URL 延迟清理不回归。
2. 后端单元测试：
