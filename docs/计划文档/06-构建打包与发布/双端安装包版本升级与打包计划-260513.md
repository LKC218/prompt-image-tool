# 双端安装包版本升级与打包计划-260513

## 思考与分析

### Skill 嗅探结果

- 本次任务是“制定计划”，且后续会涉及版本号、双端构建、产物验证和文档收口，使用 `planning-with-files` 技能，将计划沉淀为持久 Markdown 文档。
- 仓库内仅发现 `.github/skills/frontend-design/SKILL.md`，与本次打包任务不匹配，不启用。
- 结合历史打包经验，本仓库 PC 安装包应优先走现有脚本链路，不能只看构建成功；移动端应以 Android Release APK、版本号核对和必要时真机安装作为交付闭环。

### 当前版本状态

- 当前主版本号为 `2.3.1`，目标版本号为 `2.3.2`。
- Android 当前为 `versionName "2.3.1"`、`versionCode 6`，本轮应升级为 `versionName "2.3.2"`、`versionCode 7`。
- PC 核心安装包当前命名为 `PromptImageManager-Setup-2.3.1.exe`，本轮目标为 `PromptImageManager-Setup-2.3.2.exe`。
- Android 发布包当前命名为 `PromptImageManager-v2.3.1-Android.apk`，本轮目标为 `PromptImageManager-v2.3.2-Android.apk`。
- 如果继续沿用 Tauri 安装器壳作为 PC 端正式分发外壳，还应产出 `PromptImageManager-Shell-Setup-2.3.2.exe`。

### 关键风险

- 版本号分散在前端包、Android Gradle、NSIS、Tauri 配置和安装器壳硬编码路径中，若只改一个入口会导致产物名、嵌入安装核心或应用显示版本不一致。
- `package-lock.json` 顶层版本仍存在旧值，需要在版本升级时一并校正，避免后续发布核对时出现漂移。
- `installer-shell/src-tauri/src/main.rs` 和 `installer-shell/src-tauri/tauri.conf.json` 含有 `PromptImageManager-Setup-2.3.1.exe` 硬编码路径，若要打安装器壳，必须同步到 `2.3.2`。
- 当前工作区已有较多未提交改动，实施时必须只追加本轮必要变更，不回退用户已有改动。

## 实施计划

### 阶段一：版本升级预检

1. 执行 `git status --short`，记录已有改动范围，避免覆盖非本轮修改。
2. 用 `rg "2\.3\.1|versionCode 6|PromptImageManager-Setup-2\.3\.1"` 定位所有发布相关版本落点。
3. 区分“必须改”的发布配置和“保留历史”的版本记录：
   - 必须改：`package.json`、`package-lock.json` 顶层版本、`build/installer.nsi`、`build.bat`、`android/app/build.gradle`、`src-tauri/Cargo.toml`、`src-tauri/tauri.conf.json`、`installer-shell/package.json`、`installer-shell/src-tauri/Cargo.toml`、`installer-shell/src-tauri/tauri.conf.json`、`installer-shell/src-tauri/src/main.rs`。
   - 保留历史：`docs/版本记录/changelog.md` 中已有 `v2.3.1` 历史段落，仅在顶部新增 `v2.3.2` 记录，不覆写旧记录。

### 阶段二：统一升级到 2.3.2

1. 将应用版本统一改为 `2.3.2`。
2. 将 Android `versionCode` 从 `6` 递增到 `7`。
3. 将安装包硬编码文件名同步为：
   - `PromptImageManager-Setup-2.3.2.exe`
   - `PromptImageManager-Shell-Setup-2.3.2.exe`
   - `PromptImageManager-v2.3.2-Android.apk`
4. 版本修改完成后再次执行 `rg`，确认发布配置中不再残留 `2.3.1`，仅允许历史 changelog、旧计划文档和依赖版本中保留。

### 阶段三：构建前验证

1. 执行前端和脚本静态检查：
   - `npm.cmd run build`
   - `node --check` 检查安装器壳页面脚本
   - `cargo check --manifest-path installer-shell/src-tauri/Cargo.toml`
2. 检查 Android 签名配置：
   - `android/keystore.properties` 存在
   - 密钥文件存在
   - 不把密钥、密码或真实签名文件纳入 Git
3. 检查 PC 安装包配置：
   - `build/installer.nsi` 的 `APPVERSION` 为 `2.3.2`
   - `build/icon.ico` 存在
   - 前端构建产物 `dist/index.html` 可生成

### 阶段四：双端打包

1. PC 核心安装包：
   - 执行 `python scripts/build_pc_package.py --skip-env-check`
   - 验证 `build/dist/PromptImageManager/PromptImageManager.exe`
   - 验证 `build/PromptImageManager-Setup-2.3.2.exe`
   - 验证 `releases/PromptImageManager-Setup-2.3.2.exe`
2. PC 安装器壳：
   - 执行 `python scripts/build_installer_shell_package.py --skip-env-check`
   - 验证 `build/installer-shell/PromptImageManager-Shell-Setup-2.3.2.exe`
   - 验证 `releases/PromptImageManager-Shell-Setup-2.3.2.exe`
3. Android 安装包：
   - 执行 `python scripts/build_android_package.py --skip-env-check`
   - 验证 `releases/PromptImageManager-v2.3.2-Android.apk`
   - 若只生成 unsigned 包，则标记为不可正式发布，优先修复签名配置，不把 unsigned 包当作正式交付。

### 阶段五：运行与真机验证

1. PC 安装包运行验证：
   - 启动已打包 EXE 或安装后 EXE
   - 检查 `app.log`
   - 根据真实监听端口探测 `/`、`/index.html`、`/api/health`、`/api/sync/capabilities`
   - 确认端口占用时按 `8888-8897` 顺序回退，不使用随机端口
2. Android 真机验证：
   - 若手机已连接 USB 调试，执行 `adb install -r releases/PromptImageManager-v2.3.2-Android.apk`
   - 用 `dumpsys package com.promptimagemanager.app` 核对 `versionName=2.3.2` 和 `versionCode=7`
   - 从手机侧用 HTTP 探针访问 PC 局域网同步接口，确认移动端能命中真实端口

### 阶段六：文档与交付收口

1. 更新 `docs/版本记录/changelog.md`，在顶部新增 `v2.3.2` 段落，记录版本升级、双端打包产物和验证结论。
2. 更新 `docs/版本记录/README.md`，补充最新版本入口。
3. 如构建脚本、安装器壳路径或平台配置发生实际变更，同步更新：
   - `docs/apps-code-map.md`
   - `docs/构建方案/PC独立安装包快速打包流程.md`
   - `docs/构建方案/Android安装包构建方案.md`
   - 必要时补充 `docs/项目开发经验/项目开发经验.md`
4. 输出最终产物清单、文件大小、校验哈希和验证命令结果。
