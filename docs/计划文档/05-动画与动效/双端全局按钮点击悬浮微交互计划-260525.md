# 双端全局按钮点击悬浮微交互计划

> 版本：v1.0.0  
> 创建日期：2026-05-25  
> 范围：PC 端与移动端全局按钮、图标按钮、卡片入口、列表行、底部操作栏的 hover、active、focus-visible 与低动效兜底

## 一、思考与分析

### 1.1 技能与上下文嗅探

- 项目根目录未发现项目级 `SKILL.md`，本轮遵循用户提供的 `AGENTS.md` 指令、仓库文档同步规范和既有计划文档格式。
- 本次使用 `planning-with-files`，用于把计划沉淀到 `docs/计划文档/` 并同步 `docs/apps-code-map.md`。
- 本次使用 `ui-ux-pro-max`，用于校准 hover、tap、点击反馈、可访问焦点、减弱动态和响应式稳定性约束。
- 已确认 PC 端全局样式集中在 `src/css/pc.css`，通用按钮以 `.pc-btn` 为主，但页面内还存在 `.pc-section-title-action`、`.pc-library-*btn`、`.pc-detail-*btn`、`.pc-settings-action-card`、`.pc-editor-*btn` 等专用入口。
- 已确认移动端全局样式集中在 `src/css/mobile.css`，已存在 `--m-transition-fast`、`--m-transition-normal`、多处 `:active` 缩放和 `-webkit-tap-highlight-color: transparent`，适合作为全局触控反馈基线。
- 现有 PC 左侧导航图标悬浮和点击动效已经单独规划并落地，本计划不重复重写导航专属动效，只把它纳入全局交互语言的一致性校验。

### 1.2 当前现状

- PC 端部分按钮只有颜色 hover，缺少按下态、焦点态或图标层反馈，用户点击后有时无法立刻获得确认感。
- PC 端 `.pc-btn` 使用 `transition: all 0.15s`，后续应收敛到明确属性，避免宽高、排版或不可预期属性被动画化。
- PC 端存在大量页面级按钮类名，不能靠只改 `.pc-btn` 覆盖全部交互入口，需要建立一套轻量选择器矩阵。
- 移动端多数入口已有 `:active`，但强弱不统一，有的只改背景，有的只缩放，有的图标按钮缺少可见按下反馈。
- 移动端触屏没有真正 hover，重要反馈必须放在 tap / active / focus-visible 上；hover 只能在支持鼠标的调试环境或平板外设下作为增强。
- 双端都必须尊重 `prefers-reduced-motion: reduce`，低动效模式只保留颜色、边框、透明度变化，不播放位移、缩放、投影和关键帧动画。

### 1.3 交互目标

- 让所有可点击入口都有明确的“可点、已按下、已聚焦、已禁用”状态。
- PC 端 hover 以轻微上浮、柔和阴影、背景提亮、图标染色为主，幅度控制在 1px 到 2px，不改变布局尺寸。
- PC 端 click / active 以短促下压和释放反馈为主，避免全局弹跳过度干扰工作台操作。
- 移动端 tap 以即时缩放、背景压暗或透明度变化为主，不依赖 hover，不加入持续动画。
- 图标按钮优先让图标与按钮容器一起反馈，纯图标按钮必须保留稳定触控尺寸和 `aria-label`。
- 全局微交互要服务效率，不做粒子、闪光、循环跳动或跨页面转场。

## 二、实施计划

### Phase 1：交互对象盘点与分级

- 扫描 `src/css/pc.css` 和 `src/css/mobile.css` 中已有按钮、卡片入口、列表行和图标按钮选择器。
- 将交互对象分成四类：
  - 主按钮：保存、导入、导出、创建、同步、确认。
  - 次按钮：返回、取消、筛选、分页、刷新、清空。
  - 图标按钮：收藏、更多、复制、关闭、下载、删除。
  - 卡片和列表入口：首页快捷卡、提示词卡片、设置动作卡、分类行、表格行。
- 不给禁用态、纯展示卡片、只读统计块增加点击动画。
- 对已经有专属动效的侧边栏导航只做冲突检查，不覆盖 `.pc-nav-clicking` 的动画优先级。

### Phase 2：PC 端全局按钮微交互基线

- 在 `src/css/pc.css` 建立 PC 端按钮交互基线，优先覆盖：
  - `.pc-btn`
  - `.pc-section-title-action`
  - `.pc-library-icon-btn`
  - `.pc-detail-top-nav-btn`
  - `.pc-detail-action-btn`
  - `.pc-editor-save-btn`
  - `.pc-settings-action-card`
  - `.pc-image-viewer-tool`
- hover 状态采用：
  - 背景或边框轻微提亮。
  - `transform: translateY(-1px)` 或只加阴影。
  - 图标透明度或颜色增强。
- active 状态采用：
  - `transform: translateY(0) scale(0.98)`。
  - 主按钮可短暂降低阴影，表示按下。
- focus-visible 状态采用：
  - 使用 `outline` 或 `box-shadow` 明确键盘焦点。
  - 不依赖颜色作为唯一提示。
- 禁用态采用：
  - `cursor: not-allowed`、降低透明度、取消 transform 和 shadow。
- 避免使用 `transition: all`，改为 `background-color`、`border-color`、`box-shadow`、`color`、`opacity`、`transform` 等明确属性。

### Phase 3：移动端触控反馈基线

- 在 `src/css/mobile.css` 建立移动端触控反馈基线，优先覆盖：
  - `.m-action-btn`
  - `.m-save-btn`
  - `.m-confirm-btn`
  - `.m-top-nav-back`
  - `.m-top-nav-action`
  - `.m-nav-item`
  - `.m-card`
  - `.m-prompt-card`
  - `.m-backup-card`
  - `.m-lan-sync-btn`
  - `.m-image-viewer-close`
  - `.m-image-viewer-download`
- tap / active 状态采用：
  - 主按钮 `scale(0.96)` 到 `scale(0.98)`。
  - 图标按钮背景轻微加深或透明度变化。
  - 卡片入口轻微下压，不改变卡片尺寸和文本流。
- hover 只放在 `@media (hover: hover) and (pointer: fine)` 中作为调试器、桌面浏览器和外接鼠标增强。
- 移动端底部导航和底部操作栏保留固定高度，微交互不能改变安全区、底部栏高度和可点击面积。
- 移动端所有按钮保持至少 40px 触控尺寸，关键操作优先 44px 到 48px。

### Phase 4：点击确认反馈策略

- 第一版以 CSS `:active` 和 `focus-visible` 为主，不新增全局点击监听。
- 只有需要“点击后释放仍能看到一次反馈”的高价值入口，才考虑新增轻量 JS 辅助类，例如：
  - 保存成功按钮。
  - 导入导出入口。
  - 同步入口。
  - 图片下载入口。
- 若新增 JS 辅助类，必须满足：
  - 使用 `animationend` 清理类名，不用累计 `sleep` 计时。
  - 尊重 `prefers-reduced-motion: reduce`。
  - 不阻断原点击逻辑，不吞掉事件冒泡。
  - 补充对应 Vitest 测试。

### Phase 5：低动效与无障碍兜底

- PC 与移动端统一补齐 `prefers-reduced-motion: reduce`：
  - 关闭 transform、animation、filter、box-shadow 动画。
  - 保留颜色、边框、透明度等低干扰状态。
- 补齐键盘 `focus-visible`：
  - PC 端按钮、分页、筛选、弹窗按钮、图片查看器工具按钮。
  - 移动端可聚焦按钮、弹层确认按钮、设置页同步按钮。
- 图标按钮继续保留 `aria-label` 或可见文本，不让动效替代语义。
- 禁用按钮不播放 hover、active 或点击确认反馈。

  - 移动端：`docs/UI计划/移动端UI设计/00-全局设计规范.md`。
  - 技术说明：必要时补充 `docs/技术文档/pc-technical-doc.md` 和 `docs/技术文档/mobile-technical-doc.md` 的交互约定。

  - `npm.cmd run test`

## 六、后续建议

- 建议第一版只做“全局 hover / tap / focus-visible 基线”，不增加粒子、扫光、长循环或复杂 JS 点击监听。
- PC 端 hover 可以略有上浮和阴影；移动端只保留 tap 即时反馈，hover 仅作为外接鼠标增强。
- 如果工程师希望更活泼，第二阶段再挑少量高价值入口做一次性点击确认动效，例如保存、导入、导出、同步、下载。
