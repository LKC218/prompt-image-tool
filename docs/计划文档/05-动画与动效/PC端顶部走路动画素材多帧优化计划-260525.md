# PC端顶部走路动画素材多帧优化计划

> 版本：v0.1.0  
> 创建日期：2026-05-25  
> 范围：PC 端顶部欢迎横幅与提示词详情页顶部红框区域的走路动画素材替换、帧数扩展和自然度优化

## 一、思考与分析

### 1.1 技能与上下文嗅探

- 本次使用 `planning-with-files`，用于把优化方案沉淀到 `docs/计划文档/` 并同步 `docs/apps-code-map.md` 导航入口。
- 本次使用 `ui-ux-pro-max`，用于校准持续动画的克制原则、减弱动态兜底、帧动画自然度和页面可读性。
- 已确认当前走路动画由 `src/js/pc-welcome-banner.js` 的 `renderPcWelcomeWalkAnimation()` 统一输出结构。
- 已确认当前运行时素材为 `src/assets/pc/home-pixel-dog-sprite.png`，透明背景 8 帧横向精灵图，当前文件约 30KB。
- 已确认动画样式集中在 `src/css/pc.css`，通过 `.pc-welcome-pixel-stage-*` 页面变体控制轨道位置，通过 `pc-pixel-dog-frames` 和 `pc-pixel-dog-cross` 控制序列帧与横向移动。
- 已确认提示词详情页通过 `src/js/pc-detail.js` 的 `.pc-detail-walk-decoration` 接入同一走路动画，不改成共享欢迎横幅结构。

### 1.2 新素材初步判断

- 工程师提供的新图路径为：`C:\Users\Administrator\Downloads\ChatGPT Image 2026年5月25日 02_24_43.png`。
- 图片尺寸为 `2172px × 724px`，文件约 `1.34MB`，格式为 PNG。
- 当前图片像素格式为 `Format24bppRgb`，没有 Alpha 透明通道。
- 图片背景中的棋盘格是被烘焙进图片的实际像素，不是透明背景；不能直接替换到浅蓝顶部栏，否则会出现灰白格背景。
- 视觉上当前新图约有 8 个明显姿态帧，已经比旧素材角色更清晰，但帧间节奏仍需要通过裁切、对齐、补间或扩帧来优化。

### 1.3 问题根因

- 当前动画“自然度”主要受三类因素影响：
  - 素材层：每帧角色脚底基准线、身体重心和左右边距不一致，会造成播放时抖动。
  - 帧数层：8 帧可以完成基础走路，但如果步态关键帧间距过大，会显得机械。
  - 运动层：当前横向移动周期和帧播放周期固定，如果步频与位移速度不匹配，会出现脚底打滑。
- 因此本轮不应只“换图”，而应把新图处理为透明、等尺寸、基准线稳定、帧数可配置的运行时精灵图。

### 1.4 设计目标

- 将当前顶部走路动画素材替换为工程师提供的新像素狗形象。
- 动画保留像素风和轻量陪伴感，但走路节奏更自然，减少原地抖动、脚底打滑和错帧感。
- 支持 8 帧、12 帧或 16 帧横向精灵图，不把帧数写死在 CSS 和测试里。
- 不改变现有页面接入范围：首页、提示词库、分类与标签、设置、新建/编辑和提示词详情页继续复用同一动画入口。
- 不引入复杂动画库，不增加运行时 JavaScript 定时器；继续使用本地 PNG 精灵图和 CSS `steps()`。

## 二、实施计划

### Phase 1：素材清理与帧拆分

- 读取新图并确认可见帧数量、角色边界、角色脚底基准线和每帧有效内容。
- 先生成工作区临时帧，不直接覆盖运行时资源。
- 对每一帧执行：
  - 去除烘焙棋盘格背景，输出透明 Alpha。
  - 裁切角色有效区域，保留统一安全边距。
  - 对齐脚底基准线，避免播放时上下跳动。
  - 对齐身体中心点，避免播放时左右漂移。
- 如果 8 帧播放仍不够自然，优先采用两种扩帧策略：
  - 12 帧方案：在关键步态之间补 4 个过渡帧，控制体积与自然度平衡。
  - 16 帧方案：把完整步态拆得更细，适合后续如果顶部动画仍显机械时使用。

### Phase 2：运行时精灵图规格

- 输出仍落到现有运行时路径：
  - `src/assets/pc/home-pixel-dog-sprite.png`
- 保持文件名不变，减少 JS 引用改动和构建产物路径变化。
- 建议新增或更新素材元信息常量，避免 CSS 写死 `8`：
  - 帧宽：建议按实际裁切后控制在 `72px` 到 `96px`。
  - 帧高：建议控制在 `96px` 到 `120px`，避免压迫顶部栏文字。
  - 帧数：支持 `8`、`12`、`16`。
  - 播放周期：8 帧约 `0.72s`，12 帧约 `0.84s`，16 帧约 `0.96s`。
- 输出资源目标：
  - 透明 PNG。
  - 体积尽量控制在 `80KB` 以内。
  - 不保留棋盘格、白边、压缩噪点和多余空白。

### Phase 3：CSS 参数化与自然度调校

- 在 `.pc-welcome-pixel-stage` 上增加或整理 CSS 变量：
  - `--pc-pixel-dog-frame-w`
  - `--pc-pixel-dog-frame-h`
  - `--pc-pixel-dog-frame-count`
  - `--pc-pixel-dog-frame-duration`
  - `--pc-pixel-dog-cross-duration`
- 将 `background-size` 从固定 `* 8` 改为 `* var(--pc-pixel-dog-frame-count)`。
- 将 `steps(8)` 改为 `steps(var(--pc-pixel-dog-frame-count))`，或按兼容性采用帧数类名兜底。
- 调整横向移动周期，让脚步频率与移动速度匹配：
  - 首页与详情页保持最明显，但速度不超过当前视觉强度。
  - 设置、新建/编辑等工作型页面略降透明度或放慢移动，减少干扰。
- 如需要增加自然感，只做轻微垂直起伏，不做大幅弹跳：
  - 起伏幅度建议不超过 `2px`。
  - 起伏周期与步态周期绑定，避免身体漂浮感。

### Phase 4：渲染入口兼容

- `renderPcWelcomeWalkAnimation(options)` 保持现有调用方式不变。
- 如实现需要传入新帧数，优先使用默认配置，不要求每个页面重复传参。
- 保持 `aria-hidden="true"` 和 `pointer-events: none`，动画仍为纯装饰。
- 保持详情页 `.pc-detail-walk-decoration` 仅负责定位，不复制动画 DOM 和关键帧。

### Phase 5：测试与视觉验收

- 更新 `src/js/pc-home-pixel-animation.test.js`：
  - 检查动画不再硬编码只支持 8 帧。
  - 检查 CSS 存在帧数变量或帧数兜底类。
  - 检查 `prefers-reduced-motion` 仍关闭序列帧和横向移动。
  - 检查新精灵图文件存在且体积可控。
- 验证命令：
  - `npm.cmd run test -- pc-home-pixel-animation.test.js`

  - PC 首页、提示词库、分类与标签、设置、新建/编辑、提示词详情页。
- 重点观察：
  - 没有棋盘格背景。
  - 脚底没有明显打滑。
  - 身体没有上下抖动。
  - 不遮挡标题、副标题、按钮、右侧插画、详情页顶部操作区。

## 五、实现建议

- 第一轮建议先做高质量 8 帧透明替换，并把帧数参数化；如果视觉仍显机械，再扩到 12 帧或 16 帧。
- 不建议把原始 `2172px × 724px` 大图直接放入运行时目录，体积和背景都会影响前端体验。
- 不建议每个页面单独维护一张精灵图；多页面统一换肤可以减少后续维护成本。
- 如果新图背景去除后边缘有浅色残留，优先在素材阶段清理，不用 CSS 滤镜硬压。
