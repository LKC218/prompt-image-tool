# 跨端配色与主题令牌重构实施计划

> 版本：1.0 | 创建日期：2026-07-16 | 状态：待评审
>
> 范围：PC Web、移动端 Web、安装器壳的视觉配色与主题系统；不调整提示词、分类、标签等业务功能。

---

## 一、决策摘要

### 1.1 品牌方向

产品采用“可靠清爽的柯基灵感管家”作为统一视觉方向。

- 柯基为正式品牌主角，承担欢迎、空状态、完成反馈与主题叙事。
- 工具界面以雾白、蓝灰与深蓝墨夜为稳定底盘，优先保证阅读、整理和长时间使用的舒适度。
- 用户工作台主题为核心个性化能力；主题只影响品牌互动与氛围，不改变业务状态和用户内容颜色。
- 标签、文件夹和分类颜色属于用户内容索引，跨端、跨外观模式保持相同色相和稳定标识。

### 1.2 主题模型

主题拆分为三个互不混淆的维度。

| 维度 | 可选值 | 负责内容 | 禁止影响的内容 |
|---|---|---|---|
| 外观模式 | 浅色、深色、跟随系统、定时切换 | 表面、文字、边框、阴影、内容色与状态色的明暗映射 | 品牌主题语义、数据中的分类键 |
| 工作台主题 | 晴空巡逻、焦糖午后、松林远足、星夜侦察、海盐薄荷 | 主操作、选中态、焦点环、欢迎区域、柯基配件与装饰 | 成功、警告、危险、信息状态；内容分类色 |
| 内容索引 | 晴蓝、珊瑚、松绿、葡紫、琥珀、青碧 | 分类、文件夹、标签的标记、图标、浅底 | 全局品牌色、按钮主操作色、状态色 |

### 1.3 迁移目标

- 以三层 Token 取代 PC、移动端与组件内的分散十六进制色值。
- 统一 `data-appearance`、`data-workbench-theme` 和内容索引键，替代当前端侧不同的 `data-accent` 与持久化键。
- 支持连续的浅色/深色体验；深色模式采用深蓝墨夜底盘，不使用纯黑和简单反色。
- 将安装器壳的天空蓝、金色插画语言收敛到主应用的品牌主题和柯基叙事之内。

## 二、现状与问题边界

### 2.1 已确认问题

- PC 与移动端的品牌、分类和主题色各自定义，同一业务语义存在不同色值。
- PC 使用 `pc-accent`，移动端使用 `accent`；移动端启动时没有稳定恢复已保存主题。
- 组件样式中仍存在大量硬编码色，主题切换无法完整覆盖。
- 品牌粉色兼任主操作、收藏、负向提示词和部分错误背景，语义冲突。
- 状态色、内容分类色、页面身份色与用户主题色缺乏明确边界。
- 现有暗色相关文档和实际浅色马卡龙实现不一致，不能作为后续权威规范。
- 安装器与主应用的色温、饱和度和插画材质不一致，缺少品牌连续性。

### 2.2 不在本计划内

- 不重构路由、存储、提示词编辑、同步或导入导出业务流程。
- 不改变用户分类、标签和文件夹的业务结构，仅调整颜色数据契约和显示映射。
- 不在本计划内新增深色模式以外的无障碍功能；但所有色彩改造必须满足本计划规定的可访问性验收。
- 不在未完成 Token 替换前大规模重绘所有插画资源。

### 3.5 固定状态色

| 状态 | 浅色前景 / 容器 | 深色前景 / 容器 | 使用范围 |
|---|---|---|---|
| 成功 | `#1F8A5B` / `#E8F6EF` | `#6ED7A3` / `#194A37` | 保存、导入、连接正常 |
| 信息 | `#2477D4` / `#EAF3FF` | `#7DBDFF` / `#173C63` | 同步、帮助、普通提示 |
| 警告 | `#B87508` / `#FFF5DB` | `#FFCA6B` / `#604514` | 待确认、容量、未保存离开 |
| 危险 | `#C73E4B` / `#FDECEF` | `#FF9EAA` / `#642B35` | 删除、失败、异常、不可逆操作 |

状态色不可被工作台主题覆盖。所有状态反馈必须同时使用文字、图标或动作描述，不能仅依赖颜色。

## 四、技术设计与数据契约

### 4.1 Token 分层

| 层级 | 命名示例 | 职责 | 使用限制 |
|---|---|---|---|
| 原始色板 | `--palette-sky-600` | 存放基础色阶 | 组件禁止直接使用 |
| 语义 Token | `--color-brand-primary`、`--color-state-danger-fg` | 定义颜色的业务角色 | 外观和主题在此层覆写 |
| 组件别名 | `--button-primary-bg`、`--tag-category-fg` | 为组件绑定语义 Token | 只能引用语义 Token |

### 4.2 根节点属性

```text
data-appearance="light | dark"
data-workbench-theme="sky | caramel | forest | night | mint"
```

- `data-appearance` 负责全量表面、文字、边框、阴影、状态和内容索引色的明暗映射。
- `data-workbench-theme` 仅负责 `brand-*` 和柯基环境装饰 Token。
- 在页面首次绘制前读取并应用两个已保存的偏好，避免先显示默认主题再闪烁切换。
- 支持 `appearance-preference`：`light`、`dark`、`system`、`scheduled`；`system` 使用系统媒体查询，`scheduled` 以设备本地绝对时间计算下一次切换点。

### 4.3 设置与持久化契约

| 数据项 | 新键名 | 值 | 迁移来源 |
|---|---|---|---|
| 外观偏好 | `appearance-preference` | `light`、`dark`、`system`、`scheduled` | 旧主题/系统偏好逻辑 |
| 工作台主题 | `workbench-theme` | 五个主题键 | `pc-accent`、`accent` |
| 分类颜色 | `colorKey` | 六个内容索引键 | 历史 `color` 十六进制值 |

- 在保留旧键兼容读取的一个发布周期内，首次读取成功后写入新键。
- 新键写入成功后不再以旧键作为真相来源。
- 无法识别的历史颜色使用确定性回退映射，并保留原值用于诊断；不得随机分配颜色。
- PC、移动端共享同一映射表和迁移函数，避免再次产生端侧色值分裂。

### 4.4 深色模式原则

- 不使用 `#000000` 纯黑作为主画布，不通过 CSS `filter: invert()` 实现暗色。
- 深色主题提高交互色明度、降低大面积饱和度；卡片、浮层与背景通过深蓝层级区分。
- 分类与状态颜色保持色相和名称，按外观模式映射前景、容器和边框值。
- 柯基本体保持自然焦糖毛色；仅替换配件、环境和装饰元素，夜间使用月光蓝与低对比星点。

## 六、页面和组件迁移顺序

按“全局容器优先、核心操作优先、内容实体随后、装饰最后”的顺序实施。

1. 应用 Shell、侧栏、底部导航、页面背景与顶部栏。
2. 卡片、表格、列表、弹窗、输入框、按钮和焦点状态。
3. 首页、提示词库、详情、编辑器、分类与标签、设置。
4. Toast、确认对话框、同步状态、导入导出、错误和空状态。
5. 欢迎横幅、柯基插画、完成反馈、安装器壳与其他品牌装饰。

每完成一个页面，必须先替换其语义颜色，再清理硬编码色；禁止先进行纯视觉调色而保留旧主题逻辑。

## 七、可访问性与质量门槛

- 正文文字与背景、按钮文字与按钮背景必须达到 WCAG AA 常规文本对比度要求。
- 大号文字、图标按钮和非文本 UI 边界必须达到适用的最低对比度要求。
- 焦点环在浅色和深色模式中都必须清晰可见，且不能只依赖背景变化。
- 禁用态不得与普通弱文本、边框或占位符难以区分。
- 状态反馈必须提供图标、文字或动作描述；分类颜色不得承担错误、成功或警告含义。
- 每一主题在浅色和深色中均检查主按钮、次按钮、链接、选中导航、输入焦点、标签、表格行和弹窗。
- 主题切换不得造成布局抖动、图片闪烁或首屏颜色闪白。

## 九、风险与控制措施

| 风险 | 后果 | 控制措施 |
|---|---|---|
| 主题与状态色混用 | 用户无法识别失败、警告和危险操作 | 状态 Token 与品牌 Token 在代码层分组，评审禁止交叉引用 |
| 端侧分类色迁移不一致 | 同一数据在 PC 与移动端变色 | 使用单一共享映射模块，并对历史值编写固定迁移测试 |
| 硬编码清理遗漏 | 局部页面无法随主题切换 | 阶段 0 生成清单，阶段 3 以搜索结果归零为验收条件 |
| 深色模式对比不足 | 阅读疲劳、操作不可见 | 将对比度检查列入组件和截图验收 |
| 主题数量膨胀 | 维护成本成倍增加 | 外观与工作台主题分轴；新增主题必须复用同一六 Token 模型 |
| 插画过度抢占注意力 | 降低高密度内容阅读效率 | 柯基仅用于欢迎、空状态、完成反馈及角落弱装饰 |
| 首屏主题闪烁 | 损害主题卖点感知 | 在应用挂载和首屏渲染前同步读取并设置根节点属性 |

## 十一、完成定义

本计划仅在满足以下全部条件后视为完成：

- PC、移动端与安装器均使用统一的语义 Token 模型和工作台主题定义。
- 用户可以独立选择外观模式与工作台主题，重启后稳定恢复且首屏无明显闪烁。
- 深色模式是深蓝墨夜的完整映射，而不是纯黑、反色或亮色主题的简单压暗。
- 主题切换只影响品牌交互和叙事氛围，业务状态与用户内容索引色不随主题改变含义。
- 历史主题和分类颜色数据完成可逆或可追溯的确定性迁移，双端显示一致。
- 核心页面和关键状态完成测试矩阵中的组合验证，构建与静态检查通过。
- 全局规范、双端就近规范、代码地图与目录索引已经同步，且不存在冲突的色彩真相来源。
