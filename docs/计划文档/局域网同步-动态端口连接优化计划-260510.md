### 问题背景

PC 安装包入口 `build/app_main.py` 已具备端口占用回退能力：优先监听 `8888`，若端口被占用则按顺序尝试 `8889-8897`，并通过 `SERVER_PORT` 返回真实端口。

但移动端同步链路仍以 `8888` 为默认假设：

- `LanScanner.scan()` 当前只扫描单个端口。
- 最近设备渲染时会把端口重新覆盖为 `8888`。
- 手动输入只支持 IP，不支持 `IP:端口`。
- `testLanConnection()` 固定请求 `http://IP:8888/api/health`。
- `LanSync.sync()`、`getCapabilities()`、`getPairing()`、`push()`、`bidirectional()` 均固定拼接 `:8888`。
- 最近设备按 IP 去重，没有把 `port` 纳入目标身份，端口变化后容易连接旧地址。
- `python/main.py` 开发后端仍固定端口 `8888`，`/api/health` 和 `/api/network-info` 返回值也写死 `8888`，与安装包入口能力不完全对齐。

根因不是“端口不是 8888”本身，而是端口没有成为同步目标的一等参数。修复目标应是全链路使用 `{ ip, port }` 或标准化后的同步目标对象，而不是到处临时拼字符串。

## 实施计划

### 阶段一：统一局域网目标模型

1. 在 `src/js/lan-sync.js` 中新增轻量工具函数：
   - `parseLanTarget(value, defaultPort = 8888)`
   - `buildLanBaseUrl(target)`
   - `formatLanAddress(target)`
2. 支持以下输入：
   - `192.168.6.109`
   - `192.168.6.109:8890`
   - `http://192.168.6.109:8890`
3. 输出统一结构：
   - `ip`
   - `port`
   - `baseUrl`
   - `address`
4. 保持不带端口时默认 `8888`，降低已有使用习惯的迁移成本。

### 阶段二：移动端扫描支持端口范围

1. 扩展 `LanScanner._normalizeScanOptions()`：
   - 新增 `ports` 参数。
   - 兼容旧的 `port` 参数。
   - 默认扫描端口建议为 `[8888, 8889, 8890, 8891, 8892, 8893, 8894, 8895, 8896, 8897]`。
2. 最近设备优先探测：
   - 先探测保存过的 `ip:port`。
   - 若最近设备没有端口，再按默认端口范围探测。
3. 网段扫描：
   - 对每个候选 IP 扫描默认端口范围。
   - 控制并发，避免 `254 * 10` 请求一次性打满。
4. 搜索结果以 `ip:port` 去重，结果中保留后端返回的真实 `port`。

### 阶段三：移动端手动输入和 UI 状态优化

1. 输入框提示改为支持完整地址：
   - 示例：`192.168.6.109` 或 `192.168.6.109:8890`
2. 点击搜索结果时填入完整地址：
   - 不再只填 IP。
3. 设备列表元素增加 `data-port` 或直接保存 `data-address`。
4. 连接失败提示从“检查 8888 端口”调整为“检查 PC 设置页显示的端口和防火墙入站规则”。
5. 最近设备保存：
   - 按 `ip:port` 去重。
   - 保存 `port`、`address`、`deviceId`、`token`、`capabilities`。

### 阶段四：同步请求全链路使用动态端口

1. 调整 `LanSync` 方法签名，兼容旧调用：
   - `sync(target)`
   - `getCapabilities(target)`
   - `getPairing(target)`
   - `push(target, options)`
   - `bidirectional(target, options)`
2. 所有请求从统一 `baseUrl` 构造：
   - `/api/health`
   - `/api/sync`
   - `/api/sync/images/{filename}`
   - `/api/sync/capabilities`
   - `/api/sync/pairing`
   - `/api/sync/import`
   - `/api/sync/bidirectional`
3. 双向同步内部调用必须传递同一个目标对象，避免 `push()` 使用动态端口而 `sync()` 又回到 `8888`。

### 阶段五：PC 后端端口返回统一

1. `build/app_main.py` 已有 `SERVER_PORT`，需验证：
   - `/api/health` 返回实际端口。
   - `/api/sync/capabilities` 返回实际端口。
   - `/api/network-info` 返回实际端口。
2. `python/main.py` 开发后端补齐同样的 `SERVER_PORT` 变量，避免开发环境和安装包行为不一致。
3. 评估开发后端是否也引入端口回退：
   - 推荐第一版同步安装包行为，优先 `8888`，占用时按顺序尝试 `8889-8897`。
   - 若暂不回退，也必须让返回值来自同一个端口常量，减少写死点。

### 阶段六：测试验证

1. 单元测试：
   - `parseLanTarget()` 解析 IP、`IP:端口`、URL、非法端口。
   - `LanScanner` 能按端口范围构造探测请求。
   - `LanSync` 构造的所有接口地址使用传入端口。
2. Python 回归：
   - 安装包后端端口占用时返回真实端口。
   - 开发后端 `/api/health`、`/api/network-info`、`/api/sync/capabilities` 的端口一致。
3. 真机验证：
   - 先占用 `8888`，启动 PC 安装包，使其回退到非 `8888` 端口。
   - Android 搜索能发现 `PC_IP:实际端口`。
   - Android 手动输入 `PC_IP:实际端口` 能连接成功。
   - 拉取、回传、双向同步均使用实际端口。
   - 图片下载使用实际端口。

## 验收标准

- PC 安装包在 `8888` 被占用时能启动到 `8889-8897` 范围内的其他端口，并在 PC 设置页展示真实同步地址。
- Android 搜索能发现非 `8888` 的 PC 服务。
- Android 手动输入 `IP:端口` 能连接成功。
- 拉取、回传、双向同步和图片下载都使用真实端口。
- 最近设备再次连接时不回退到错误的 `8888`。
- 文档中不再把 `8888` 描述为唯一端口，而是描述为默认优先端口。

## 风险与处理

| 风险 | 影响 | 处理 |
|---|---|---|
| 扫描端口范围扩大导致搜索变慢 | 用户等待时间增加 | 最近设备优先、限制端口范围、控制并发、进度提示 |
| 防火墙只放行 `8888` | 非 `8888` 端口仍无法访问 | PC 设置页和移动端提示用户放行实际端口 |
| 旧最近设备没有端口 | 老用户升级后首次连接失败 | 原始旧记录无端口时按默认端口范围探测，命中后保存为 `ip:port` |
| 同一 IP 多个端口都有服务 | 用户可能选错实例 | 展示设备名、端口、能力和最近同步时间 |
| 开发后端与安装包行为不一致 | 测试通过但发布后失败 | 两个入口同步返回真实端口，并补回归测试 |
