## 一、思考与分析

### 1.1 技能与上下文嗅探

- 使用 `planning-with-files` 技能：本次属于多阶段测试、修复、打包交付，需要先沉淀计划文件并在确认后推进。
- 使用 prompt-image-tool 本地预览记忆工作流：本仓库稳定预览入口是前端 `http://127.0.0.1:5173/?ui=pc` / `?ui=mobile`，后端健康检查是 `http://127.0.0.1:8888/api/health`，优先复用 `scripts/start_dev_server.py`。
- 使用 prompt-image-tool PC 安装包记忆工作流：PC 端带安装壳 UI 的最终交付应走核心 PC 安装包加 Tauri installer-shell UI 包，最终核对 `releases/` 产物。
- 根目录未发现独立 `SKILL.md` 或 `AGENTS.md` 文件，本轮以工程师提供的项目指令和仓库现有文档为准。

### 1.2 现有上下文

- 当前仓库版本为 `2.3.3`，如无额外要求，本计划默认不升级版本号，只基于当前版本和当前工作区改动进行测试、修复与打包。
- 已有相关计划：
  - `docs/计划文档/PC端保存导致内容丢失排查计划-260524.md`：PC 保存失败与内容丢失风险已实施，但验收清单仍需在总回归中复核。
  - `docs/计划文档/双端设置页图片下载记录与移动端相册保存计划-260524.md`：图片下载记录与移动端相册保存已实施，但 Android 真机相册可见性仍待验收。
  - `docs/计划文档/移动端下载记录UI布局排版优化计划-260524.md`：移动端下载记录布局属于本次移动端视觉回归重点。
- 当前工作区存在大量未提交改动和未跟踪文件；后续构建产物会包含这些差异。实施前需要先记录 `git status`，并在最终回复明确“产物基于当前工作区状态构建”。

### 1.3 测试策略

本次不只跑构建命令，而是按“服务可用性 -> 双端功能回归 -> 问题修复 -> 自动化验证 -> 安装包构建 -> 产物核验”的闭环推进。发现问题时直接进入最小修复，修复后回到同一用例复测，直到核心流程通过或明确受限原因。

优先级排序：

1. 数据安全和保存链路：保存失败、数据丢失、导入导出、同步覆盖风险优先处理。
2. 移动端真实能力：相册保存、下载记录、移动端页面导航、Android 构建优先处理。
3. PC 安装体验：带安装壳 UI、嵌入核心安装器、快捷方式、安装后启动优先处理。
4. 视觉与交互问题：只修影响可用性、明显错位、遮挡和触控失败的问题，避免临时扩大范围。

## 二、实施计划

### Phase 0：基线确认与数据保护

目标：确认本次测试基于正确服务和正确数据源，避免测试过程破坏真实数据。

- 记录 `git status --short`，标记当前工作区已有改动。
- 检查 `package.json`、`scripts/README.md`、`scripts/start_dev_server.py`、打包脚本和构建方案文档是否仍与计划一致。
- 用 `scripts/start_dev_server.py --check-only` 检查本地前后端。
- 如服务不可用，使用 `scripts/start_dev_server.py --no-open --timeout 35` 启动。
- 验证：
  - `http://127.0.0.1:8888/api/health`
  - `http://127.0.0.1:5173/?ui=pc`
  - `http://127.0.0.1:5173/?ui=mobile`
- 复现前备份运行时数据：
  - `python/data/prompt_sets.json`
  - `python/data/folders.json`
  - 必要时同步备份图片目录和下载记录相关本地数据。

### Phase 1：PC 端 Bug 测试

目标：覆盖 PC 端当前最容易影响交付的主流程。

- 首页：
  - 页面加载、统计卡、最近使用、收藏分类、快速创建入口。
- 提示词库：
  - 搜索、分类筛选、分页、选中预览、复制、收藏、更多操作。
- 详情页：
  - 正向提示词、负向提示词、版本记录、图片预览、下载当前图片。
- 新建/编辑：
  - 新建保存、编辑保存、图片粘贴、图片预览、分类标签选择、保存失败时草稿保留。
  - 重点复核“保存失败”和“已创建”不应同时出现。
- 分类与标签：
  - 新建、重命名、删除、拖拽排序、标签改色。
- 设置页：
  - 图片下载记录、清空历史、导入 JSON、导出 JSON、局域网互通状态展示。
- 后端接口：
  - `/api/health`
  - `/api/prompt-sets`
  - 导入导出接口
  - 图片保存和读取接口
  - 局域网同步能力接口。

### Phase 2：移动端 Bug 测试

目标：覆盖移动端页面可用性、触控、下载和 Android 能力。

- 移动端首页：
  - 页面加载、底部导航、搜索入口、统计与最近内容。
- 移动端提示词库：
  - 搜索、筛选、列表滚动、卡片点击、更多菜单。
- 移动端详情页：
  - 返回、复制、收藏、图片查看器、下载当前图片、底部弹层。
- 移动端编辑页：
  - 新建保存、编辑保存、标签选择、分类选择、图片预览、输入框长文本。
- 移动端分类与标签：
  - 新建、重命名、删除、排序、批量操作。
- 移动端设置页：
  - 图片下载记录、清空历史、局域网 PC 搜索、手动 `IP:端口` 连接、拉取、回传、双向同步。
- Android 专项：
  - `ImageGallerySaverPlugin` 是否注册。
  - 图片下载后是否进入系统相册。
  - App 关闭后相册内图片仍可见。
  - 返回键和权限相关行为不影响主流程。

### Phase 3：发现问题后的修复规则

目标：自行修复问题，但保持最小改动和可回归。

- 先写清楚复现路径和影响范围，再修改代码。
- 优先修改对应模块，避免跨层绕路：
  - PC 页面问题优先看 `src/js/pc-*.js` 和 `src/css/pc.css`。
  - 移动端页面问题优先看 `src/js/mobile-*.js` 和 `src/css/mobile.css`。
  - 共享下载、导入导出、同步问题优先看 `src/js/image-download-utils.js`、`src/js/backup-utils.js`、`src/js/lan-sync.js`。
  - 后端问题同步检查 `python/main.py` 和 `build/app_main.py`。
  - Android 原生问题检查 `android/app/src/main/java/com/promptimagemanager/app/`。
- 修复后补对应测试；没有合适测试入口时，至少补运行时探针和测试记录。
- 如果改动功能模块，按项目规范同步 `docs/apps-code-map.md` 和就近模块说明文档。
- 如果问题属于可复用工程经验，补充 `docs/项目开发经验/项目开发经验.md`。

### Phase 4：自动化验证

目标：在打包前确认源码状态可构建、关键测试通过。

- 前端单元和页面级回归：
  - `npm.cmd run test`
- Python 后端测试：
  - `python -m pytest python/tests -q`
- PC 安装版后端专项：
  - `python -m pytest python/tests/test_build_app_main.py -q`
- 前端构建：
  - `npm.cmd run build`
- Android 编译专项：
  - `android/gradlew.bat compileDebugJavaWithJavac`
- 如 installer-shell 有改动：
  - `node --check installer-shell/src/main.js`
  - `node --check installer-shell/src/pages/*.js`
  - `cargo check --manifest-path installer-shell/src-tauri/Cargo.toml`

### Phase 5：安装包构建

目标：测试通过后生成移动端 APK 和 PC 端带安装壳 UI 的安装包。

- 先构建 PC 核心安装包和 Android APK：
  - `python scripts/build_release_packages.py --all --skip-env-check`
- 再构建带安装壳 UI 的 PC 安装包：
  - `python scripts/build_installer_shell_package.py --skip-pc-build`
- 产物核对：
  - `releases/PromptImageManager-v2.3.3-Android.apk`
  - `releases/PromptImageManager-Shell-Setup-2.3.3.exe`
  - `build/PromptImageManager-Setup-2.3.3.exe`
- 如版本号已被工程师要求升级，则先统一修改版本面，再执行构建；否则不主动改版本号。

### Phase 6：安装包验收

目标：确认产物不是“只生成文件”，而是可安装、可启动、可使用。

- Android：
  - 校验 APK 存在、大小非零。
  - 尽量执行签名和包信息检查。
  - 如已连接设备，安装 APK 并打开应用验证核心页面与相册保存。
  - 如未连接设备，明确标记真机安装和相册验收未完成。
- PC 带安装壳 UI：
  - 校验 shell 安装包存在、大小非零。
  - 启动安装壳，确认欢迎页和步骤页可见。
  - 使用临时目录执行安装，验证安装完成。
  - 验证安装后主程序能打开前端 UI。
  - 验证桌面快捷方式、开始菜单启动快捷方式、开始菜单卸载快捷方式真实存在。
  - 卸载后验证临时安装目录和快捷方式清理情况。

### Phase 7：文档与收口

目标：让交付结果可追溯。

- 更新本计划文档的任务清单和验证结果。
- 如实际修复了功能模块，更新：
  - `docs/apps-code-map.md`
  - 最近的 PC / 移动端 / API / 构建方案说明文档
  - 必要时更新 `docs/测试记录/`
  - 必要时更新 `docs/版本记录/`
- 最终回复列出：
  - 修复的问题
  - 验证命令结果
  - 安装包路径
  - 未完成或受限的真机验收项
  - 项目级约束校验单
