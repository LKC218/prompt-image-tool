# 版本发布与更新记录维护指南

## 目标

本指南用于每次发布前汇总最新修改、判断迭代版本号、同步跨端版本配置，并将面向用户的内容更新到 PC 端“更新记录”弹窗。

后续只需提供本文件，即可按本流程完成版本发布维护；未明确要求时，不创建 Git 提交、不推送远端、不生成安装包。

## 输入与输出

### 发布前输入

- 当前工作区修改、未跟踪文件和最近提交记录。
- 已完成的功能、修复和优化说明，以及对应测试结果。
- 当前发布基线版本和目标平台：Web、PC Tauri、PC NSIS、安装器壳、Android。

### 必须输出

- 语义化的新版本号和 Android `versionCode`。
- [完整版本记录](版本记录/changelog.md)。
- [版本记录索引](版本记录/README.md)。
- PC 端更新记录弹窗的结构化内容。
- 所有发布配置的同步修改与验证结果。

## 标准流程

### 1. 收集本轮变更

1. 检查 `git status --short --branch`、`git log --oneline -n 20`、`git diff --stat` 和未跟踪文件。
2. 按“新增、优化、修复、发布与打包”归纳用户可感知的变化；不要将格式化、行尾变化或依赖自身版本误写为应用功能。
3. 查阅相关测试、构建和模块文档，只将实际完成且已验证的内容写入“验证”小节。
4. 工作区若混有无关改动，应在记录中只描述本次发布范围，不覆盖、不重置用户已有修改。

### 2. 判定版本号

遵守 `主版本.次版本.修订版本` 三段式语义化版本：

| 情况 | 版本策略 | 示例 |
|------|----------|------|
| 仅修复缺陷、文案或兼容性问题 | 递增修订版本 | `2.4.0` → `2.4.1` |
| 新增向后兼容的用户功能、明显交互或能力增强 | 递增次版本并将修订版本归零 | `2.4.0` → `2.5.0` |
| 存在不兼容的数据、接口或使用方式调整 | 递增主版本并归零 | `2.4.0` → `3.0.0` |

- 版本字符串在配置中一律使用 `2.5.0`，不带 `v`。
- 界面、更新记录标题、徽标和产物文件名使用 `v2.5.0`。
- Android `versionCode` 必须是严格递增整数，每发布一次递增 `1`，不得根据语义化版本推导。
- 若当前版本配置、锁文件或安装器壳版本不一致，先以根目录 `package.json` 的发布版本为基线统一修正，再继续升级。

### 3. 同步版本配置

将 `<版本号>` 和 `<安卓版本码>` 替换为本次真实值；不要批量替换历史版本记录或第三方依赖版本。

| 文件 | 必须同步的字段 |
|------|----------------|
| `package.json` | `version` |
| `package-lock.json` | 根 `version`、`packages[""].version` |
| `src/index.html` | `<meta name="version" content="<版本号>">` |
| `src/js/version-info.js` | `DEFAULT_VERSION` |
| `src-tauri/tauri.conf.json` | `version` |
| `src-tauri/Cargo.toml`、`src-tauri/Cargo.lock` | 应用包 `version` |
| `android/app/build.gradle` | `versionCode <安卓版本码>`、`versionName "<版本号>"` |
| `build/installer.nsi` | `APPVERSION` |
| `build.bat` | 构建标题、核心安装包文件名与复制路径 |
| `installer-shell/package.json` | `version` |
| `installer-shell/src-tauri/tauri.conf.json` | `version`、内嵌核心安装包资源路径 |
| `installer-shell/src-tauri/Cargo.toml`、`installer-shell/src-tauri/Cargo.lock` | 安装器壳包 `version` |
| `README.md` | 版本徽标 |

## 更新记录维护

### 完整发布档案

在 [版本记录](版本记录/changelog.md) 顶部插入本次版本，格式固定如下：

```md
## v<版本号> (YYYY-MM-DD)

### 新增

- **能力名称**：用户可感知的结果与影响。

### 优化

- **优化名称**：说明体验或行为变化。

### 修复

- **问题名称**：说明修复的场景与结果。

### 版本与打包

- 主应用、PC Tauri、Android、NSIS 安装器和安装器壳统一升级至 `<版本号>`。
- Android `versionCode` 从 `<旧值>` 递增至 `<安卓版本码>`。

### 验证

- `实际执行的命令`：通过，填写真实测试数量或构建结果。
```

- 无对应内容的“新增、优化、修复”小节可省略。
- 未运行构建、安装、签名或真机验证时，不得写“通过”；应明确写“未执行”或不创建该验证项。
- 更新 [版本记录索引](版本记录/README.md) 的“最新版本”，只描述当前真实状态，不提前声明 GitHub Release、安装包或校验和。

### PC 更新记录弹窗

运行时弹窗由 `src/js/release-notes-data.js` 的 `RELEASE_NOTES` 提供数据，必须在数组首项加入本次版本，保持新到旧排序：

```javascript
{
    version: '<版本号>',
    date: 'YYYY-MM-DD',
    sections: [
        {
            title: '新增',
            tone: 'pink',
            items: ['面向用户的简短说明。']
        },
        {
            title: '优化',
            tone: 'blue',
            items: ['面向用户的简短说明。']
        },
        {
            title: '修复',
            tone: 'yellow',
            items: ['面向用户的简短说明。']
        }
    ]
}
```

- 每个版本必须包含 `version`、ISO 日期 `date` 和 `sections`。
- 仅使用已定义色调：`pink`、`blue`、`yellow`。
- 弹窗文案应简短、以用户结果为中心，不写内部文件名、测试数量、构建命令或未交付内容。
- 当前版本必须同时存在于 `meta[name="version"]` 和 `RELEASE_NOTES`，否则不会出现未读提示或自动弹窗。
- PC 端以 `pc-release-notes-last-seen-version` 保存已读版本。点击“我知道了”才可写入已读；关闭、稍后查看、遮罩点击和 ESC 只关闭弹窗。
- 更新 `src/js/release-notes.test.js` 中的版本基线与已读断言，确保覆盖当前版本未读和确认阅读后的状态。

## 文档同步

- 必须更新 [版本号模块说明](模块说明/版本号模块.md) 中的默认版本、版本来源示例和发布维护范围。
- 必须更新与构建版本有关的 [PC端构建流程](构建方案/PC端构建流程.md) 版本表。
- [应用代码地图](apps-code-map.md) 仅维护本指南、版本号模块和更新记录模块的导航；不记录本次发布历史或验证结果。
- 当新增稳定源码模块时，才新增模块说明；常规版本发布不新建无职责的文档。

## 验证清单

按实际变更范围执行，最低要求如下：

```powershell
npm.cmd run test
python -m pytest python/tests -q
npm.cmd run build
```

还需检查：

- `package.json`、`package-lock.json`、`index.html`、Tauri、Android、NSIS 和安装器壳的版本号一致。
- Android `versionCode` 比上一发布版本大 `1`。
- `RELEASE_NOTES` 首项版本与 `meta[name="version"]` 一致，当前版本可显示在 PC 更新记录弹窗中。
- [完整版本记录](版本记录/changelog.md) 和 [版本记录索引](版本记录/README.md) 的最新版本一致。
- 使用 `git diff --check` 检查本次变更；若发现既有无关空白或行尾问题，单独说明，不随意清理。

仅在用户明确要求发布包时，额外执行对应构建、签名、安装验证和校验和记录；仅在用户明确要求时再创建 Git 提交或推送远端。

## 交付说明模板

```md
已将项目升级至 v<版本号>。

- 版本配置：已同步主应用、PC、Android、安装器与安装器壳；Android `versionCode` 为 `<安卓版本码>`。
- 完整记录：已更新 `docs/版本记录/changelog.md` 与索引。
- 弹窗记录：已在 `src/js/release-notes-data.js` 添加当前版本，PC 端首次启动会按未读规则展示。
- 验证：列出实际已通过和未执行的命令。
- 发布状态：未创建提交、未推送、未生成安装包，除非已明确执行。
```
