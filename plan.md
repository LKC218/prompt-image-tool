# 生图提示词管理器 v2.0 — 优化计划文档

## 项目概况

- **技术栈**：Vite + 原生 JavaScript（无框架）+ 原生 CSS
- **PC 端**：Tauri 2.x (Rust) 壳 + Python HTTP 后端 (端口 8888) + JSON 文件存储
- **Android 端**：Capacitor 8.x + SQLite 本地存储 + Filesystem 管理图片
- **存储层**：策略模式 — PC 用 `ApiStorage`（HTTP API），Android 用 `SqliteStorage`（Capacitor SQLite）
- **架构**：单页应用 (SPA)，无路由库，通过 DOM 显示/隐藏切换视图

### 阶段一：Bug 修复（优先级：🔴 高）

#### 任务 1：移动端导入图片无法显示，直接显示为空

**原始描述**：移动端导入图片无法显示，直接显示为空

**优化描述**：

- **问题定位**：Android 端（Capacitor）导入图片后，图片卡片区域渲染为空白，无图片内容显示
- **根因分析方向**：
  1. `SqliteStorage.getImageUrl()` 方法（[sqlite-storage.js:262-274](file:///c:/Users/34267/Downloads/prompt-image-tool/src/js/sqlite-storage.js#L262-L274)）通过 `Capacitor.convertFileSrc()` 将本地文件 URI 转换为可访问的 Web URL，若 `Filesystem.getUri()` 返回的路径与实际写入路径不一致，会导致转换失败
  2. `processImageFiles()` 方法（[app.js:393-416](file:///c:/Users/34267/Downloads/prompt-image-tool/src/js/app.js#L393-L416)）中 `uploadImage()` 返回的 `file` 字段值可能与 `getImageUrl()` 中读取时使用的路径不匹配
  3. `getImageUrl()` 的 catch 分支返回 `img.data || ''`，但图片数据中可能没有 `data` 字段（仅存了 `file` 路径），导致返回空字符串
- **具体操作**：
  1. 在 `processImageFiles()` 中添加日志，确认 `uploadImage` 返回的 `file` 值格式
  2. 在 `getImageUrl()` 中添加日志，确认 `Filesystem.getUri()` 返回的 URI 和 `Capacitor.convertFileSrc()` 转换后的 URL
  3. 检查 `renderVersionDetail()` 中图片 `src` 的赋值逻辑，确认是否正确调用了 `getImageUrl()`
  4. 修复路径不匹配或 fallback 逻辑缺失的问题
- **验证方式**：在 Android 设备/模拟器上导入图片，确认图片正常显示
- **涉及文件**：`src/js/sqlite-storage.js`、`src/js/app.js`

#### 任务 2：新增版本时不应复制旧版本的提示词内容

**原始描述**：新增版本会复制旧版本的信息，不需要带有旧版本信息

**优化描述**：

- **问题定位**：点击「+ 新版本」按钮后，新版本自动携带上一版本的正向提示词 (`prompt`) 和反向提示词 (`negativePrompt`)，用户期望新版本为空白起始
- **根因分析**：`addNewVersion()` 方法（[app.js:194-208](file:///c:/Users/34267/Downloads/prompt-image-tool/src/js/app.js#L194-L208)）中明确复制了 `lastVersion.prompt` 和 `lastVersion.negativePrompt`：
  ```javascript
  const result = await getStorage().addVersion(currentPromptSetId, {
      prompt: lastVersion ? lastVersion.prompt : '',        // ← 复制了旧版本内容
      negativePrompt: lastVersion ? lastVersion.negativePrompt : '',  // ← 复制了旧版本内容
      note: ''
  });
  ```
- **具体操作**：
  1. 将 `addNewVersion()` 中的 `prompt` 和 `negativePrompt` 初始值改为空字符串 `''`
  2. 保留版本名自动递增逻辑（`v{n+1}`）
  3. 注意：现有的「复制为新版本」功能（`duplicateVersion`）应保持复制行为不变，这是两个不同的用户意图
- **验证方式**：新建版本后确认提示词区域为空；使用「复制为新版本」后确认提示词内容被复制
- **涉及文件**：`src/js/app.js`

### 阶段二：移动端体验优化（优先级：🔴 高）

#### 任务 3：移动端支持手机手势返回上一级

**原始描述**：移动端需要增加手机手势返回上一级，而不是直接退出软件

**优化描述**：

- **问题定位**：Android 端在详情页按系统返回键或使用手势返回时，应用直接退出，而非返回到集合列表页
- **根因分析**：当前应用没有使用任何路由机制（无 hash 路由、无 History API），所有视图状态仅在内存中维护。系统返回手势/按键无法感知应用内的导航层级
- **具体操作**：
  1. **引入 History API 导航**：在 `selectPromptSet()` 调用 `showDetailView()` 时，使用 `history.pushState()` 推入一个导航状态
  2. **监听 `popstate` 事件**：当用户触发系统返回时，`popstate` 事件触发，执行 `showMobileList()` 返回列表视图
  3. **处理多层导航**：图片查看器、模态框等也需推入状态，确保返回时先关闭 overlay 再返回列表
  4. **导航状态映射**：
     - 列表页 → 基础状态（无额外 history entry）
     - 详情页 → pushState({view: 'detail', id: xxx})
     - 图片查看器 → pushState({view: 'imageViewer'})
     - 模态框 → pushState({view: 'modal'})
  5. **PC 端兼容**：PC 端同样监听 `popstate`，实现浏览器后退按钮功能
- **验证方式**：在 Android 设备上从列表进入详情，使用系统返回手势/按键，确认返回列表而非退出应用；在详情页打开图片查看器后返回，确认关闭查看器而非返回列表
- **涉及文件**：`src/js/app.js`

### 阶段三：功能增强（优先级：🟡 中）

#### 任务 4：首页提示词集合长按弹出操作菜单（删除、重命名、移动）

**原始描述**：在首页，按住提示词合集弹窗删除，重命名选项，移动

**优化描述**：

- **问题定位**：当前首页侧边栏的提示词集合项（`.prompt-item`）仅支持点击选中，没有快捷操作入口。删除、重命名等操作需要先进入详情页再操作，流程繁琐
- **具体操作**：
  1. **实现长按/右键菜单**：
     - 移动端：监听 `touchstart`/`touchend` 实现长按检测（约 500ms），触发上下文菜单
     - PC 端：监听 `contextmenu` 事件（右键菜单），阻止默认行为后显示自定义菜单
  2. **菜单选项**：
     - 🗑️ 删除 — 调用现有 `deletePromptSet()` 逻辑
     - ✏️ 重命名 — 弹出重命名模态框，调用 `updatePromptSetName()`
     - 📁 移动到文件夹 — 弹出文件夹选择器（依赖任务 5 的文件夹功能，若未实现则先预留接口）
     - 📋 复制 — 复制当前集合为新集合
  3. **自定义上下文菜单组件**：
     - 创建浮动菜单 DOM，定位在长按/右键位置附近
     - 点击菜单项后执行对应操作并关闭菜单
     - 点击其他区域或滚动时自动关闭菜单
  4. **视觉反馈**：长按过程中集合项显示轻微缩放 + 背景高亮效果
- **验证方式**：移动端长按集合项弹出菜单；PC 端右键集合项弹出菜单；各菜单项功能正常
- **涉及文件**：`src/js/app.js`、`src/css/main.css`、`src/index.html`

#### 任务 5：首页增加文件夹模式

**原始描述**：首页增加一个文件夹模式，可以创建文件夹整理归类，增加颜色标签，文件夹可以重命名

**优化描述**：

- **功能概述**：在首页侧边栏增加文件夹层级，用户可创建文件夹并将提示词集合归类到文件夹中，支持颜色标签和重命名
- **数据模型扩展**：
  ```
  新增 Folder 数据结构：
  {
    id: string,          // 唯一标识
    name: string,        // 文件夹名称
    color: string,       // 颜色标签（如 '#FF6B6B', '#4ECDC4' 等）
    sortOrder: number,   // 排序序号
    createdAt: string,   // 创建时间
    updatedAt: string    // 更新时间
  }

  PromptSet 扩展字段：
  {
    ...现有字段,
    folderId: string | null  // 所属文件夹 ID，null 表示未分类
  }
  ```
- **存储层修改**：
  - `ApiStorage`：Python 后端新增 `/api/folders` CRUD 接口，`prompt_sets` 表增加 `folder_id` 字段
  - `SqliteStorage`：新增 `folders` 表，`prompt_sets` 表增加 `folder_id` 字段
- **UI 交互设计**：
  1. **侧边栏布局**：
     - 顶部增加「全部」「未分类」筛选标签
     - 文件夹以可折叠列表形式展示，每个文件夹前显示颜色圆点
     - 文件夹内展示所属的集合项，文件夹外展示「未分类」集合
     - 底部增加「+ 新建文件夹」按钮
  2. **文件夹操作**：
     - 点击文件夹 → 展开/折叠，展示内部集合
     - 长按/右键文件夹 → 弹出菜单：重命名、更改颜色、删除（集合移至未分类）
     - 拖拽集合项到文件夹 → 移动集合到该文件夹
  3. **颜色标签**：
     - 预设 8-10 种颜色供选择
     - 颜色以小圆点形式显示在文件夹名称前
  4. **视图模式切换**：
     - 侧边栏顶部增加「列表视图/文件夹视图」切换按钮
     - 列表视图保持现有平铺展示（兼容旧习惯）
     - 文件夹视图启用文件夹分组展示
- **具体操作**：
  1. 扩展数据模型和存储层（Python 后端 + SQLite）
  2. 实现文件夹 CRUD 操作
  3. 修改 `renderPromptList()` 支持文件夹分组渲染
  4. 实现文件夹交互（创建、重命名、颜色选择、删除）
  5. 实现集合拖拽移动到文件夹
  6. 实现视图模式切换
- **验证方式**：创建文件夹、设置颜色、移动集合到文件夹、重命名文件夹、删除文件夹、切换视图模式
- **涉及文件**：`src/js/app.js`、`src/js/api-storage.js`、`src/js/sqlite-storage.js`、`src/js/storage.js`、`src/css/main.css`、`src/css/responsive.css`、`src/index.html`、`python/main.py`

#### 任务 7：PC 端与移动端 UI/UX 保持统一

**原始描述**：UIUX方面需要pc端和移动端保持统一

**优化描述**：

- **问题定位**：当前 PC 端和移动端存在以下不一致：
  1. PC 端侧边栏固定 300px 宽度，移动端全屏切换
  2. 移动端有返回按钮，PC 端无对应导航方式
  3. 移动端图片卡片操作按钮常驻显示，PC 端 hover 才显示
  4. 模态框尺寸和间距在不同端差异较大
- **统一原则**：
  - **响应式而非双套设计**：使用同一套 CSS 变量和组件，通过断点调整尺寸而非重写样式
  - **交互一致性**：相同功能在不同端使用相同的视觉表达（图标、颜色、布局结构）
  - **平台适配**：仅在输入方式（鼠标 vs 触摸）和屏幕尺寸上做必要适配
- **具体操作**：
  1. **建立统一设计 Token**：
     - 在 `main.css` 中定义 CSS 变量体系：颜色、间距、圆角、阴影、字体大小
     - 移动端和 PC 端共享同一套 Token，仅通过媒体查询调整数值
  2. **统一组件样式**：
     - 按钮：统一圆角、内边距、hover/active 状态
     - 输入框：统一高度、边框、聚焦样式
     - 卡片：统一圆角、阴影、间距
     - 模态框：统一动画、最大宽度、内边距
  3. **统一导航模式**：
     - PC 端增加面包屑导航或侧边栏高亮，与移动端返回按钮功能对应
     - 统一 Escape 键和移动端返回手势的行为（已在任务 3 中处理）
  4. **统一图片卡片交互**：
     - PC 端和移动端统一使用 hover/长按 显示操作按钮
     - 操作按钮的图标和布局保持一致
  5. **清理 `responsive.css`**：
     - 将媒体查询中重复定义的样式改为 CSS 变量覆盖
     - 减少移动端样式覆盖量，让默认样式本身就是响应式友好的
- **验证方式**：同功能在 PC 端和移动端对比检查，确认视觉表达和交互逻辑一致
- **涉及文件**：`src/css/main.css`、`src/css/responsive.css`、`src/js/app.js`

## 执行依赖关系

```
任务1 (Bug: 移动端图片显示) ──→ 无依赖，可立即开始
任务2 (Bug: 新版本复制内容) ──→ 无依赖，可立即开始
任务3 (手势返回) ──→ 无依赖，可与任务1/2并行
任务4 (长按菜单) ──→ 建议在任务3之后（共享交互模式）
任务5 (文件夹模式) ──→ 建议在任务4之后（任务4的"移动"选项依赖文件夹功能）
任务6 (现代化UI) ──→ 建议在任务5之后（避免功能变更导致UI返工）
任务7 (PC/移动统一) ──→ 必须在任务6之后（统一需基于最终UI方案）
```

| 第1批 | 任务1 + 任务2 | Bug 修复，可并行，无依赖 |

## 风险提示

1. **任务5（文件夹模式）** 是最大的功能变更，涉及数据模型扩展、存储层修改（Python 后端 + SQLite）、UI 重构，建议拆分为多个子任务逐步实现
2. **任务6+7（UI 重构）** 可能影响所有现有功能的视觉回归，需在每个功能点完成后进行回归测试
3. **移动端测试** 需要实际 Android 设备或模拟器，部分问题（如图片显示、手势返回）仅在真机上可复现
